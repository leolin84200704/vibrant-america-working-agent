## Ticket Analysis

### [2026-06-04]
- **Title**: [BE] Database Schema - Meeting Type Entities. Epic VP-15817. Repo LIS-transformer-v2, calendar postgres (DATABASE_URL_CALENDAR), Prisma.
- 建 3 張**全新**表（PRD 明文「separate from existing practice-level appointment types」→ 不是改 v2_practice_event_type）：
  - `MeetingType`：id, **provider_id**, practice_id, name, slug(unique per provider), duration_minutes, duration_type(preset/custom), location_id(nullable), video_conferencing_enabled, video_conferencing_provider, buffer_minutes, display_order, is_active, is_deleted, created_at, updated_at。
  - `MeetingTypeWeeklyAvailability`：meeting_type_id(FK cascade), day_of_week(0-6), start_time, end_time。
  - `MeetingTypeDateSpecificHour`：meeting_type_id(FK cascade), date, start_time?, end_time?, is_unavailable。
- AC：表+FK+型別正確；availability/date 表 cascade delete；unique(provider_id, slug)；index on meeting_type_id/provider_id/slug/is_active；migration 不掉資料。
- **這是 provider/customer 層級**（有 provider_id）。VP-16513(CRUD)/VP-16514(toggle+reorder)/VP-16516(availability) 都站在這張表上。

### 既有 schema 慣例（要對齊）
- calendar postgres。既有 v2_ 表 PK = **Int @id @default(autoincrement())** + 常配一個 `uuid String @default(gen_random_uuid()) @db.Uuid` 欄位（v2_calendar:307-308）。**PRD 寫 UUID PK，但 repo 慣例是 Int PK**。
- 「provider」在 calendar = `calendar_owner_id`(Int)；v2_calendar `@@unique([practice_id, calendar_owner_id])`，role_type enum 預設 provider。
- location 表 = `v2_clinic_location`（schema:586）→ location_id FK。
- timestamps `@db.Timestamptz(6)`；命名 snake_case + `v2_` prefix。

## Approaches Considered

### [2026-06-04] 待 Leo 確認的 schema 決策
- **PK 型別**：建議跟 repo 慣例走 **Int PK + uuid 欄位**（非 PRD 的 UUID PK），與 v2_calendar/v2_event 一致、FK 也都 Int。
- **table 命名**：建議 `v2_meeting_type` / `v2_meeting_type_weekly_availability` / `v2_meeting_type_date_specific_hour`（對齊 v2_ 慣例），非 PRD 的 PascalCase。
- **provider 表示**：`provider_id Int`（語意 = calendar_owner_id / customer_id）。unique (provider_id, slug)。是否真的設 FK 到 v2_calendar 待議（v2_calendar PK 是 calendar_id，不是 provider_id；可能存 raw Int 不設 FK，跟既有 practice_id 一樣存 raw）。
- **location_id**：FK→v2_clinic_location（nullable）。
- **is_active / is_deleted 並存**：is_active=停用切換（VP-16514）、is_deleted=soft delete（VP-16513 DELETE）。

### 候選方案與正反取捨
- **PRD 原樣：UUID PK + PascalCase 表名 + 硬 FK 到 provider/practice 表**
  - 取捨：嚴格符合 PRD 字面，但與 repo 全部既有 v2_ 表（Int PK、raw Int 外鍵、snake_case）不一致，FK join 型別不匹配、下游程式碼風格分裂；Leo 已拍板放棄。
- **在 weekly 子表加 @@unique([meeting_type_id, day_of_week]) 防同日重複；在 DB 層用 TIME 型別存 start/end_time**
  - 取捨：可在 DB 層擋掉重疊/格式錯誤，更安全；但 PRD 允許同一 day 多段 availability（多 row），硬 unique 會擋掉合法多段；time 格式校驗目前刻意留給 VP-16516 應用層 DTO。屬 VP-16516 決策，不應在本 schema 票擴張。

### 風險與邊界案例
- start_time/end_time 用 VARCHAR(5) 存 'HH:mm'，無 DB 層時間驗證 — 依賴 VP-16516 寫入時 DTO 校驗格式與 start<end。
- provider_id/practice_id/location_id 不設硬 FK → 孤兒列風險，依賴應用層 ownership check（VP-16513 的 assertProviderAccess）。location_id 指向 v2_clinic_location 但無 FK，刪 location 不會 cascade。
- weekly 表無 unique 防同 day_of_week 時段重疊 — 重疊規則留給 VP-16516 應用層；若 VP-16516 未做會產生不可預期可訂時段。
- AC5 驗證僅靠 additive-only 推論 + STM 記載兩 schema 建表 success；建議 QA 在 staging 跑一次 cascade delete 實測（建 meeting_type → 建子列 → 刪 parent → 確認子列 100% 消失）而非只信 migration 文本。
- 回退：本票宣稱 prod calendar_prod 已建表，但 prod DB 寫入應 verify live 非 mock；目前只有 STM 文字記載，無附 SELECT count 證據。建議在收尾前對 prod+staging 各 information_schema 確認 3 表/unique index/FK 實際存在。
