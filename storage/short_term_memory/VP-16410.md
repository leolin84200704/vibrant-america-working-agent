- **修法**：把 `.ts` script 移到 `~/vp-16410-integration-test.ts.local-backup`，重 build dist 結構就恢復扁平
- **教訓**：
  1. 在 repo root scripts/ 寫 utility 一律用 `.js`（call PrismaClient JS API 即可，不用 ts-node 也能跑）；要寫 TypeScript 就放 `src/` 之外、且加進 `tsconfig.build.json` 的 exclude

## Lessons Learned

### 技術 (technical)
- **nest build dist 結構由 tsc 自動推導 rootDir 決定**：所有 source 的最低共同父目錄。加 src/ 之外的 .ts 會把 rootDir 拉到 repo root，dist 變多一層 `dist/src/`，導致 source 內的 relative import（譬如 `prisma2.service.ts` 的 `'../../prisma2/generated/client2'`）跨層解析錯誤。修法：(a) tsconfig 顯式 `"rootDir": "./src"`、(b) 在 `tsconfig.build.json` exclude `scripts/**`、(c) TS test 一律放 src/ 內
- **calendar DB 三個 schema**：`calendar_dev` (空)、`calendar_dev_new`、`calendar_prod`。DB migration 前先 `SELECT schema_name FROM information_schema.schemata WHERE schema_name LIKE 'calendar%'` enumerate
- **Prisma `ON DELETE CASCADE` 不寫 audit**：手動 SQL DELETE v2_event 會 cascade 清 claim row，但不會走 service → 沒 audit log。要保留 audit 必須走 service mutation

### Repo pattern
- **AccessionClaim service 設計可重用**：tx 參數讓 service 在 caller 的 interactive transaction 內 atomic；P2002 fallback + pre-check findMany 雙保險；rejected audit 用 separate transaction（因為 main tx 會 rollback）
- **prod backfill 的 dirty data 採「先到先得」可接受**：歷史資料不一致時 reject row 留 audit，不嘗試 heuristic 重分配，未來用 reset API 解決 by case

### Process
- **Leo 否定句指令要拆 scope**：「過去 event 不動」≠「rollback claim 表」。先回問「不動的是 X 還是 Y？」比 implicit 擴大省一回合
- **prod 測試的不可逆 side effect**：Kafka topic publish + email send 都 fire-and-forget，建立的測試 event 即使刪 row 也救不回 downstream 動作。明確警告「local server 連 dev_new 才能測 mutation」
- **API doc 走 team-standard markdown**（已知偏好，再次驗證）：Confluence-style ToC + Background + Architecture + DB schema + Endpoints with example response。不要交付 raw OpenAPI YAML
