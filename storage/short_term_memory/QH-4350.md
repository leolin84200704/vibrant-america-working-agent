### 建議方案（改前 / 改後為何有效 / 改什麼）
純資料驗證、零 code change（方案 A）。改前現狀：MSH-5/MSH-12 完全由 DB 驅動——MSH-5 = integration.ehr_vendor?.code || legacy_emr_service || 'DEFAULT'（src/modules/result/services/result-generation.service.ts:728，另一路徑 :849），MSH-12 = integration.hl7_version || '2.3'（同檔 :734、:851）；encoder 只原樣輸出（src/modules/hl7/services/hl7-encoder.service.ts buildMSH：MSH-5=receivingApplication、MSH-12=hl7Version）。改後為何有效：只要 Charm vendor 的 ehr_vendors.code = 'ChARM_EHR' 且該 integration 的 hl7_version ∈ {2.3.1, 2.5.1}，encoder 即輸出正確值，無特例 code、零回歸。具體動作：(1) prod 跑 SELECT ei.customer_id, ei.clinic_id, ei.hl7_version, v.id AS vendor_id, v.code FROM ehr_integrations ei JOIN ehr_vendors v ON v.id=ei.ehr_vendor_id WHERE v.code LIKE '%CHARM%'，確認 VP-16010 是否已落地。(2) 若 code='ChARM_EHR' 且 hl7_version IN ('2.3.1','2.5.1') → 已落地：抓「真實已傳給 Charm 的 HL7」人工檢視 MSH-5/MSH-12 後關票（per memory verified_means_live_not_mock，禁用 mock/spec）。注意 Charm 是雙傳（SFTP + HTTP POST，isCharmEhrVendor @ result-generation.service.ts:965），HTTP POST payload 也要驗（不可只看 SFTP archive）。(3) 若仍是 CHARMEHR / 2.3 → bounded UPDATE（明確 vendor_id/customer_id/clinic_id），逐 row 驗 100%，再協調 repush。修正 MSH-5 必須落在 ehr_vendors.code / ehr_vendor_id 對應，白名單（含 CHARMEHR 與 ChARM_EHR）不能代替字面值。注意：工程師原分析把檔案路徑寫成 src/modules/hl7/services/result-generation.service.ts，實際在 src/modules/result/services/。

### 候選方案與正反取捨
- **方案 B：在 encoder 對 Charm 硬寫 MSH-5/MSH-12**
  - 取捨：違反 source-of-truth（MSH 值來自 DB integration），與其他 vendor 行為分歧，VP-16010 已用資料解法 Done，無理由引入特例 code；否決。
- **完全不查 prod，直接信任 VP-16010 Done 就關票**
  - 取捨：違反 iron rule verify-live-not-mock 與 QA 驗收本職；萬一 vendor code 在某些 Charm integration 仍是 CHARMEHR 或 hl7_version 漏更新（Charm 可能有多 practice：5144 / 16289 / provider 15185 等），會 silently 漏；至少要抓一筆真實 push 過的 HL7 檢視。

### 風險與邊界案例
- Charm 是雙傳通道（SFTP + HTTP POST）；驗收必須同時檢視 HTTP POST 送出的 payload，不能只看 SFTP archive 檔，否則漏掉 HTTP 路徑的 MSH 值
- Charm 可能跨多個 practice/provider（relate 票涵蓋 practice 5144、16289、provider 15185 / 10464）；SELECT 與驗證要 cover 全部 Charm ehr_integrations row，逐 row 看 hl7_version 與 vendor_id，不可只看一筆
- MSH-12 究竟落 2.3.1 還是 2.5.1：兩者皆 Charm 可接受，但 encoder 其餘 segment 仍按 2.3 佈局；若 DB 設成 2.5.1，需確認 Charm 不會因 segment 結構不符 2.5.1 schema 而 reject（查實際已成功 ACK 的 HTTP transmission 即可佐證）
- ehr_vendors.code 為 legacy mixed-case 且 MySQL collation case-insensitive；WHERE 比對不受影響，但寫入/驗證時要用實際字面值 'ChARM_EHR'（含底線、混合大小寫）
- 若需 UPDATE 修正：prod 寫入必 bound scope（明確 vendor_id + customer_id + clinic_id + time），raw SQL 不觸發 Prisma @updatedAt 要手動 SET updated_at=NOW()，UPDATE 後反向 broader-criterion audit 找漏網 row
- repush 已由 VP-15948 完成；避免重複 repush（會對 Charm 重送），關票前向 Kristine 確認驗收標準即可，勿擅自再觸發
