### 風險與邊界案例
- 4 個 RPC（SHIPPING/AUDIT/DASHBOARD/ISSUE）目前無 cloud endpoint，每次呼叫 silent-fallback 回 on-prem；若 QA 不覆蓋 fallback 路徑，cloud cutover 後才會爆。必須驗 fallback 路徑且 link VP-16784-87。
- fallback 只對 transient code + 'not initialized'/'not available' 字串觸發；若 cloud 回非標準 error（如自訂 message 但 code 為 OK/業務碼），會誤判為 business error 直接 throw 不退 on-prem — QA 應構造一個 cloud-down 情境驗證真的退到 on-prem（peer-observed：確認流量打到 on-prem 端）。
- byte-parity 要含隱形欄位（proto3 zero-value、timestamp）；只比 happy-path 200 不夠，需涵蓋 NOT_FOUND/INVALID_ARGUMENT 等 business error 兩端是否一致 propagate。
- 切 branch 後若漏跑兩個 prisma generate，start:dev 會因 stale client 假性報錯（start:dev iron rule，VP-16521 前例）；當前 working tree 不在 stage_test。
- byte-parity 必須用真實 dist service 對打，不可用 mock unit test 充當『線上驗證』（verified-means-live 教訓 VP-16850）。
- 勿動同 module 的 QH-2648（provider→clinician 轉移）/QH-1591（optimize）未 merge 改動，避免驗證對象漂移。
