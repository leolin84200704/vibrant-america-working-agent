## Ticket Analysis

**Goal（目的）**：When the "result ready" email is sent to the customer, the product_list field must be built from the V2 endpoint (base-report-service/result/getReportStatusListV2?barcode=) instead of the legacy getReportStatusList. The ticket body is a one-liner with example barcode 2410246397, no AC. Net effect: same email, product names sourced from V2.

**Classification**：`notification-email` | **Effort**：S | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-setting-consumer

### 建議方案（改前 / 改後為何有效 / 改什麼）
目的: result_ready_new email 的 product_list 改用 V2 取數來源。

改前現狀 (已 ground 到 code): LIS-setting-consumer/src/setting-consumer/setting-consumer.controller.ts 的 beta 分支 (clinic>=136622 || is_beta, Tag=result_ready_new) L1438-1451 用 this.getRequest(process.env.url_report_product_list + accession_id) 打 V1 endpoint (.env:114 = getReportStatusList?barcode=)，並在兩個 TemplateModel (prod TemplateId 35058386 L1472-1477 / 非prod 38859237 L1506-1514) 對 report_list['finished_reports'] 直接 .join(', ')。V1 的 finished_reports element 是字串可直接 join。

改後為何有效: V2 helper getReportStatusListV2(accession_id, tracking_id) (L7465) 已存在且 prod-proven — 讀 process.env.getReportStatusListV2 (.env:164 已設好)，含 Redis cache (EX 8640) + 10x retry + Sentry/lis_front_logger。同檔 L8215-8224 已有另一段現役 code 用同一 helper 並以 finished_reports[i].report_staus == 'Final' 過濾後取 finished_reports[i].report_name — 證實 V2 element 是物件 { report_name, report_staus(此拼字為 codebase 既定，勿改) }，且 codebase house pattern 就是 Final-only。直接把 V1 字串 join 換成 V2 物件 → report_name 陣列 join 即正確。

具體改什麼 (單一檔案 setting-consumer.controller.ts):
1. L1438-1451: 把 getRequest(url_report_product_list...) 換成 const v2 = await this.getReportStatusListV2(list_sample.sample[0].accession_id, tracking_id);（沿用既有 isEmpty 重試包法或直接靠 helper 內建 retry）。
2. L1472-1477 與 L1506-1514 兩個 TemplateModel 的 product_list: 改為先 const names = (v2?.finished_reports ?? []).filter(r => r.report_staus == 'Final').map(r => r.report_name); product_list: this.settingyool.isEmpty(names)==false ? names.join(', ') : ''。沿用 L8222-8224 既有 Final-only 寫法保持 codebase 一致。
3. 不動 .env / setting-consumer.yaml (V2 env 已存在，無新 process.env.X，避開 config-yaml coupling 鐵律)。
4. 不動非 beta 分支 (Tag result_ready, TemplateId 33002864) — 它本來就沒有 product_list 欄位。

鐵律: branch feature/leo/VP-9299；PR base stage_test (不直推 staging/main)；改完跑 npm run start:dev 須過；對新 mapping 補/跑 unit test (codebase test 稀少，需手動加 spec 或至少 ship 前用 ticket 提供的 prod barcode 2410246397 實打 V2 確認欄位拼字與 Final 取值，verified-means-live)；code/comment 全英文。

### 候選方案與正反取捨
- **方案 B: 不動 getRequest，只把 URL 換成 process.env.getReportStatusListV2**
  - 取捨：diff 最小，但會壞 — V2 element 是物件，.join(', ') 會輸出 [object Object], [object Object]，且放掉現成 V2 helper 的 cache/retry/log。否決。
- **Final-only filter vs 全列 (不 filter)**
  - 取捨：codebase house pattern (L8223) 是 Final-only，建議預設採 Final-only 並對齊既有 code；但 V1 舊行為是把全部 finished_reports 不過濾 join，改 Final-only 可能讓 product_list 變短。差異輕微但屬內容變化，值得對 Leo 一句 heads-up，不應 block。

### 風險與邊界案例
- V2 element 物件型別與 V1 字串型別不同：若忘記改 mapping (只換 URL) → product_list 變 '[object Object]'。這是最大回歸風險。
- report_staus 是 codebase 既定拼字 (typo 但一致)，新 mapping 必須沿用此拼字否則 filter 永遠不 match → product_list 空字串。
- Final-only filter 改變 email 內容：V1 不過濾全列，V2 Final-only 可能漏掉 non-Final 報告 → 顧客 email 列出的產品變少。需確認此語意可接受。
- getReportStatusListV2 失敗時 return null (helper L7500)；新 code 須 null-safe (v2?.finished_reports ?? []) 否則 .finished_reports 會丟 TypeError 中斷 email 發送。
- 兩個 TemplateModel (prod/非prod) 都要改，漏改一個會造成環境間行為不一致。
- 不能改非 beta 分支 (它無 product_list)，避免範圍蔓延。
