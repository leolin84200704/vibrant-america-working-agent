## Ticket Analysis
[BE][EMR] Proactively Call Pricing API to Substitute NY Bundle ID Before Submitting Order. P1, assignee Leo, reporter Xiaoye Li, Dev In Progress.
- 背景: EMR 不知道 provider 選的 bundle 是否含 GZ。送單前要主動呼叫 pricing API,若 bundle 含 GZ 且病人 NY-eligible → 取 NY-variant bundle id 替換。
- AC: NY-eligible+含 bundle 時送單前呼叫 pricing API; 正確替換 nyBundleId; 原 bundleId 一併帶上(給未來 edit order); NY 替換時含 $80; API error → 擋單; 不改 pre-order display。
- 是 VP-16832(頂層 GZ package VAREQUISTION463↔485) 沒處理的 **bundle(VACP)內含 GZ** 情況。

## API (doc: /Users/hung.l/src/ny-variant-api.md)
`POST /item/promotion/resolveNyVariant` (LIS-backend-v2-pricing)。
- 輸入 exactly one: `bundleIds[]` 或 `EMRCodes[]`(VACP<order_type_id>)。
- 回 `results`(保序),每筆 `hasNyVariant` + (true 時)`nyBundleId`(NY twin promotion 主 id)+ `bundleOrderTypeId`。
- standard GZ item id = 836/845 → NY = 867。twin 只換 GZ item,name/price/其他不變。
- custom panel(customer_id/clinic_id>0): get-or-create twin 並 sync lis_re 後才回。official bundle: 須預先產,否則 hasNyVariant:false。
- hasNyVariant:false 條件: 找不到/不含 GZ/本身是 twin/x-choose-y(selection_quantity)/official 未預產。
- v1(/orderTest, emr-v2 用這) → 把 order line item_id 換成 nyBundleId。
