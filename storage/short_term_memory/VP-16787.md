# VP-16787 - [BE] Migrate AUDIT_RPC to cloud interface

> Created: 2026-06-06 (batch 分工辯證 of all open tickets)
> Status: active
> Type: rpc-cloud-migration | Priority/Status from Jira at analysis time

---

## Ticket Analysis

**Goal（目的）**：將 transv2 的 AUDIT gRPC client 從 on-prem (192.168.60.6:32700) 遷移到 cloud (lis-auditlog-grpc-service.default.svc.cluster.local:30113)，使所有 Calendar/EMR audit 路徑以 cloud 為 primary，並補齊缺失的 AUDIT_RPC_CLOUD config 注入（此為 parent VP-16685 卡 Dev Blocked 的真正部署缺口）。最終目標是 audit 流量走 cloud，on-prem 不再被 default 使用。

**Classification**：`rpc-cloud-migration` | **Effort**：L | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
採方案 B：對齊 VP-16685 既有的 withCloudFallback (cloud-primary + on-prem backup + GRPC_CLOUD_FALLBACK_ENABLED kill switch) pattern，但修正 engineer 嚴重低估的 scope。

改前現狀：grpc.option.ts 已有 audit (L104, process.env.AUDIT_RPC) 與 audit_cloud (L277, process.env.AUDIT_RPC_CLOUD)；Notification 已 cloud-primary+fallback；但 yaml/st.yml(L164) 與 yaml/prod.yml(L171) 只注入 AUDIT_RPC，AUDIT_RPC_CLOUD 在任何 configMap/yaml 皆不存在 → 連 Notification 的 cloud path 在 st/prod 都拿到 undefined、實際永遠走 on-prem fallback。

改後為何有效：補齊 config 後 cloud 才真正生效；所有實際 invoke audit 的 service 收斂到同一 withCloudFallback，cloud primary 達成 migration 目的，on-prem 留 backup 確保 audit 不掉（audit 多為 fire-and-forget），kill switch 可免 revert 退場。

具體改什麼（已逐檔 grep 驗證 — 修正 engineer 漏算）：
1. yaml/st.yml + yaml/prod.yml 各加 AUDIT_RPC_CLOUD env block（仿 AUDIT_RPC，valueFrom configMapKeyRef），並在對應 configMap (lis-transv2-config-st / prod) 補 AUDIT_RPC_CLOUD key 指向 cloud endpoint。鐵律：新 env 同 PR 改兩份 yaml 並明說。
2. grpc.option.ts 無需新增（audit_cloud 已存在）。
3. 為「實際 invoke audit RPC」的 5 個 on-prem-only consumer 各加第二個 cloud client + 用 withCloudFallback 包呼叫點：
   - src/rbac-audit-log/rbac-audit-log.service.ts (SearchAuditLog L165 + RecordAuditLog L246)
   - src/practice-audit-log/practice-audit-log.service.ts (SearchAuditLog L131)
   - src/vitals/vitals.service.ts (recordAuditLog L167)
   - src/utility/utility.service.ts (RecordAuditLog L1959)
   - src/setting/setting.service.ts (RecordAuditLog x9: L422/428/434/496/563/752/1022/2540 + recordAuditLog L3353) — 最大宗，engineer 完全漏掉
   各自 module 註冊 cloud client (audit_cloud)。
4. src/trans/* 的 7 個 @Client(audit) 宣告 (merchandise/utility/setting/patientProfile/service/utility.api/PNS) 實測零 auditService 呼叫 → 屬未使用宣告；建議直接刪除這些 dead @Client(audit) 以滿足 AC2「on-prem reference fully removed」（不需加 fallback）。
5. 切 branch 後跑 npx prisma generate（兩份 schema）+ npm run start:dev 必過；unit test 涵蓋 cloud-success / transient→fallback / kill-switch 三分支。
6. AC2 達成方式：on-prem 保留為 fallback backup（非徹底刪 client），但移除 dead trans/* 宣告 + 移除任何硬編 192.168.60.6 字串 — 需 PM 確認語意（見 clarifyingQuestions）。

### 候選方案與正反取捨
- **方案 A 硬切：直接把 audit client url 改 cloud、刪 on-prem client，無 fallback**
  - 取捨：最符合 AC2 字面『fully removed』且無 dead code，但違反 repo 既定 VP-16685 fallback pattern、與已落地的 Notification 不一致、cloud audit 不穩時 audit 直接掉且無 revert-free 退路。風險高，不建議。
- **只改 Notification + 補 config（最小化，假設其他 consumer 不在 Calendar/EMR scope）**
  - 取捨：AC1 要求『all relevant Calendar/EMR code paths』；rbac/practice/vitals/utility/setting 仍連 on-prem，AC2 未達成，會留 migration 半成品，需後續 ticket 補。除非 PM 明確縮窄 scope 否則不合格。

### 風險與邊界案例
- Engineer 漏算 consumer：實際 invoke audit 的有 5 個 on-prem-only service（含 setting.service.ts 的 9 個 RecordAuditLog 呼叫），非 engineer 宣稱的 3 個。漏改 utility 與 setting 會導致 AC1 未達成且 regression。
- src/trans/* 7 個 @Client(audit) 宣告無實際呼叫；若誤加 fallback 是 over-engineering，若不處理則 AC2『fully removed』字面未過 — 應確認後刪除 dead 宣告。
- AUDIT_RPC_CLOUD 在 configMap source 必須先存在且值正確，否則 env undefined → 即使 code 改好仍全程走 on-prem fallback（Notification 已踩此坑）。部署後須 peer-observed 驗證 cloud service 真的收到流量。
- cloud endpoint 用 default svc cluster-internal DNS：st cluster 是否有同名 service？若 st 無此 service，AC3 cloud-env regression 無法驗。
- audit fire-and-forget：fallback 退化雖安全，但若 cloud 與 on-prem audit 寫入不同 DB，遷移期間 audit 記錄可能分散兩處（需確認 cloud audit service 是否讀同一資料源）。
- withCloudFallback 只對 transient code (4/13/14/2 + not initialized/available) fallback；若 cloud 回 business error 不 fallback — 須確認 cloud audit service 的 SearchAuditLog 契約與 on-prem 完全一致（欄位/分頁/timestamp 行為），否則 rbac/practice query 結果可能 silently 不同。
