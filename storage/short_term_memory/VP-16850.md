## Ticket Analysis

- Bug: "Trouble with VI calendar". Customer_id 46607, Calendar_id 30791 (Emaline Brown).
- 症狀：availability 設在 :15 / :45 的時段（e.g. 09:45, 13:15）不會顯示也不能 booking；客戶問是否只能改成整點/半點。
- Repo: LIS-transformer-v2，query `getLabClinicianAvailability` → `ProviderAvailabilityService.getProviderAvailability()`。

## Root Cause
### [2026-06-02 17:56]
`provider-availability.service.ts:899 generateSlotsFromPeriods()`：
- `SLOT_INTERVAL_MINUTES = 30` 硬編碼 (line 112)。
- line 919 `Math.ceil(minutes/30)*30` 把每個 period 的起點「往上對齊」到全域 :00/:30 grid。
- 每個 DB availability row 只有 30 分鐘寬。起點 :15→:30、:45→下個整點，往後移 15 分鐘後，30 分鐘的 slot 的 slotEnd 已超過 period.end → line 939 `slotEnd <= period.end` 不成立 → 整段被當 skippedOutOfRange 丟棄。
- 兩條路徑都走這函式：無 exception 走 `generateSlotsForTimeRange`(995)→`generateSlotsFromPeriods`；有 exception 走 `calculateSlotsWithExceptions`(797)→merge→`generateSlotsFromPeriods`。
- 重複實作：`event.service.ts:2481 generateScheduleAwareSlots()` 有同樣 grid 邏輯。
- 不一致點：booking-fit 驗證 `getProviderAvailability` line 436 用 `availablePeriods.some(p => p.start<=startTime && p.end>=endTime)`，不套 grid → 09:45 其實「裝得進 period」。所以 display 丟掉但 fit 邏輯認得，兩邊不一致。

## 實證驗證
