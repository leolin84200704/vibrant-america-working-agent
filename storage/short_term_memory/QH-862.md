# QH-862 - Add readiness and liveness Probe to transv2

> Type: infra-devops | Priority/Status from Jira at analysis time

## Ticket Analysis

**Goal（目的）**：讓 AKS 能對 transv2 pod 做健康 gating：liveness 失敗重啟 pod，readiness 失敗暫時移出 Service endpoint。實際 gap 經查證僅剩 staging：prod.yml 早已正常配置 liveness+readiness（GET /health:3390），而 st.yml 的 probe 整段被註解、port 寫錯（3190 應為 3390），且 readiness 區塊不完整。本 ticket 是 VP-13703(Done) 的 QA 鏡像，無 AC。

**Classification**：`infra-devops` | **Effort**：S | **Confidence**：5/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
採方案 A 最小切片，只改 yaml/st.yml，零程式碼、零新 env。改前現狀：prod.yml:26-41 已有完整 liveness/readiness 打 GET /health:3390（已驗證），src/app.controller.ts:13-19 的 /health 已存在回 {status:'ok',timestamp}；唯一缺口是 yaml/st.yml:25-38 整段 probe 被註解，且 port 寫成 3190（containerPort 實為 3390），readiness 區塊還被截斷（缺 timeoutSeconds/failureThreshold）。改後為何有效：解開註解後 staging pod 也納入 AKS 健康 gating，補上唯一缺口；prod 不動故零回歸。具體改什麼（branch bugfix/leo/QH-862，base=stage_test，PR 不直 push）：1) 解開 st.yml livenessProbe/readinessProbe 註解；2) 兩處 port 3190→3390；3) 補齊 readiness 的 timeoutSeconds:3 / failureThreshold:3（對齊 prod，目前 commented 版本缺這兩行）；4) staging replicas=1，liveness 用較寬鬆值（initialDelay30/period15/failure3）避免冷啟動被誤殺單一 pod。/health 為 shallow check（不檢 DB/Redis/gRPC），維持與 prod 單一 path 一致即可。注意：repo 內 src/mypractice/mypractice-health.controller.ts 已有 /mypractice/health 與 /health/redis 的 Redis-aware pattern，若日後 Leo 要 deep readiness 可參考，但本 ticket 不引入。

### 候選方案與正反取捨
- **方案 B：引入 @nestjs/terminus，拆 /health/live 與 /health/ready，readiness 檢 Prisma(MySQL+PG)/Redis ping**
  - 取捨：真正反映可服務性、避免 shallow 假健康，但 over-engineering：新 dep、改 app.module、雙 Prisma client 注入、需改 prod.yml probe path（碰 prod）；若把上游 gRPC 算進 readiness，上游抖動會整批踢出輪替擴大故障面；且正進行的 RPC 雲端遷移(VP-16685 等)會改變上游拓樸。應另開 ticket。
- **方案 C：完全不改 code/yaml，只把 ticket 當純 QA 驗證（驗 prod probe 已生效）並關掉**
  - 取捨：承認 prod 已達標，但 staging 仍無 probe 是實質缺口，留著反而誤導；需 Leo 確認 staging 是否在範圍內。

### 風險與邊界案例
- staging replicas=1：liveness 過嚴會重啟唯一 pod 造成短暫 downtime，須用寬鬆 initialDelay/period
- /health 是 shallow check：DB/Redis/gRPC 死掉時 readiness 仍回 ready，可能放錯流量進來（同 results-grpc /swagger 假健康陷阱）— 本 ticket 維持現狀，需在報告標明這是已知限制
- commented 版 readiness 區塊不完整（缺 timeoutSeconds/failureThreshold），直接解註解會用 K8s default，須主動補齊對齊 prod
- YAML 縮排錯誤會讓整個 Deployment apply 失敗 — 改完須 kubectl apply --dry-run=client 或 kubeval 驗證
- 驗證須 peer-observed：deploy 後 kubectl describe pod 確認 Liveness/Readiness 欄位出現且 probe 成功計數遞增，不能只看 my-side log
