## Phase 2b — outbound upload 序列化 [2026-06-25] branch bugfix/leo/VP-17217-phase2b (PR #201, commit 2d1f0af)
- 線上發現新症狀：sample 2566061 檔案已上傳到 vendor 但 gRPC 不回。根因：**兩筆並發 outbound result-push 共用 outbound singleton**——9:10:03 2566061 put 到 /morcom 中，9:10:10 另一筆(2566246 /public)上傳完 disconnect() → 砍掉 2566061 在傳的 socket → bytes 已 flush(檔案出現)但 ACK 回不來 → put() hang 到 10min timeout → gRPC 不回。Phase 1 只隔離 inbound↔outbound，**沒保護並發的 outbound 彼此**。觸發=手動送 overlap kafka pipeline(ResultGenerationProcessor) / batch overlap，POD_ROLE=all。
- 修法：SftpService 加 promise-chain `runSerialized`，把 uploadHL7File 的 connect→put→disconnect 串成不重疊 critical section（一次一筆）。每段自帶 finally disconnect、chain 吞 result 不被單筆失敗卡死。**不用 connection-level mutex**(避免 SftpController 裸 connect/disconnect 死鎖)。throughput 無損(vendor 本來就喜歡一次一條)。
- test：sftp 4 suites/42 green，新 sftp.service.spec 3 條(maxActive===1 / connect-disconnect 不重疊 / 失敗不卡 chain)。build clean、無 CJK。
- 未涵蓋(低風險 follow-up)：base-report scheduled / SFTP test endpoint 直接用 connectionService 的路徑；outbound 同 host 連線重用(優化，非必要)。

## Phase 2a — order-fetch 連線重用 [2026-06-25] branch bugfix/leo/VP-17217-phase2 (PR #198, commit d642bb5)
- 根因確認：MDHQ 一台 host ~172 folder，cron 每 15min 對同 IP ~172 次 connect → 踩 Bitvise per-IP 連線上限 → `Unexpected end event` ~6000/day。**並發會更糟**，正解=連線重用。
- 改 `Hl7OrderFetchService.runIngestion`：`groupFoldersByHost`(host:port:username 分組) → `processHostGroup`(每組連 1 次、組內 folders 重用、組末 disconnect) → `processFolderOnConnection`(不 connect/disconnect，靠 SftpFileService 內建 ensureConnected 中途死自動重連)。
- 保住 hang guard：folder timeout → drop+reconnect 下一個；reconnect 失敗→該 host 剩餘 folder 延到下個 tick。connect/reconnect 都包 runWithTimeout。
- 跨 tick 持久連線**不做**（idle reap + stale 風險）；每 tick 重用即可。
- test：4 suites/53 green（新增 same-host→1 connect / 多 host→各 1 / no-creds skip；hang test connect 3→2）。build clean、無 CJK。
- Phase 2b（result-push outbound 同 host 重用）未做：要動 result-generation.service:249 的 per-upload connect→put→disconnect，碰核心送單 pipeline，風險高，留獨立 PR。

## Ticket Analysis
從「result 發不出去 / trigger order 就卡住」診斷出：`SftpConnectionService` NestJS singleton 共用單一 `this.client`、`connect()` 第一行無條件 `safeDisconnect()`（sftp-connection.service.ts:47-48）、無 mutex。`Hl7OrderFetchService` `@Cron 15min` 逐 ~200 folder `connect()`（line 269）→ 砍掉正在 `put()` 的 result 上傳 socket → 卡到 10min timeout / `Unexpected end event`（單日 6000+）。重現 INCIDENT-20260601。live：兩 on-prem pod POD_ROLE 未設=all、REDIS_HOST=localhost（不共用）→ pod-role 分流救不了。詳見 [[project_emr_v2_sftp_pod_role]]。

## Decisions Made
- 不走 Path A（POD_ROLE 分流）：需共用 Redis + 第二 prod deployment。Leo 確認不需要兩個 pod。
- 走 Path B code 修。**Phase 1 放棄 mutex**（`SftpController` 有獨立 connect/disconnect HTTP endpoint → acquire-in-connect/release-in-disconnect 會全 pod 死鎖）。
- Phase 1 改用**連線隔離**：給 inbound order-fetch 專屬 SftpConnectionService+SftpFileService instance（與 outbound result-push 的共用 client 分開）→ 同 pod 內不互砍，零死鎖風險，是 Phase 2 pool 的子集。Leo approve。
- branch base = origin/staging（已含 VP-17117 PR#194）。
