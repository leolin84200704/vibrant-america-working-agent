# VP-16885 - [BE] Support EHR enrollment flag on invite

> Created: 2026-06-06 (batch 分工辯證 of all open tickets)
> Status: active
> Type: practice-integration-epic | Priority/Status from Jira at analysis time

---

## Ticket Analysis

**Goal（目的）**：讓 practice admin 在「邀請新成員」modal 勾一個 optional checkbox (enrollToEHR)，把被邀請的 provider 一併報名進該 practice 的 EHR integration，省去事後到 portal Third-party Integration 手動設定。AC: invite payload 收 enrollToEHR (default false) → true 時觸發 enrollment → enrollment 失敗不可擋邀請 → 狀態之後在「既有」portal Third-party Integration settings 管理（不新建管理 UI/endpoint）。

**Classification**：`practice-integration-epic` | **Effort**：M | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer, lis-backend-emr-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
目的: invite 端 own「使用者意圖」(flag)，enrollment 真實寫入交給 emr-v2 + VP-16164 的 provider_integration_memberships，invite 不直接碰 ehr_integrations。

改前現狀（已 grep/STM 驗證，含對 engineer 的修正）:
- Invite 端點在 LIS-transformer (v1) 非 emr-v2: POST /reInviteCustomerWithoutPayload (src/trans/trans-patient-info.controller.ts:1100) → TransService.reInviteCustomerWithoutPayload (src/trans/trans.service.ts:20610)，DTO createCustomerEmailV2 (src/trans/dtos/patient.dto.ts:1508，已含 optional npi)；持久化走 invitation_history + invitation_email_payload JSON (trans.service.ts:20399/20446)。engineer 此部分正確。
- **重大修正**: engineer 說 provider_integration_memberships「目前不存在 → AC2/AC4 完全 blocked，VP-16164 是 hard blocker」。實際: VP-16164 STM (2026-05-27) 顯示該表 + practice_integrations 已用 raw SQL 建好並 apply 到 PROD (523 practice / 761 membership) 與 staging (3/26)，100% live 驗證。**表存在於 DB，只是 Prisma model 定義與 committed migration 還沒進 repo**（當時刻意 defer commit）。engineer 只 grep schema.prisma 就斷定「表不存在」，正是「verified means live not mock」反例。故真正 dependency 是「把 Prisma model 補進 emr-v2 schema.prisma + 補 committed migration」，不是「等 VP-16164 ship schema」。
- **修正 AC4**: epic out-of-scope 明寫「portal Third-party Integration settings UI (enrollment is only triggered here, managed there)」。AC4 是復用既有 portal 管理，不是新建管理端點。只要 enrollment 寫對 membership row (status/added_by) 即滿足。

改後為何有效: invite 成功且 enrollToEHR=true 時，v1 以 fire-and-forget try/catch 呼叫 emr-v2 一支 enrollment endpoint，寫 provider_integration_memberships(status='pending_approval', added_by='practice_admin', practice_integration_id 依 practice_id+vendor 解析, npi)；async/try-catch 天然滿足 AC3（失敗不擋 invite），狀態落 membership 天然滿足 AC4（既有 portal 可讀）。VP-16168 STM 確認 added_by enum 含 'practice_admin'、status 含 'pending_approval'，proposal 對齊既有 schema。

具體改什麼（執行期，dev start 6/22）:
- LIS-transformer: patient.dto.ts:1508 加 @IsOptional() @IsBoolean() enrollToEHR (default false)；trans.service.ts invite 成功分支讀 flag、持久化 enrollToEHR（建議獨立欄位於 invitation_history 以便查詢，而非塞 JSON，因為是可查狀態而非 email payload），並 true 時 fire-and-forget 呼叫 emr-v2 enrollment。
- lis-backend-emr-v2: (1) 先補 PracticeIntegration + ProviderIntegrationMembership Prisma model 進 schema/integration-management-schema.prisma + 補 committed migration（對齊 DB 既有結構，drift check）；(2) 新增 enrollment endpoint，寫 provider_integration_memberships。
- 任何新 process.env.X（emr-v2 enrollment URL）→ 同 PR 更新兩份 lis-emr-v2-config*.yaml（IRON RULE）。
- Branch: feature/leo/VP-16885；PR target stage_test，不直推。

### 候選方案與正反取捨
- **方案 C — Thinnest slice: v1 只收+持久化 enrollToEHR flag，enrollment 串接掛 TODO 等 Prisma model 補齊**
  - 取捨：完全不 blocked、可獨立交付、未勾選者零行為變化；但只交付 AC1，AC2/AC4 業務核心未做，是半成品，需 PM 明確同意 Phase 1 只存旗標
- **方案 A 完整 — 同 PR 補 Prisma model + enrollment endpoint + v1 串接，一次做完 AC1-4**
  - 取捨：一次交付完整業務價值；但 emr-v2 補 model/migration 需先做 DB drift check（VP-16164 raw SQL 結構 vs Prisma 定義可能漂移），且跨兩 repo 同步上線風險較高，需協調 FE VP-16887 contract
- **v1 直接寫 ehr_integrations（不經 emr-v2 endpoint）**
  - 取捨：省一個 service hop；但違反 routing（emr-v2 才是 integration owner）、ehr_integrations 是 integration-level 非 per-provider membership join 表，語意不符，且 v1 直碰 emr DB 破壞職責邊界，反推薦

### 風險與邊界案例
- enrollToEHR=true 但未提供 NPI: epic 列為待 PM 確認項（reject / 存 flag 不 enroll / 警告）；designer note 說 checkbox always-display 與 NPI 驗證狀態獨立，故 true+no-NPI 是真實可達狀態，必須定義
- invite 當下 provider 通常還沒接受邀請、沒有 customer_id(provider_id)；provider_integration_memberships.provider_id FK 填不了 → 可能需走 provisional path (VP-16168 provisional_providers) 或先以 npi-only membership 暫存，scope 可能膨脹
- VP-16164 表是 raw SQL 建的、Prisma 未 commit → 補 model 前必須先對 PROD/staging 既有表做 schema drift check（欄位/型別/enum 對拍），不可憑 PRD 或 STM 摘要假設結構
- 重複 enrollment: 同 provider 對同 practice_integration 重複 invite → membership 需 dedup（unique 或 upsert），避免重複 pending_approval row
- practice_id → practice_integration_id 解析: 一個 practice 可能多 vendor/多 integration，要決定 enroll 進哪一個 practice_integration
- fire-and-forget 失敗只記 log 無 retry → 若要可重試需有狀態可查（AC4 portal 可見即可間接補救）
- HIPAA: epic 明列 NPI 不可進 client/server log/analytics，enrollment 呼叫與 log 不可印 NPI
- RBAC: invite+enroll 是 practice-admin 動作，須與 VP-16172 權限 gate 一致
