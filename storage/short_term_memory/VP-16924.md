## Ticket Analysis

### [2026-06-08 17:36]
- Title: Clinical Consult Timezone Issue (Bug, VP project, status Dev In Progress, assignee Leo)
- Symptom: providers wrote in to confirm consult times; times they assumed were wrong.
  - Dr. Socha: actual 12PM **Eastern** (not 11AM ET)
  - Dr. Cameron: actual 12PM **Mountain** (not 11AM MT)
  - PM: "I tried checking their portal view, but it always puts it in PDT for me" → suspects times shown in UTC/PT instead of provider's real TZ.
- Repo: LIS-transformer-v2 (calendar module, practice_id=150105 clinical consult).

### Live-data root cause (verified against calendar_prod, read-only)
- **ALL 39 June-5 events for practice 150105 have `v2_event.timezone = 'UTC'`.**
- Cameron = Philip Cameron, cal 37866/14118, **`v2_calendar.timezone = 'UTC'`** (both his calendars). Socha = Megan Socha, cal 41834/13769, also `'UTC'`.
- Cameron's actual consult = event 9817, start_time **18:00 UTC**:
  - America/Denver (MDT, his real zone) = **12:00** ✓ (matches PM's "actual 12PM MT")
  - America/Los_Angeles (PDT) = **11:00** ✗ (matches the wrong "11AM" providers saw)
- Mechanism: `event.service.ts:4295-4319` `resolveEventTimezone` / `resolveRecipientTimezone`:
  both treat `'UTC'` as "no real tz" → `if (tz && tz !== 'UTC') return tz; ... return 'America/Los_Angeles';`
  Since event.timezone='UTC' AND participant calendar timezone='UTC' (always, this practice), **every** consult time renders in **Pacific**.
- `formatDateInTimezone` (4327) itself is DST-correct (date-fns-tz `toZonedTime` + `getTimezoneAbbreviation`). The bug is the **wrong target timezone fed into it**, not the formatter.
- Result: non-Pacific providers see times shifted by their offset from PT. Mountain = −1h (Cameron, exact match). Eastern would be −3h.
  - ⚠️ Could NOT locate Socha's event in calendar_prod for that week → cannot confirm her exact case. Pure PT-fallback makes Eastern off by 3h, not 1h, so Socha's "11AM ET" is either imprecise or has another factor. FLAGGED, not assumed.
- This is the user-facing manifestation of patterns.md "Email 時間 timezone 地雷三連" #2 (UTC fallback asymmetry) + #3 (~18k provider/clinicadmin v2_calendar timezone='UTC').
- GraphQL/portal path (`mapEventToGraphQL`) returns raw ISO + event.timezone='UTC' → frontend applies its own default (PM's "always PDT"). Frontend portal = separate repo, not examined.

## Approaches Considered

### [2026-06-08 17:36]
Core problem is **missing timezone DATA**, masked by a confidently-wrong LA fallback. No correct code-only fix exists without a real tz source. Approaches:
- **A. Backfill timezone data** (v2_calendar.timezone from real source + stop defaulting event.timezone/booking to 'UTC'). True fix, but prod-wide migration, needs reliable tz source, likely belongs to migration umbrella not this ticket.
- **B. Smarter display fallback** — resolve recipient tz from a better live source (setting service / CRM clinician.time_zone / clinic tz) at render time. Smaller, but still needs a source; doesn't fix portal or stored data.
- **C. Stop confidently-wrong display** when no real tz known (show UTC explicitly / block). Honest but poor UX.
- **BLOCKING QUESTION for Leo/PM**: where does each provider's true timezone actually live? (setting service? CRM `crm.clinician.time_zone`? user self-set?) Fix is undefined until answered.
