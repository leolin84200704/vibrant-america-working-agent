## Ticket Analysis

**Goal（目的）**：把 VP-15659 開發的「依 patient_id + 時間範圍取 HL7 lab content」read-only API 接上真實生產資料並驗證能取到真資料。當初開發時 prod DB / 真實 patient 資料尚未就緒，code 寫好但沒對真資料驗證過。本 ticket 是 verification + connection，不是新功能。

**Classification**：`emr-ops` | **Effort**：S | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：lis-backend-emr-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
採方案 A（verify-first，最小切片），但已修正 engineer 的兩個錯誤前提。

【改前現狀（已逐項在 repo 確認）】
- Endpoint POST /result/patient-hl7-content (result.controller.ts:185)，admin-only、read-only，回 JSON（不寫 sandbox 檔）。
- Service result.service.ts:225 getPatientHl7ContentReadOnly → grpc getPatientSamplesByTimeRange 取 sampleIds → 逐筆 generateResultHl7ContentReadOnly (result-generation.service.ts:512)。
- gRPC grpc-client.service.ts:1248 走 tryCloudThenOnPrem('sampleCloud','sample') cloud-primary+on-prem fallback。
- proto sample.proto:82 已有 GetPatientSamplesByTimeRange + req/resp (1033/1039) — coreSample 依賴 proto 側已交付。

【修正 engineer 兩點】
1. engineer 說『dev sample target 指不到 prod 資料、connect=改 target』為未經驗證之推測。實際上兩份 yaml 的 GRPC_SAMPLE_* 早已指向 live prod gRPC（on-prem 192.168.60.6:30276 primary + cloud 10.224.0.199:30276 fallback，與 SampleTestResult Steps1-4 同一組 prod 服務）。所以多半『不需要改任何 env/target』。真正缺的是：(a) 確認 prod coreSample server 端是否已 deploy GetPatientSamplesByTimeRange 實作（proto 有≠server 有，否則回 UNIMPLEMENTED）；(b) 用真實 patient 端到端驗證。
2. engineer 把 95s 效能風險完全劃為 over-engineering/另開 ticket — 但 95s 瓶頸（reference-range enrichment）就在本 endpoint 用的 generateHl7ContentReadOnly→generateHl7Content 路徑內，多 sample 的真實 patient 驗證一定會踩到，屬於本 ticket 的 verification edge case，不能完全忽略（至少要量測+記錄，超時才另開）。

【改後為何有效】connection 與 fallback 邏輯早已存在且已指向 prod；缺的只是『對真實 patient 端到端驗過 live 非 mock』。確認 prod coreSample 已 deploy 該 RPC + 用真 patient 驗證即滿足 ticket。

【具體改什麼】
1. 驗證為主（live 非 mock）：對一個有多筆 lab 的真實 prod patient（先確認 999997 testing account 對應的 patient_id）打 POST /result/patient-hl7-content，確認回真實 sample_ids + 非空 hl7_content，success_count 與 sample 數相符。
2. 僅在 prod target 真的不對時才改 GRPC_SAMPLE_CLOUD_HOST/PORT 或 GRPC_SAMPLE_*，且同 PR 更新 lis-emr-v2-config.yaml + lis-emr-v2-config-prod.yaml 兩份（IRON RULE，env 名見 grpc.config.ts:116-129）。預設不需改。
3. branch feature/leo/VP-16186；不碰 main/staging；如需改 config 才開 PR target stage_test。
4. 驗證結果同步給 QH-4590（is tested by）。

### 候選方案與正反取捨
- **方案 B — 順手補 prod readiness（明確 error、reference-range 95s timeout/批次化）**
  - 取捨：超出 'connect' scope，違反 Confirm-Before-Over-engineering；效能優化應另開 ticket。但若驗證時多 sample patient 真的超時到不可用，則此項從 nice-to-have 變成 blocker，需回報 PM 決定是否納入。
- **方案 C — 不只驗 read-only endpoint，連 VP-15659 真正的 Agent enrollment pipeline（generateHl7ForAgent + 寫 sandbox patient_xxx/reports/ 檔）一起接 prod 驗**
  - 取捨：更貼近 VP-15659 的 AC1-AC6（檔案落地 sandbox），但範圍變大且依賴 Composer/sandbox 端就緒；ticket 字面只講 'the connection'，建議先確認 PM 要的是 read-only endpoint 還是整條 pipeline。

### 風險與邊界案例
- prod coreSample server 端可能尚未 deploy GetPatientSamplesByTimeRange → 回 gRPC UNIMPLEMENTED，service.ts:243-258 未特別 catch UNIMPLEMENTED 會直接往上拋 500；非 BE 單方可解，需 coreSample team 確認 server 已上線
- 95s 慢查詢風險：endpoint 走 generateHl7ContentReadOnly→generateHl7Content 含 reference-range enrichment（VP-15659 comment：2 samples 跑 95s）。多筆 lab 的真實 patient 驗證極可能超時或被 ingress/gateway timeout 砍斷；驗證務必用多 sample patient 量測實際耗時
- patient_id vs customer_id 混淆：VP-15659 起初只有 GetCustomerSamplesByTimeRange，後補 patient 版。驗證時確認傳的是 patient_id 不是 customer_id，且 testing account 999997 對應的是哪個
- cloud-primary 失敗靜默 fallback on-prem：tryCloudThenOnPrem 在 cloud 192.168.60.6... 不通時會 fallback；驗證時要確認實際命中哪個 endpoint（peer-observed），不要只看 my-side 回 200 就當 cloud 通了
- 『無 lab 資料 patient』邊界：VP-15659 AC4 要求 graceful，read-only 路徑回 sample_ids=[]、success_count=0；需驗證不報錯
- admin-only auth：endpoint 在 controller:185-209 有 admin guard，驗證需用具 admin 權限的呼叫端，否則 403 會被誤判成 connection 問題
