# VP-17120 - Work Loop Record

> Created: 2026-06-19 23:55:00 UTC
> Status: active

---

## Ticket Analysis
- Self-created Bug (assignee Leo) while diagnosing why provider NPI 1396844346 (customer 4953) HL7 VACP orders could not be placed.
- Root cause of the order failure was a customer-resolution ambiguity (NPI had two LIVE+ordering ehr_integrations: 3057 newer vs 4953; resolveOrderingIntegration picked 3057 by typeRank→updated_at DESC, but the bundle was on 4953). Leo set 3057→PENDING; 16 stuck orders then placed manually.
- The deeper bug surfaced: those orders did NOT auto-recover after the config fix. emr-v2 marks parse failures terminal on the first attempt; Java did not.

## Decisions Made
- This is a regression from the EMR-Backend→emr-v2 migration (commit c1706b4, VP-16463), filed under migration umbrella as a Bug (per ticket-scope rule, not the originating order ticket).
- Restore Java `ParseOrder.updateHL7FileInput` semantics: parse_finished=true ONLY on success; failures decrement retry_num and stay retryable.
- Add the missing DB retry-rescan (Java STEP 7: re-query parse_finished=false AND retry_num>0, re-enqueue) to the fetch cron.
- ny_reject_code KEPT terminal (deliberate regulatory rejection, not config-fixable — retrying re-rejects). Deviates from "all failures retry"; intentional.
- ALL other parse_finished=false outcomes (error_codes, customer_not_found, parse-incomplete, sendorder-failed) must decrement retry_num — otherwise the new rescan re-enqueues them forever.

## Code Changes
- processor: error_codes → markFailure(tag, decrement, no terminal); markCustomerNotFound → decrement (was terminal); parse-incomplete → markFailure; sendorder-failed → retry_num decrement; success path → clear emr_code_not_found/customer_not_found; markFailure refactored to accept optional tags + log warn.
- fetch service: runIngestion returns +requeued; new requeueRetriableOrders() (parse_finished=false AND retry_num>0, take 500, jobId `hl7-{id}-r{retry_num}` for per-retry-level dedup, warn when capped). Early no-folders return also runs rescan.
- Tests: new hl7-order.processor.retry.spec.ts (4); fetch spec +2 (rescan re-enqueue + empty) + mock for hl7_file_input.findMany.

## Test Results
- Targeted jest: 15/15 pass. `npm run build` (nest build) clean. No CJK. Prod rescan-eligible count pre-deploy = 2 (no first-tick surge).
