## Ticket Analysis

**Goal（目的）**：將某位 Vibrant Clinical Team clinician（practice 150105）的 clinical consultation 收信 email 改為 caroline@doctorcaroline.com，使其後續 booking confirmation 與 reminder email 寄到正確信箱。這是一筆 prod data-fix，非 code change。

**Classification**：`calendar-clinical` | **Effort**：S | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2 (calendar PostgreSQL DB only; no code change)

### 建議方案（改前 / 改後為何有效 / 改什麼）
目的：修正 clinical-consult 通知信收件地址。改前現狀：clinical consultation 的收件 email 唯一來源是 calendar PostgreSQL DB `v2_calendar.calendar_owner_email`（prisma/schema.prisma:322，practice_id=150105）。VP-16413 STM 已逐行驗證：createEventByPatient→sendAppointmentScheduledEmailToNonPatients（event.service.ts:4116）與 reminder pipeline（reminder.service.ts:117/158/172）都讀此欄位；沒有獨立的 consultation-contact setting 表。既有 code 寫入路徑只有 self-service getOrCreateCalendar mutation，不提供 admin 改別人 email 的 endpoint。改後為何有效：直接改對正確 row 的 calendar_owner_email，event 與 reminder 兩條 pipeline 即 end-to-end 生效，無需 deploy。具體改什麼：(1) 先用 Zendesk request 2605196651 的 requester 取得該 clinician 的內部 calendar_owner_id（customer_id），用 `SELECT calendar_id, calendar_owner_id, calendar_owner_firstname, calendar_owner_lastname, calendar_owner_email FROM v2_calendar WHERE practice_id=150105 AND calendar_owner_id={ID}`（最穩定，因 @@unique([practice_id, calendar_owner_id]) 保證唯一一筆；若只有姓名/舊 email 則退而用 ILIKE 反查並回 Leo 確認）。(2) 回 Leo 確認是哪一筆後執行 `UPDATE v2_calendar SET calendar_owner_email='caroline@doctorcaroline.com', updated_at=now() WHERE calendar_id={X}`（單列、explicit ID bound）。(3) UPDATE 後 re-SELECT 該 row 100% 驗證 email 已更新。注意：updated_at 是 @default(now()) 而非 @updatedAt，任何寫入都不會自動更新此欄，故必須手動 SET，並非「raw SQL 繞過 Prisma @updatedAt」。先 branch / PR 規則不適用（無 code），但 prod DB 寫入須遵守 bound scope + 100% verify 鐵律。

### 候選方案與正反取捨
- **方案 A（推薦）：先用 Zendesk requester 取得 calendar_owner_id（或姓名/舊 email 反查），回 Leo 確認唯一 row 後單列 UPDATE WHERE calendar_id={X}，再 re-SELECT 驗證。**
  - 取捨：符合 prod 寫入鐵律與 @@unique 約束，避免改錯人；代價是需先向 Zendesk/Leo 取得 requester 識別線索，多一回合。
- **方案 B：直接 ILIKE 'caroline' 命中即改，不等確認。**
  - 取捨：快但違反 explicit-ID-bound 鐵律；practice 150105 為全 clinical team 共用，重名/撞錯人風險高，會把別人的信導向 Caroline。否決。
- **方案 C：寫 ad-hoc audit/update TS script（如 scripts/ 既有 repush 模式）。**
  - 取捨：可重用且有 log，但單筆 data-fix 過度工程；除非 Leo 要正式 audit 軌跡，否則不需要。

### 風險與邊界案例
- practice 150105 可能有多個 Caroline 或重名，純 name/email ILIKE 反查會撞錯人 — 必須用內部 calendar_owner_id 唯一定位並經 Leo 確認
- 2605196651 是 Zendesk request ID，不是 calendar_id/customer_id；無此映射無法安全定位 row
- 若該 clinician 在 150105 沒有 v2_calendar row（從未建過 calendar），UPDATE 會 0-row affected，需改走建 row 或回報 requester
- updated_at 為 @default(now()) 非 @updatedAt，忘記手動 SET 會留舊時間戳，遮蔽變更軌跡
- 改 email 影響 event confirmation 與 reminder 兩條 pipeline；舊 email 的在途/未來 reminder 是否需處理需確認（reminder claim 以 email 為 recipientKey）
- calendar_owner_email 與 LIS 其他系統（EMR/portal）的 email 設定不同步 — 確認 scope 是否僅此一欄
