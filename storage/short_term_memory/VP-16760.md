## Lessons Learned

### [2026-05-27]
1. **`ehr_vendor_inquiry` = 「Not on the list」vendor request 表**（VP-14787 建 create/list/get；VP-16760 補 approve/reject/update/history + audit）。Provider Portal 選「Not on the list」就打 `POST .../vendor-inquiries`。status: PENDING_REVIEW→APPROVED/REJECTED。**純需求追蹤，approve 不建 EhrVendor/EhrIntegration、不寄信**。Sales review queue = `GET .../vendor-inquiries?status=PENDING_REVIEW`。
2. **「Every resolution action writes an audit log entry」型 AC → append-only `*_status_history` 表**（mirror `EhrIntegrationStatusHistory`：inquiry_id FK / from_status / to_status / reason / changed_by / created_at），且**與 status change 同一個 `$transaction`**。inline 的 reviewed_by/reviewed_at/reason 只是最新狀態、會被覆蓋，不能當 audit trail。
3. **VP-16163 (EHR Integration V2) epic ticket：body/AC 常從 quarantine PRD 段誤抄，title 才準**。VP-16629 + VP-16760 兩例確認。disambiguation 訊號：title + `split from` link + reporter/Leo comment。先信 title，跟 Leo 確認，別照 body 硬做 quarantine。
4. **同一功能的 internal gate 要一致**：approve/reject 用 `internal_user_role`，則 list「跨 customer 看全部」也要放寬到 `internal_user_role`（原本只 isAdmin）→ 否則 Sales 能 approve 卻看不到待審佇列。
5. **Step 4 propose 方案前先 grep 既有承載表**：別預設要新建 schema；`ehr_vendor_inquiry` 早在 schema 裡，能省掉自建 quarantine 表那條路。
