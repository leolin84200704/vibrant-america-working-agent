### 根因（已用程式碼確認）
EMR customer-pay 收費 = call `https://api.vibrant-wellness.com/v1/charging/transaction/pay`。
- **Java(reference)** `ParseHL7.java:992-998`：`new TransactionPayInput()` 靠**欄位 initializer 預設值**帶 `currency="usd"` / `charge_type="testorder"` / `type="card"` / **`token_platform="stax"`** / `payment_source="emr"` / `new_sample=true`，只顯式 set `account_id/account_type/amount/type/payment_token/customer_token`。
- **emr-v2(VP-16463 Stage 3b port)** `order-finalizer.service.ts:78-89`：inline 建 input，只給 `account_id/account_type/amount/type/payment_token/customer_token`。**TS interface 不帶 runtime 預設** → 漏送 `currency / charge_type / token_platform / payment_source / new_sample`。charging API 缺 `token_platform` → 收費未成立。
- DTO 檔 `dto/payment.dto.ts` 已定義 `TRANSACTION_PAY_DEFAULTS`（含 token_platform:'stax'…）+ 註解「callers should rely on those defaults」——但 caller 根本沒 spread 它。經典「defaults const 建了但 caller 忘了用」。
- grep 確認：emr-v2 只有 order-finalizer 這一個 caller 建 TransactionPayInput。

### 影響範圍
- emr-v2 走 customer-pay 的 EMR order（VP-16463 cutover 後改走 emr-v2 的那批 client）自切換起都漏收費。
- 需 remediation：已受影響 order（2605216847/2605196560 + 同期）要手動補收（finance/Fangyuan），非程式碼能補。

## Approaches Considered

### [2026-05-27]
**A（推薦）— spread `TRANSACTION_PAY_DEFAULTS`**（1:1 還原 Java 預設）
```ts

    ...TRANSACTION_PAY_DEFAULTS,           // currency/charge_type/type/token_platform/payment_source/new_sample

```

**C — 在 DTO/service 層給 runtime 預設（factory function）**：較大改動；A 已足夠。除非有第二 caller 才值得。

**推薦 A**。Step 4 暫停等 Leo 確認（含 remediation 要不要我撈受影響 order 清單）。

## Lessons Learned

### [2026-05-27]
1. **Java field-initializer 預設值不會 port 到 TS interface**（bug class）：Java `private String x = "stax"` 在 TS interface 是 `x?: string`，無 runtime 預設。port Java model 時，每個 construction site 都要明寫預設（或 spread 一個 defaults const），並 grep 所有 caller 確認。VP-16777：`TransactionPayInput` 6 個預設（token_platform="stax"/currency/charge_type/type/payment_source/new_sample）只在 DTO 檔做了 `TRANSACTION_PAY_DEFAULTS` const，但 caller 沒 spread → charging API 缺 token_platform → 卡不被收。
2. **EMR customer-pay 收費流程（emr-v2）**：`order-finalizer.service.ts`（charge branch，customerPay + 有 stax method 才收）→ `ChargeClientService.getFirstPaymentMethod` + `transactionPay` → `api.vibrant-wellness.com/v1/charging/{paymentMethod/allSharedPaymentMethods, transaction/pay}`。Java reference = `ParseHL7.java:988-1006` + `ChargeClient.java` + models `TransactionPayInput`/`StaxPaymentMethod`。Auth header 無 "Bearer " prefix（Java quirk，已保留）。
3. **emr-v2 不 durably 存 per-order 收費結果**：`order_intake_records`（dormant，近 10 天 0 筆）+ `emr_sample` 都不可靠當收費查詢源。要查某 EMR order 的收費/payment 狀態 → 上游 LIS-core / charging 系統，不是 emr-v2 DB。
