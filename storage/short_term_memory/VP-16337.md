## Ticket Analysis

### [2026-04-27 23:28]
**Background:**
- Epic-level goal: retire `order_clients`, drive incoming HL7 orders entirely from `ehr_integrations` in lis-emr-v2; EMR-Backend (Java legacy) gets decommissioned.
- Phase 0 = enable a new lookup path needed by Phase 2: `customer_npi → clinic_ids` via coreSamples RPC, so we can match HL7 orders to clinic-level integrations.

**Why this RPC:**
- HL7 incoming files always carry `customer_npi` (Type 1, individual provider). Vendors do not send `clinic_npi`.
- Provider-level match (`ehr_integrations.customer_npi = ?`) covers most cases.
- Miss → call `GetClinicIDsByNPINumber(npi)` → list of clinic_ids that the NPI's customer belongs to → check if any of those clinics has clinic-level integration (`customer_id = '-1' AND clinic_id IN (...) AND ordering_enabled = 1`).

**Codebase facts (from explore):**
- coreSamples already has the RPC at `proto/customer.proto:79-92` (returns `repeated int32 clinic_ids`, no clinic active/deleted filter).
- lis-emr-v2 `src/proto/customer.proto:75` only has `GetCustomerByNPINumber` — the new RPC must be added.
- lis-emr-v2 has both `src/proto/` and `dist/proto/` directories — keep them in sync.
- Existing gRPC client wrapper: `src/modules/grpc/services/grpc-client-v2.service.ts` (already connects to coreSamples; uses provider-style DI, no `ClientsModule.register`).
- `ehr_integrations` schema already has `customer_npi`, `clinic_npi`, `effective_npi`, `ordering_enabled`, nullable `clinic_id` — no migration needed for Phase 2.
- Clinic-level marker convention is `customer_id = '-1'` (from `result-generation.service.ts:744-762`); Leo confirmed sticking with this (option A).

## Approaches Considered

### [2026-04-27 23:28]
**NPI matching strategy (resolved with Leo):**
- A. Single query on `effective_npi` — initially recommended, but rejected.
- B. Two-stage fallback `customer_npi` → `clinic_npi` — rejected.
- **C (final, per Leo):** Direct `WHERE customer_npi = ?`. Vendors never send `clinic_npi`, so the fallback was over-engineered. `clinic_npi` column stays untouched (other services may still use it).

**Clinic-level marker (resolved with Leo):**
- A. `customer_id = '-1'` magic value — chosen (matches existing `result-generation` convention, no schema changes).

**Phase 0 scope (this ticket):**
- Sync proto, add gRPC client wrapper, write verify script, run against prod coreSamples (Leo OK'd prod for read-only verification).
- Out of scope: DB writes, Phase 1+ ordering pipeline, schema migrations.

## Failures

### [2026-04-27 23:38]
**Mistake:** Initially added the new RPC + messages to `src/proto/customer.proto` and `dist/proto/customer.proto`.

**Root cause:** Two parallel proto trees exist in this repo:
- `src/proto/customer.proto` — `package lis;`, used by v1 client connecting to legacy LIS gRPC `192.168.60.6:30276` (`grpcConfig.customer`)
- `src/proto-v2/customer.proto` — `package coresamples_service;`, used by v2 client connecting to coreSamples `10.224.0.199:32100` (`grpcConfigV2.customer`)

`GetClinicIDsByNPINumber` lives only in coreSamples — must go into proto-v2.

**Recovery:** Reverted both `src/proto/` and `dist/proto/` edits, then applied the changes to `src/proto-v2/customer.proto` only (no `dist/proto-v2/` exists, so single file).

**Detection trigger:** Reading `src/config/grpc.config.ts` for endpoint info — saw `getProtoV2Path` and `package: 'coresamples_service'` for the v2 customer client.

## Retrospective
## Lessons Learned

### [2026-04-27 23:45]
**lis-backend-emr-v2 has TWO parallel proto trees** — always confirm which gRPC server hosts the RPC before editing:

| Path | Package | Host (default) | Hosted by |
|------|---------|----------------|-----------|
| `src/proto/*.proto` + `dist/proto/*.proto` | `lis` | `192.168.60.6:30276` | legacy LIS service (lis-pkg) |
| `src/proto-v2/*.proto` (no dist mirror) | `coresamples_service` | `10.224.0.199:32100` | coreSamples (Go) |

Decision rule: if the upstream RPC source is `LIS-backend-v2-coreSamples` (Go repo with `package coresamples_service`), edit `src/proto-v2/`. If it's the legacy `LIS-transformer` family, edit `src/proto/`.

**Connectivity:** Both `192.168.60.6:30276` and `10.224.0.199:32100` are Azure private IPs but reachable from Leo's machine over VPN — no need to run scripts inside the cluster.

**`/scripts/` is gitignored** — new standalone test/verify scripts won't be tracked unless `git add -f`. Existing ~1396 tracked scripts predate the rule. Must decide per-ticket whether the script should ship with the change or stay local.
