## Code Changes

### [2026-05-28] PR #135 `bugfix/leo/v2-fetch-hang-timeout`
`Hl7OrderFetchService.handleCron` 改：
- `DEFAULT_MAX_RUN_DURATION_MS = 13 min`（原無、< 15min cron interval）
- `DEFAULT_PER_FOLDER_TIMEOUT_MS = 2 min`
- `runWithTimeout(op, ms, label)` helper — `Promise.race(op, setTimeout(reject FetchTimeoutError))`
- `forceReleaseTimer` 雙保險：即使 `Promise.race` 失靈，hard deadline 也強制 release lock
- `releaseOnce` dedupe guard：force-release timer 跟 finally 兩條 path 都會試 release、保證只呼叫一次
- Per-folder timeout fire 時 force `sftpConn.disconnect()` 清 shared client state
- 兩個 timeout 都可 env override (`HL7_FETCH_MAX_RUN_MS` / `HL7_FETCH_FOLDER_TIMEOUT_MS`)

新 spec `hl7-order-fetch.service.spec.ts` 9 tests：lock lifecycle (skip / happy / timeout / non-timeout error)、runIngestion folder-level timeout (hang folder isolation)、runWithTimeout helper edge cases。Full hl7-order-processing suite 46/46 pass、`tsc` + `nest build` 0 error。

## Retrospective

### [2026-05-28 22:10 UTC]

**初始假設 vs 結果**
- 我先假設 hang 元凶是某個 vendor SFTP dead host → **錯**。實際是 (a) ssh2-sftp-client `client.list('.')` validation step 在 MDHQ home dir 有 172 個 subdir 時偶發慢 + (b) lock-leak 真正 root cause 是 code 缺 timeout 不是 host 問題
- 我假設 folder 260 是 deterministic hang → **錯**。22:00 tick 同 folder 順利通過。是 transient。

**真正 root cause 鏈**
1. **5/27 18:09 UTC** Leo `UPDATE sftp_folder_mapping SET use_v2_pipeline=1` (196 row 全開) — 把 24 個從未跑過 v2 的 vendor host 第一次拉進 v2 fetch loop
2. **5/27 22:48 UTC** 舊 v2 pod 最後一次 fetch 成功（cutover 後跑了 4.5h 沒事，是 transient slowness 還沒打中）
3. **5/27 23:54:59 UTC** 兩個 v2 pod 同時被 deploy/replace（不是 restart、是新 pod 從 0 起、`Restart Count: 0`）
4. **5/28 00:00 UTC** 新 pod 第一個 cron tick 跑 fresh sequential 掃 196 folder，撞上某個瞬間 MDHQ host 慢、`client.list('.')` validation 卡住 → `runIngestion` await 永遠不回 → `finally` 不跑 → `extendTimer` 持續 extend lock → **lock leak**
5. **5/28 00:15 onwards** 每 15min cron tick 都看到「lock held」skip。**19h 零 fetch**
6. **5/28 18:25 UTC** Leo 通報

**為什麼 PR #135 雙層 timeout 是對的 fix**
- Per-folder timeout (2min)：把 transient slowness 從「卡死整 run」降級到「單 folder 損 2min」
- runIngestion 級 hard timeout (13min)：防 runWithTimeout 本身失靈 + bound 整 tick 時間 < 15min cron interval
- forceReleaseTimer + releaseOnce：再失靈一次也保 lock 一定 release

**信心度**：4/5
- Code-level fix 完整 (9 specs + 同套 lesson 已在 INCIDENT-20260518 BullMQ 寫過、這次補到 redlock cron handler)
- Diagnostic 過程兩次走錯路（dead-host 假設、folder 260 deterministic 假設）— 都因為沒先用真實證據驗
- Evidence preservation 失誤 — restart 前沒 dump log

## Lessons Learned

### [2026-05-28] 寫進 LTM 的
1. **Cron handler + Redlock auto-extend 必須配 hard timeout** — INCIDENT-20260518 已有 BullMQ 版、這次擴展到 redlock-wrapped cron handler（不同 code path 同類 bug）→ 寫 patterns.md
2. **`SftpConnectionService` 是 singleton、shared `this.client`，每個 processFolder 都 reconnect** — MDHQ 一台 host 上有 172 folder → 7min 內 172 次 connect、`client.list('.')` validation 在 home dir 大 (172 subdir) 時是性能熱點。長期解：folder by-host grouping、共用 connection → 寫 emr-integration.md（建議性 follow-up）
3. **`emr_sftp_source.port` 必查、不 hardcode 22** — VP-16180 早就標 PF=2222、本次又重踩 → 寫 patterns.md
4. **Hang pod 必先 preserve evidence 再 restart** — `kubectl logs > /tmp/preserve.log` + `describe > /tmp/preserve_describe.txt`。否則 root cause 證據隨 pod GC 永久遺失 → 寫 user memory feedback
5. **`lis-emr-v2` migration apply 兩個 DB** (`192.168.60.11` + `lisportalprod2`) — patterns.md 304-308 早就寫、本次第三次踩、改寫成更醒目的 alert（同類 incident 重複 = patterns.md 規則不夠強）

### [2026-05-28] STM-only（不夠泛化）
- Hang folder 抓不到「具體哪個」時，可以 deploy 帶 per-folder timeout 的 fix、然後正向重 flip、log 會印出 `Folder id=X emr=Y exceeded`。本案 21:45 tick id=260 + 22:00 tick 通過 → 證實 transient + isolate
- `lis-emr-v2-deployment` (non-prod) + `lis-emr-v2-deployment-prod` 都釘 appserver04、共用同台 node 的 hostPath PV `/EMR_storage`，但**各自有獨立 redis sidecar** → 兩邊 cron 不互鎖（per-pod sidecar redlock 不是真 distributed lock）
- 14 個 keep-list verified cutover row：67 (OPTIMANTRA), 78/83/84/86/87/145/201 (MDHQ batch 1 + 1.5), 154/183/252/256/258/275 (MDHQ Cerbo template, sftp_source_id=3)
