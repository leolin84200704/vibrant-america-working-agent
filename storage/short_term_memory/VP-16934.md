### [2026-06-08] VP-16938 exactly-once 設計方向（已與 Leo 定）
**決策：exactly-once 靠「刷卡前先寫唯一鍵（intake 表 unique on placer/order id）」，不是靠關閉 retry。**

Leo 問「能不能不要 retry、只有成功/失敗」。結論：不 retry 只解一半、且更糟，不採為主解：
- **重複扣款有兩源**：(1) **vendor 自己重送**（同步 API 回程斷線時 EMR 會 resend，我們控制不了）—— 不 retry 也擋不住；(2) 刷卡成功但 emr_sample 寫入失敗 —— 不 retry 雖不會再刷，但留下「**扣了卡卻沒單**」的孤兒狀態且無自動修復，醫療收費情境更難善後。
- **正解**：收單、刷卡**之前**先寫一筆 intake row（placer/order id 設 UNIQUE）。重送/重跑同一張單 → 撞 UNIQUE → 直接回上次結果、不再刷。與 retry 開關無關，且有此鍵後可安心保留 retry 處理暫時性抖動。

**能否在 lis-backend-emr-v2 自己做 idempotency？**
- **App 層去重（intake 表 + 刷卡前 unique key + 短路）= 可以，完全在本 repo 做**：本 repo 有自己的 Prisma/MySQL(lis_emr)，新增 intake 表、charge 前寫入並短路即可，**不依賴 charging team**。涵蓋 vendor 重送 + 我方重跑。
- **唯一殘留小破口**：寫 intake → call charge → charge 已成功但「我方在記錄結果前 crash」這個視窗，retry 時不確定那次 charge 到底過了沒。要 100% 關掉這個視窗需要其一：
  1. charging wrapper 願意接受/轉發 `Idempotency-Key` 給 Stax（**需 charging team，非本 repo 能單獨決定**）；或
  2. 本 repo 向 charging 服務 query-back「這筆 charge 成功了嗎」再決定；或
  3. 接受這罕見 ambiguous case 進**人工 reconciliation/manual-release 佇列**（emr-v2 可自己做）。
- **結論**：**主防線（app 層 unique key）可以、且應該在本 repo 自己做**；gateway 級真 idempotency-key 不能單靠本 repo（要 charging team），但有了 app 層去重後它只是「錦上添花」，非必要。
- **prod 注意**：emr-v2 prod 無 `_prisma_migrations`，新增 intake 表要 manual SQL apply + `SHOW COLUMNS` 驗（見 repos.md）。新 `process.env.X` 同 PR 更新兩份 config-yaml。
