### [2026-05-13 20:30] 完整收尾回顧

**最初範圍**：把 EMR-Backend Java 的 order intake port 到 lis-backend-emr-v2，包含 4 個 batch 的 SFTP folder cutover、Phase B' shadow-compare 29 fixture byte-identical。

**實際路徑**（vs 初始計畫）：
1. ✅ A1~A5e port + 29 fixture shadow-compare 全過、Phase B' merge 上 staging
2. ✅ batch 1 cutover (id=83/84/86/87 MDHQ)、設 hourly monitor
3. ✅ batch 1.5 expansion (id=78/145/201) — 因 batch 1 流量太低、user 授權擴大
4. ❌ **6053/6058 兩個檔被標 NO_NPI、卡 DB**（red flag at H+21）→ 引出後續 5 層 bug
5. ✅ Bug 修正 PR #104（5 commit、含 NO_NPI parity + Hl7Decoder line ending bug + OML normalize + PVC mount + Jenkinsfile auto-apply）
6. ✅ 6053/6058 backup 找回（在 prod pod /tmp 內、PVC 還沒掛時的 ephemeral 殘留）、用戶重 push SFTP → v2 fetch → v1 race 處理（dedup 保護下出 sample_id 2557536/2557539）
7. ✅ v1 Java image rebuild、honor VP-15460 use_v2_pipeline filter

**5 層 bug（一個 NO_NPI 揭發的）**：
1. (表層) `markCustomerNotFound` 沒 set `parse_finished=true` → BullMQ 一直 retry terminal state
2. (語意層) NPI hard-gate 在 processor 加了 Java 沒有的前置條件 → v1 不會 reject 的檔 v2 reject
3. (decoder 層) `Hl7DecoderService.parseSegments` 用 `.split("\r")` → MDHQ vendor 的 `\n`-only HL7 沒拆 segment → ORC 找不到 → NPI 為空 → trigger NO_NPI gate
4. (持久化層) `HL7_LOCAL_ROOT` 預設 `/tmp/hl7` 是 pod ephemeral → SFTP `deleteFile` 又在 download 後立即執行 → pod 重啟 = 檔案永久失蹤、SFTP 也沒了（VP-15460 的 PVC mount 沒套用）
5. (deploy 層) deployment yaml 沒掛 PVC、image registry 寫 Azure ACR（cluster 拉不到、無 imagePullSecret）、Jenkinsfile 不自動 apply deployment yaml → repo yaml ≠ cluster reality 多月 drift

**結果比較**（初始假設 vs 實際）：
- 假設：只是 NO_NPI gate 太嚴、放鬆即可
- 實際：放鬆 gate 後新檔還是會被舊版 Hl7DecoderService 拆錯 → 還是會走錯路徑、只是 fail mode 不同
- 真正 root cause 是 decoder line ending bug
- 又因 `/tmp` ephemeral + SFTP delete 立即執行 + pod 跨節點問題 = 已下載檔案丟失

**失敗 root cause**：
- VP-16463 phase A~B' 時、我**完全沒讀 `EMR-Backend/deployment_prod.yml`**、不知道 v1 用 PVC `emr-pv-claim` 掛 `/EMR_storage`、不知道 v1 釘 `nodeName: appserver04`、不知道 v1 用 `FOLDER_PATH` env
- 因此 v2 fetch service 預設 `/tmp/hl7` 看起來沒問題、Phase B' 跑得過（因為 fixture 是 file path、不經 SFTP fetch service）
- 部署到 prod 才爆炸：production SFTP fetch + ephemeral storage = 雙失

**模式識別（這次新發現的）**：
- 「Code review pass」不等於「deploy ready」— infra 層 parity（volume mount、env var、node placement）跟 code 層 parity 同等重要
- `default` / `fallback` 值在 prod 環境必須當作 prod 行為來檢視、不是「dev 才會 hit」
- 「上游刪除 + 下游處理」flow 任何中間 crash 都會雙失、必須延後刪除或加 idempotency
- Per-pod sidecar Redis + Redlock 看起來像 distributed lock 但完全不是
- Container restart 跟 pod restart 是不同事、`kubectl get pods` RESTARTS 欄位容易誤導
- Multi-namespace 殭屍 pod 會偷做事不被發現（staging-ns 那隻 13d CrashLoop）

**信心度**：3 / 5
- code-level Java parity 沒問題（5）
- infra / k8s 推理太弱（2、撞 5 個獨立 bug）
- 修完之後驗證流程紮實（4）
- 跟 user 溝通的時候有過度 over-promise 「PR 改好了可以 deploy」、實際 PR 沒包 PVC mount + Jenkins auto-apply、deploy 才發現整路斷掉

## Lessons Learned

1. **Port 既有系統時、infra config 跟 code 同等重要、必查 v1 deployment yaml** — 已寫入 LTM patterns.md
2. **改 K8s yaml 前必 `kubectl diff` 抓 drift** — 已寫入 LTM
3. **CI/CD 完整鏈路理清（branch → image → registry → cluster pull → ConfigMap source）才能改 yaml** — 已寫入 LTM
4. **Multi-deployment 共用外部資源前盤點所有 worker** — Per-pod Redis Redlock 不能跨 pod、已寫入 LTM
5. **Container restart ≠ Pod restart** — 已寫入 LTM
6. **Java → TS port 的 6 條 parity rule**（terminal state parse_finished、不加前置 gate、segment terminator 多 variant、OML normalize、null serialization、Set 順序）— 已寫入 LTM
7. **多 namespace audit (`kubectl get pods -A | grep`) 必做** — 已寫入 LTM
8. **`/tmp` 對 prod 是 anti-pattern**、任何 fallback 預設值要當 prod 行為檢視 — 已寫入 LTM
9. **上游刪除時機：永遠延後到 local 持久化 + 處理完成** — 已寫入 LTM
