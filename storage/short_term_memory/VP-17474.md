# VP-17474 — "View Report" deep link access error (alleged quoted-printable issue)

## Diagnosis chain (2026-07-21)

1. **Ticket claim**: QP encoding (`=` -> `=3D`, soft line breaks) corrupts the deep-link URL -> provider sees "This account doesn't have access to this report." Example order 2606306611.
2. **FE outcome mapping** (va-portal `signIn.js handleReportUrl`): that exact message maps ONLY to resolve outcome `access_denied`. `invalid` (token not found — what a mangled token would produce) shows "This link is invalid...". => QP theory structurally inconsistent with the symptom.
3. **Resolve semantics** (LIS-transformer `trans-reports.controller.ts` VP-16859): access_denied requires token FOUND + not expired + recipient_customer_id != login customer_id.
4. **Prod resolve logs** (AKS lis-trans-deployment, tag deep_link_resolve, 44h window): 18 calls total, ALL 2026-07-21 22:27–23:23 UTC (= post-ticket debugging session). 9 access_denied: 999997 (Leo test acct) x4, 43416 (Nirusha Kumaran, "Internal special NPI") x1, login_customer_id="null" x4 — all targeting recipient 28976 (Bradly Holland). 8 resolved incl. 28976->28976 twice and 22515->22515 (the ticket's order) at 23:23Z. **True recipients succeed; non-recipients are denied by design.**
5. **Raw email dumps** (Postmark server e4e352..., tag result_ready_new, ~650/day): both 7/21 email (token i7NhiW... = exact DB row for sample 2587895 / accession 2606306611 / recipient 22515) and 7/15 email show well-formed QP; href decodes to
   `https://portal.vibrant-wellness.com/#/login?report_url=https://portal.vibrant-wellness.com/r/<24-char-token>`. No corruption.
6. **Structural finding**: recipient addresses are commonly clinic shared inboxes (office@, info@ — setting-consumer getCustomerEmailBoth returns clinic-level email), while the token is bound to ONE provider customer_id. Clinic staff / clinic-role logins (JWT customer_id=null — matches the "null" log entries; va-portal has user_roles "clinic" accounts) opening the email => access_denied BY DESIGN (VP-16859 C2 anti-forwarding compliance).

## Facts / infra notes
- report_email_tokens: 21,448 rows since 2026-06-05, ~500-1100/day; order 2606306611 -> sample 2587895, customer 22515 (MEGAN LYONS), single token row created 2026-07-21.
- Deep-link href present in template since at least 7/15. AKS trans pods (44h uptime) show zero resolve traffic before 7/21 22:27Z — earlier traffic likely went to on-prem trans-service pre-AKS-migration (va-portal 3.91.0 released 7/20 included migrate/trans-service-to-aks); logs unavailable. 401s at the JWT guard produce NO deep_link_resolve log line (guard throws before logging).
- Postmark server token available from noti/notification-center-deployment pod env (POSTMARK_KEY). Recipient search + /dump endpoint give raw MIME.
- lis_frontend_service DB reachable locally via setting-consumer .env creds + emr-v2's mysql2 (lis_core_emr Azure acct cannot see that schema).

## Conclusion
Not an encoding bug. The deep link works end-to-end for the intended recipient. "Access denied" comes from non-recipient logins — mostly clinic shared-inbox workflows and internal/test accounts (the internal repro itself gets denied by design, which likely reinforced the bug report). Fix is a product/compliance decision (see options in report), not a QP patch.
