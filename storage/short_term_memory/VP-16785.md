## Ticket Analysis

**Goal（目的）**：把 IssueService gRPC client 從 on-prem ISSUE_RPC (192.168.60.6:30071) 重新指向 cloud in-cluster endpoint lis-issue-system-service.issue.svc.cluster.local:30071，移除 on-prem 依賴。AC1 cloud 連通、AC2 移除 on-prem ref、AC3 既有 issue 相關 flow regression 通過。真正改動 repo 是 LIS-transformer-v2（非 lis-backend-emr-v2，EMR repo 對此 RPC 無 client code，為 runtime no-op）。

**Classification**：`rpc-cloud-migration` | **Effort**：M | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
【目的】見 goal。【改前現狀（已實地查證）】LIS-transformer-v2 src/grpc.option.ts:95 microserviceOptions_issue 讀 process.env.ISSUE_RPC（on-prem）。共有 8 個 @Client(microserviceOptions_issue) caller（engineer 誤報 6 個，且部分檔名路徑寫錯）：src/trans/merchandise.service.ts、src/trans/utility.service.ts、src/trans/setting.service.ts、src/trans/services/PNS.service.ts、src/trans/services/utility.api.service.ts、src/trans/services/patientProfile.service.ts、src/trans/services/service.ts、src/utility/utility.service.ts。每個 caller 在 onModuleInit 用 this.client5.getService('IssueService')，呼叫 queryForeignLink / getIssues。grpc.option.ts:267 microserviceOptions_issue_cloud（讀 ISSUE_RPC_CLOUD）已定義但零 caller 引用（dangling stub，VP-16685 留下）。yaml/st.yml:149 + yaml/prod.yml:155 只有 ISSUE_RPC，無 _CLOUD（注意 key 有 trailing whitespace 'ISSUE_RPC '）。既有 src/calendar/shared/with-cloud-fallback.util.ts（VP-16685 已 merge、Leo 驗收過）提供 withCloudFallback(label, primary, fallback, logger)：cloud-primary、僅 transient gRPC error（DEADLINE_EXCEEDED/UNAVAILABLE/UNKNOWN/INTERNAL + 'not initialized/available'）才 fallback，business error 直接 throw，並有 GRPC_CLOUD_FALLBACK_ENABLED=false kill switch。issue 兩個 client 共用同一 proto protos/issue-system-protos/issue_service.proto（proto byte-identical，marshalling parity 風險低）。【改後為何有效】沿用 VP-16685 已驗收的 cloud-first + on-prem-fallback：cloud 正常走 cloud（AC1），on-prem 過渡期 cluster DNS NXDOMAIN 時 transient fallback 不中斷（AC3），啟用既有 dangling _issue_cloud stub。【具體改什麼】base branch stage_test，branch feature/leo/VP-16785。(1) 8 個 caller 各 import microserviceOptions_issue_cloud + withCloudFallback，加第二個 @Client(microserviceOptions_issue_cloud) private clientXcloud，onModuleInit 多一個 this.issueServiceCloud = clientXcloud?.getService('IssueService')（仿 customer.service.ts:60-62 的 null-guard 寫法）。(2) 每個 queryForeignLink / getIssues call site 包成 this.issueServiceCloud ? withCloudFallback('IssueService.queryForeignLink', ()=>lastValueFrom(cloud...), ()=>lastValueFrom(onprem...), {warn}) : 原 on-prem call（仿 customer.service.ts:74-81）。(3) yaml/st.yml + yaml/prod.yml 新增 ISSUE_RPC_CLOUD env（沿用既有 valueFrom configMapKeyRef pattern，對應 ConfigMap key）= lis-issue-system-service.issue.svc.cluster.local:30071。(4) 因新增 process.env.ISSUE_RPC_CLOUD → 鐵律：同 PR 同步更新兩份 yaml 並明說。(5) 切 branch 後先 npx prisma generate 兩份 schema（prisma + prisma2，VP-16521 教訓），npm run start:dev + npm run build 必過。【先確認 AC2 語意再執行】見 clarifyingQuestions。

### 候選方案與正反取捨
- **方案 A — 直接換 URL／刪 on-prem option（字面滿足 AC2 hard remove）**
  - 取捨：完全契合 AC2「fully remove on-prem ref」且改動最集中；但無 fallback，若 transformer-v2 prod pod 未與 issue service 同 cluster／.svc.cluster.local NXDOMAIN（VP-16784-87 STM Lesson 5 已驗證 cluster 外 NXDOMAIN），IssueService 8 caller 全掛（billing/skincare 驗證、PNS issue 查詢全爆），違反 VP-16685 安全 pattern。除非 PM 確認 transformer-v2 已在 issue service 同 cluster 才可採。
- **方案 C — 最小可獨立交付片：只先包 PNS + utility.api 兩個流量最大的 caller 當 pilot，其餘 6 個下一個 PR**
  - 取捨：Leo 偏好 thinnest slice 先交（ticket-routing.md PM 慣例）；風險最低、可先驗 cloud 連通；但 AC1「all relevant code paths」未一次到位，需第二 PR 收尾，且 8 caller 同構、分批反而增加 review 次數。

### 風險與邊界案例
- caller 數實為 8 不是 engineer 講的 6 — 漏改任一個 caller 會留 on-prem 殘留，違反 AC2；逐一 audit 8 個檔案確認全部包到。
- issue service 用 getService('IssueService') 動態拿 service，不像 customer.service 的單一 typed method；每個 caller 要在 onModuleInit 多 wire 一個 cloud service handle + null-guard，比 customer 範例多 boilerplate，容易漏 null check 造成 CLI/standalone context crash。
- NXDOMAIN：transformer-v2 prod pod 若不在 issue service 同 cluster，方案 B 靠 transient fallback 撐住但每次 cloud call 都先吃一次失敗 latency；方案 A 會直接全掛。必須先確認 pod cluster 位置。
- yaml key 'ISSUE_RPC ' 有 trailing whitespace，新增 ISSUE_RPC_CLOUD 時別複製到 trailing space，且確認 Azure AKS ConfigMap 真 source of truth 已建對應 key（EMR repo 經驗：yaml 在 repo 但真值在 Azure ConfigMap，pipeline 會覆蓋）。
- queryForeignLink 回空 foreign_links 是正常 business 結果（PNS.service.ts:538 isEmpty 判斷），不可被 fallback 當 transient error 重試 — withCloudFallback 只對 transient code 重試，business 結果照 propagate，符合需求，但要驗 cloud 真的回相同空結果語意。
- getIssues 批次帶 issue_ids，cloud/on-prem 回傳順序或欄位若有差異會造成 PNS timeline 顯示差 — 需 end-to-end diff 前後 output（鐵律 v2 parity）。
- 本張 issue image SHA-pinned（issue_system:c0f8779）風險最低可當 pilot；但同 epic 的 dashboard image 是 mutable :latest、audit image 有 _cloud suffix 是分支 build proto 不可假設同步 — 若四張合 PR，dashboard/audit 要額外驗 proto byte-equivalence。
