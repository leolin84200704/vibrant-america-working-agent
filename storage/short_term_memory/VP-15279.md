## Ticket Analysis

**Goal（目的）**：讓 Staff+Delegate 員工代 provider 操作時，系統能記錄 provider-of-record 與實際操作人，並用 delegation link 驗證授權（audit / attribution，AC4 防止 staff 用自己身分下單）。但依 Leo 2026-03-04 comment，VP-15279 的實際交付邊界已被縮窄為「coreSamples delegation table + RPC CRUD」，description 的 orders-table 加欄位/display/audit AC（AC1/AC2/AC5/AC6）已不屬本票。

**Classification**：`practice-integration-epic` | **Effort**：S | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2, LIS-backend-v2-coreSamples

### 建議方案（改前 / 改後為何有效 / 改什麼）
改前現狀（已驗證）：(1) delegation table + DelegationService RPC（CreateDelegationLink / GetStaffsAllowedProviders / ValidateDelegationLink 回 delegation_link_id / DeleteSingle / DeleteAll / DeleteAllByProvider + Disclaimer CRUD）已在 coreSamples(Go) 實作完成（git log 顯示 VP-15314 delegation edge cases 已落地），proto 在 LIS-transformer-v2/protos2/coresamplev2/delegation_service.proto 且確實被 main.proto:20 import 進 bundle。(2) transformer-v2 端尚未接線：src/grpc.option.ts 的 coresamplev2 ClientOptions 載入 main.proto（含 delegation），但 src/role/role.service.ts:269-271 與 334-336 是兩個未實作 TODO（assign/remove DELEGATE 角色時應 Create/DeleteSingleDelegation）。(3) orders 端 attribution 欄位（entered_by_user_id / acting_provider_id / delegation_link_id）在 transformer-v2 src grep = 0 筆；order_service.proto 只有 provider_id，無 entered_by。\n\n改後為何有效：本票既在 QA Review 且 RPC CRUD 已落地，正確做法是先與 Leo/PM 對齊「VP-15279 = delegation table + RPC CRUD」這個縮窄 scope 即為驗收標準，並把 description 的 orders-schema/display/audit AC 移到下單路徑票（FE 票 VP-15311 已 Resolved，需確認 FE 是消費哪些 RPC）。orders attribution 屬另一條下單路徑，emr-v2 是 HL7 inbound 不適用，貿然在此加 orders schema 會改錯 repo。\n\n具體改什麼：本票交付物視為已完成（coreSamples Go side）。建議列為「選項 C 最小可獨立片」待 Leo 拍板再做、且不動任何 order schema：在 LIS-transformer-v2/src/grpc.option.ts 視需要明確化 delegation client 取用（main.proto bundle 已含 DelegationService，可直接 getService），並實作 src/role/role.service.ts:269-271/334-336 兩個 TODO（assign DELEGATE → CreateDelegationLink；remove → DeleteSingleDelegation），metadata 沿用既有 createOAuth2Metadata，audit 沿用 recordAudit，cloud fallback 走 with-cloud-fallback.util.ts。注意當前 working branch 是 feature/leo/VP-16513，本票若要動 code 需另開 feature/leo/VP-15279 並先 prisma generate + start:dev 驗證。

### 候選方案與正反取捨
- **方案A：維持現狀，本票 = 純 delegation RPC CRUD（符合 Leo comment），交付物視為已完成，僅與 PM 對齊 AC 後關票**
  - 取捨：最小交付、可獨立 ship、時間線符合 epic；但 description AC1/2/5/6 與此不符，QA 若照原 AC 驗會 fail，必須先改 AC 才驗得下去
- **方案B：在 transformer-v2 補完整 attribution（orders 加 entered_by_user_id/acting_provider_id 欄位 + 改下單 API + validate）**
  - 取捨：真正滿足 description AC，但 portal 下單寫入點未定位（emr-v2 是 HL7 inbound 不適用），可能改錯 repo，scope 從一張 QA Review 票膨脹成跨 repo schema 變更，違反 minimal-change
- **方案C（建議備選）：只接 transformer-v2 RPC client 並實作 role.service.ts 兩個 DELEGATE TODO，不動任何 order schema**
  - 取捨：填掉真正的 gap（角色指派/移除時 delegation 不同步），讓 role 流程閉環；但這嚴格說屬 role API 票（VP-15274 系列）範圍，需 Leo 確認算進本票還是另開

### 風險與邊界案例
- AC 與實作不符是本票最大風險：QA 票 QH-3183 仍 To Do，若照 description AC1-7 驗會全 fail（orders 欄位/display 根本不存在）。必須先改 AC 對齊 RPC CRUD 範圍。
- VP-15311 [FE] Provider-of-Record Selector 已 Resolved（2026-05-22）— 表示 FE 已對某個後端契約開發完成。需確認 FE 實際消費哪些 RPC（GetStaffsAllowedProviders 填 dropdown？ValidateDelegationLink 送單驗證？），避免後端契約與 FE 已上線的假設不一致。
- role.service.ts 兩個 TODO 未實作 = 指派 DELEGATE 角色時 delegation table 不會被寫入，ValidateDelegationLink 永遠回 is_valid=false → 即使做了 orders 端 AC4 也會全部擋掉下單。這是真正的功能斷點。
- AC7 graceful API error：依 ticket-routing PM 慣例（VP-16416），delegation RPC 失敗應 warn log + 不中斷，不要直接 throw 阻斷角色指派流程。
- 驗證需 live-not-mock：delegation CRUD 要對真實 coreSamples Go service（dist）跑 Create→Get→Validate→Delete 全鏈，確認 is_canceled 軟刪後 ValidateDelegationLink 回 0；不要只跑 mock unit test 就宣稱線上 OK。
- provider_user_id vs provider_id：proto 用 user_id（int32），注意 Provider ID != Practice ID，customer 權威來源是 gRPC GetCustomer，接線時別把 clinic_id/provider_user_id 混用。
