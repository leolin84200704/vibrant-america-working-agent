# VP-16759 - Fix getTimeline API on staging - 403 error

## Ticket Analysis

**Goal（目的）**：FE Patient Profile redesign (QH-4511) runs on a Cloudflare preview origin (*.va-portal.pages.dev) and calls GET /dashboard/user/timeline on staging, which returns 403, blocking the timeline section and failing QA. Goal: make staging getTimeline return 200 for this FE flow. NOTE: ticket is Dev Blocked and the only hard blocker is missing repro data (URL/payload/token) — Leo already requested it on 5/29 and QA has not responded.

**Classification**：`calendar-clinical` | **Effort**：S | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2, LIS-transformer

### 建議方案（改前 / 改後為何有效 / 改什麼）
Diagnose-first (Approach A); do NOT change app code until the 403 layer is identified. Verified facts: (1) GET /dashboard/user/timeline exists in BOTH repos with identical path — v2 LIS-transformer-v2/src/dashboard/dashboard.controller.ts:47-54 (@Controller('dashboard') + @UseGuards(UnifiedAuthGuard)) and v1 LIS-transformer/src/dashboard/dashboard.controller.ts:105 (@UseGuards(CustomJwtAuthGuard)). (2) Both guards throw ONLY UnauthorizedException/401 — unified-auth.guard.ts every throw is UnauthorizedException; custom-jwt-auth.guard.ts:71-80 same. No RolesGuard/APP_GUARD on the dashboard route in either repo, so no role-based 403. (3) The dashboard service path (dashboard.service.ts ~357 createOAuth2Metadata then gRPC userTimelineFetch) has NO gRPC-code->HTTP mapping, so a downstream PERMISSION_DENIED would surface as 500, not 403 (contrast patienttags.service.ts:84 which DOES map gRPC code 7 -> 403). Conclusion: a NestJS-origin 403 on this exact endpoint is effectively impossible with current code; the 403 must come from staging infrastructure (ingress / Cloudflare / WAF / API gateway) rejecting the preview origin or token, OR from the upstream identity/OAuth2 token endpoint. CONCRETE STEPS once repro token+URL obtained: (a) curl -i the staging URL with FE token and grep response for NestJS JSON error body vs ingress/Cloudflare HTML, and grep `access-control-allow-origin` explicitly (per LTM feedback_cors_debug_204_not_pass — never assume CORS without seeing the actual header). (b) Determine via ingress route whether /dashboard/user/timeline maps to v1 (port 3190) or v2 (3390) on staging — paths collide, so confirm WHICH service before touching any repo. (c) If 403 has no Nest error body -> infrastructure layer; fold into CORS/ingress allowlist work (QH-5840/QH-5409) and route to DevOps/PM for *.va-portal.pages.dev allowlist decision; do NOT widen v2 main.ts CORS (it is already bare enableCors() reflecting any origin, so changing it is pointless if the block is at ingress). (d) If somehow a Nest 403 -> add targeted handling, but evidence says this won't happen. Branch when a real fix is identified: bugfix/leo/VP-16759.

### 候選方案與正反取捨
- **Approach B: assume preview-origin CORS block and add *.va-portal.pages.dev to staging allowlist directly**
  - 取捨：Risky/likely wrong: CORS preflight failures surface as browser-blocked status 0 or missing Allow-Origin, not a server 403 (LTM feedback_cors_debug_204_not_pass). v2/v1 main.ts already use bare enableCors() reflecting any origin, so a code-level CORS change has no effect if the block is at ingress/Cloudflare. Also widening an origin allowlist is a security/PM decision, not an agent call.
- **Add gRPC-code->HTTP error mapping to dashboard.service (mirror patienttags.service.ts:84) so downstream PERMISSION_DENIED maps cleanly**
  - 取捨：Premature: no evidence the 403 originates downstream, and current code maps that case to 500 not 403, so it does not explain the symptom. Only worth doing if step (a) curl shows a Nest-origin error tied to the gRPC call.

### 風險與邊界案例
- Path collision: /dashboard/user/timeline exists identically in v1 and v2; fixing the wrong repo wastes effort — must confirm ingress routing first.
- CORS misdiagnosis risk (prior incident, LTM): a 403 with HTML body is infra/Cloudflare, not NestJS; always grep the actual access-control-allow-origin header before claiming CORS.
- Token scope/audience: FE preview build may mint a token with wrong audience/issuer that staging ingress or the OAuth2 introspection rejects with 403 — verify token claims, not just presence.
- If staging routes to v2 but the downstream gRPC userTimelineFetch / OAuth2 metadata is not provisioned on staging, error would be 500 (not 403) — would indicate a different bug than reported, possibly intersecting VP-16685 migration work.
- Do not push to staging directly; any repro/test must be via PR. Verify the fix live (real staging request returns 200), not via mock (LTM verified-means-live-not-mock).
