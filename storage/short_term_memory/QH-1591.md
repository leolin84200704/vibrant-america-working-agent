### 建議方案（改前 / 改後為何有效 / 改什麼）
採方案 A（index-only 最小切片），但修正 engineer 的 index 清單——他直接照 AI roadmap 列 4 條複合 index，未對齊真實 query shape，部分 index 無效。逐條核對 src/calendar 真實查詢後修正：

改前現狀（已對 prisma/schema.prisma 驗證，全部屬實）：v2_event 只有 creator_calendar_id/event_type/practice_event_type_id 單欄 index；v2_calendar 只有 @@unique([practice_id, calendar_owner_id])；v2_schedule 只有 calendar_id；v2_meeting_request 有 status 單欄；v2_event_participant 完全無 index（連 participant_calendar_id 都沒有，只有 exception 表有）。

改後為何有效 + 具體改什麼（修正版）：
1. 【先量再加，鐵律】連 calendar_prod（lis-postgresql.postgres.database.azure.com:5432, ehradmin, SET search_path=calendar_prod, /opt/homebrew/opt/libpq/bin/psql, .env DATABASE_URL_CALENDAR）對真實熱查詢跑 EXPLAIN ANALYZE + 量各表列數，確認真有 seq scan 再加，避免盲加。
2. 修正 event range index：engineer 提的 (creator_calendar_id, start_time, end_time) 對 event.service.ts:339 的 query 只能用到 creator_calendar_id 前綴——該 query 的時間條件是 OR 跨兩個不同欄位 `OR:[{start_time:{lte:endDate}},{end_time:{gte:startDate}}]`，composite 的 start_time+end_time 部分用不到。建議改為 (creator_calendar_id, start_time) + (creator_calendar_id, is_canceled)，或經 EXPLAIN 確認後僅補單欄 start_time/end_time index（schema 已有 start_time/end_time 欄位 in v2_event）。
3. v2_event_participant：補 @@index([participant_calendar_id])（含其 status 視 EXPLAIN）——event.service.ts 大量用 `v2_event_participant: { some: { participant_calendar_id } }` nested filter，這是最明確真缺、效益最高的一條，engineer 把它列「待確認」其實確認為缺且最該補。
4. v2_meeting_request：補 @@index([provider_calendar_id, status])——list query (meeting-request.service.ts:1514) 以 provider_calendar_id(+status) 過濾、created_at desc 排序；filter 受益，sort 不受益，仍合理。
5. v2_calendar (calendar_owner_id, practice_id, is_active)：edge——真實 query 是 calendar_owner_id+role+practice_id+is_active，且已有 @@unique([practice_id, calendar_owner_id]) 覆蓋高選擇度對；建議 EXPLAIN 確認既有 unique 不夠用再加，否則跳過。
6. v2_schedule (calendar_id, weekday)：低風險可補，但 calendar_id 通常已高選擇度，效益小，視 EXPLAIN。
7. prod 一律 CREATE INDEX CONCURRENTLY（不可在 transaction，故不走 prisma migrate；schema 改 @@index 僅同步 client/dev），逐一 enumerate calendar_dev_new + calendar_prod 兩個 schema。
8. branch feature/leo/QH-1591；切 branch 後 npx prisma generate（兩個 schema）+ start:dev 必過（鐵律 VP-16521）；PR base stage_test。

注意：engineer 目前在 feature/leo/VP-16513 branch（meeting-type ticket）讀 schema，其讀到的 schema 可能含尚未進 stage_test 的 meeting-type 改動——須切回 stage_test 重新確認 v2_meeting_request/v2_event 欄位後再下 index。

### 候選方案與正反取捨
- **方案 B：照 roadmap 做 Phase 2-4（BaseService/Repository/CoreModule 重構、降 forwardRef）**
  - 取捨：高風險且與 ticket 不相稱。calendar 是 Clinical Consult(150105) 線上關鍵路徑(event email/Kafka/Zoom/accession-claim VP-16410/reschedule VP-16520-21)；大規模搬 module 結構幾乎必撞 start:dev/prisma drift；且直接動 VP-16685 進行中的 src/calendar/shared gRPC 層必衝突。違反 Confirm Before Over-engineering，應拆獨立 Story 並等 VP-16685 merge。
- **完全不做 / close as wontfix（來源 roadmap 已過時、現狀多已由 VP-16685 完成）**
  - 取捨：會放掉真實存在的缺 index 效能問題（尤其 v2_event_participant 完全無 index）。不建議——index 那一小片有實質價值。

### 風險與邊界案例
- prod CREATE INDEX 即使 CONCURRENTLY 也會吃 IO/CPU，大表須挑 maintenance window；CONCURRENTLY 失敗會留 INVALID index 需手動 DROP
- engineer 在錯的 branch(feature/leo/VP-16513)讀 schema，可能含未進 stage_test 的 meeting-type 改動——下 index 前須切回 stage_test 重新確認欄位存在
- 盲加 index 風險：engineer 的 (creator_calendar_id,start_time,end_time) 對實際 OR-跨欄位 query 用不到後兩欄；必須 EXPLAIN ANALYZE 驗證才加，否則白佔空間+拖慢寫入
- 與 Meeting Type epic(VP-16512/13/14/16)同改 v2_meeting_request——若他們也加 index 須對齊欄位避免重複/衝突 index
- 三個 schema(calendar_dev 空殼 / calendar_dev_new / calendar_prod)——manual migration 須先 SELECT schema_name enumerate，勿只 apply default
- 驗證須 live-not-mock：用真實 calendar_prod 跑 before/after EXPLAIN ANALYZE 比對 plan(seq scan→index scan)與 latency，不可只憑 roadmap 數字宣稱效益
