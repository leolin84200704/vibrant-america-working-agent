## Failures

### [2026-05-27] v1 schema 過度簡化（照 PRD 沒做完整盤點）
- v1 只給 practice 單一 sftp_path，漏掉 order/result pipeline 真正會用的十幾個欄位（傳輸方式/分開的 order/result path/enabled flags/legacy fields）。沒考慮 CharmEMR HTTP 模式。
- Root cause：照 PRD 的精簡 schema 直接做，沒先盤點「pipeline 實際讀 ehr_integrations 哪些欄位」。
- 修法：Leo 質疑後派 explore 做完整 pipeline 欄位盤點 + 真實資料 COUNT(DISTINCT) per-group 一致性分析，才知道哪些該 practice / 哪些 per-provider。
- **教訓**：要「取代既有表」的 schema，先盤點既有表在所有 pipeline 被讀的完整欄位清單，再設計。不要信 PRD 的精簡 schema。

### [2026-05-27] COUNT(DISTINCT) 把 null 當一致的陷阱
一致性分析說 legacy_result_send_type「always consistent」，但 pipeline parity 抓到 1 個 group 有 [null, SFTP]。COUNT(DISTINCT) 忽略 null，所以判為一致。實際 backfill 把 null 正規化成 group 值。本案 pipeline 等價（null→SFTP）無害，但要記得 null 在一致性分析會被低估。

## Lessons Learned

### 技術
1. **要取代既有表的 schema，先盤點該表在所有 pipeline 被讀的完整欄位** —— 不要照 PRD 精簡版直接做。VP-16164 v1 漏掉十幾個 order/result 欄位 + CharmEMR HTTP。
2. **欄位該放哪一層（parent vs child）用真實資料 per-group COUNT(DISTINCT) 驗證** —— 別憑直覺。Leo 直覺 folder per-clinic，資料證明 per-provider。
3. **COUNT(DISTINCT) 忽略 NULL** —— 一致性分析會把 [null, X] 判為一致。要 pipeline parity 對拍才抓得到。
4. **CharmEMR HTTP endpoint/auth 在 env config（charm-ehr.config）不在 DB** —— 系統靠 ehr_vendor.code 判斷 vendor，再決定 SFTP/HTTP/雙傳。新表只要提供 vendor code + transmission method。
5. **lis-backend-emr-v2 SFTP 連線幾乎都從 ehr_vendors 取**（prod 僅 1 個 use_vendor_sftp_config=false override）。

### Process
6. **raw SQL（mysql2 + DATABASE_URL env）做 DDL/backfill 完全不依賴 git branch / prisma client** —— user 頻繁切 branch 或有 uncommitted 工作時，這是不污染對方 working tree 的有效手法。
7. **欄位歸屬類設計決策，Leo 要逐欄討論** —— 呈報時逐欄列出 + 給 default，讓他點。他會給很精準的判斷（含推翻自己直覺）。
8. **drop+recreate 新表（無 read path 時）是安全的 schema 迭代手法** —— v1→v2 直接 drop 重建，因為沒人 read。趁早迭代成本最低。
