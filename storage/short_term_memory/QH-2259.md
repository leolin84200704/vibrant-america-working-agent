### 建議方案（改前 / 改後為何有效 / 改什麼）
方案 A（純 read-only reconciliation，推薦）— 不碰 prod 寫入。

改前現狀：VP-14750 開發已 Done，history schedule 理應已寫入 prod calendar schema（calendar_prod），但尚無 QA 證據確認資料與舊系統逐欄位對齊。

改後為何有效：用「source 全集反查 target」的 reverse audit（符合 [JOIN scope reverse audit] 鐵律）+ 100% row 比對（[Batch DB verify]）+ 經 API 端到端抽驗（[verified means live not mock]），能抓出 silent miss（NULL=NULL=false、weekday 對映錯、TZ 偏移）而完全不引入 prod 變更風險；若驗出缺漏則回開新 bug 票補跑，不在 QA 票內動 prod。

具體改什麼（驗證步驟，非 code）：
1. 連線確認：先 `SELECT schema_name FROM information_schema.schemata WHERE schema_name LIKE 'calendar%'` enumerate（calendar_dev / calendar_dev_new / calendar_prod）；prod 用 /opt/homebrew/opt/libpq/bin/psql 連 lis-postgresql.postgres.database.azure.com:5432（DATABASE_URL_CALENDAR，URL-encoded pwd），`SET search_path = calendar_prod`。
2. 取 VP-14656/VP-14750 受影響 clinician 名單 + 舊 Lab Consult source DB 連線（migration service 用 mysql2/promise 直連，連線資訊在 src/calendar/migration/ 的 service 或 .env）。
3. 逐 clinician 比對：舊每週工時 → prisma/schema.prisma:393 v2_schedule(calendar_id, weekday, start_time, end_time) 逐欄位含 TZ 換算；舊例外 → prisma/schema.prisma:404 v2_schedule_exception(calendar_id, date, start_time, end_time, is_available)。
4. 反向 audit：以 source 全集 SELECT 找 target 漏網 row（100%，非 spot check）；特別注意 v2_schedule_exception 的 @@unique([calendar_id, date, start_time, end_time]) 含 nullable start_time/end_time → NULL 比對陷阱。
5. 端到端：抽樣經 src/calendar/models/schedule/schedule.service.ts:42 getCustomerSchedule（以 calendar_id 撈 weekly+exception）驗證，而非只查 DB。
6. 結果寫回票：全通過 → 移 Done；發現缺漏 → 回開 bug 票（feature/bugfix branch，禁動 prod）。

### 候選方案與正反取捨
- **方案 B：在本票內重跑/補跑 migration script 寫入 prod calendar**
  - 取捨：若驗出大量缺漏可一次補齊，但對 prod calendar 寫入風險高；VP-14750 已 Done 代表 PM 認定完成，未確認即重跑等於越權改 prod，違反 Confirm Before Over-engineering 與 prod 寫入須 bound scope 鐵律。不應在 QA 票內做，應回開獨立 bug 票。
- **方案 C：只做指定子集 clinician 抽樣比對（非全集）**
  - 取捨：省時，但 spot check 違反 Batch DB verify 鐵律，會 silently miss 漏網 row；僅在 PM 明確限定範圍且風險可接受時採用。

### 風險與邊界案例
- v2_schedule_exception unique key 含 nullable start_time/end_time → SQL NULL=NULL=false，比對時會 silently miss，須以 IS NOT DISTINCT FROM 或 COALESCE 處理
- TZ 偏移：舊系統存的 local time vs 新 schema start_time/end_time 若 TZ 基準不同，會整批時間平移而看似有資料實則錯位
- weekday 對映：0=Sunday vs 1=Monday 等基準差異，整週排班錯位但 row count 相符
- VP-16685 系列正把 calendar on-prem 依賴換成 cloud interface，若 getCustomerSchedule 端到端走 gRPC 路徑須留意 fallback 行為，避免驗到錯誤來源
- 舊 Lab Consult source DB 連線/schema 對照表若拿不到，整個驗證會卡住（最可能的阻塞）
- 只查 DB 不經 API 可能漏掉 service 層的 filter/transform 差異，須端到端抽驗
