### 建議方案（改前 / 改後為何有效 / 改什麼）
先當 QA 驗證任務，目前不需改任何 code（VP-16878 已 Done，重做違反 Understand Before Act）。

改前現狀（已 ground 在 repo，CORS 兩層）：
1) NestJS app — src/main.ts:70-82：讀 process.env.ALLOWED_ORIGINS，已支援 "*" → origin:true reflect any origin（搭 credentials:true，避免 literal "*" 被瀏覽器拒）；allowedHeaders 已含 baggage,sentry-trace。
2) K8s NGINX Ingress — k8s/base/ingress.yaml:37-41：enable-cors:"true" 但 cors-allow-origin 只寫死 2 個（portal.vibrantamerica.com、admin.vibrantamerica.com）。
3) App 層 ConfigMap — k8s/base/configmap.yaml:23：ALLOWED_ORIGINS 有 4 個（多了 portal.vibrant-wellness.com、www.vibrant-america.com），比 ingress 多 → 兩層 allowlist mismatch（VP-16009 的歷史 root cause）。

改後為何有效：QA 用真實前端 origin 打曾失敗的 endpoint 確認 CORS pass。驗證方式（memory iron rule：NestJS 對非 allowlist origin 仍回 204+其他 CORS header、靜默缺 Allow-Origin，不能只看 status code）：curl -i -H "Origin: <VI-domain>" -H "Access-Control-Request-Method: GET" -X OPTIONS '<prod-url>/api/v1/emr/.../integration-management/auto-integrate/requests?customerId=...' 然後 grep response 的 access-control-allow-origin header 本身。Pass → 收掉 QH-5840；Fail → 帶 curl 實證回 Leo 評估補 ingress（方案 B）。

具體改什麼（僅 fail 時才動）：k8s/base/ingress.yaml:38 的 cors-allow-origin 補齊到與 configmap.yaml:23 一致（加 portal.vibrant-wellness.com、www.vibrant-america.com）。動前必 kubectl diff 確認 repo≠cluster drift（patterns.md VP-16463），且先確認 prod 流量是否真走此 ingress 而非外部 lisapi gateway。注意：此 ticket 不需新增 process.env.X，故不觸發 lis-emr-v2-config*.yaml 雙檔更新規則。

### 候選方案與正反取捨

  - 取捨：對齊 ticket 真實性質與 tests link、零 prod 風險、成本最低；但若 prod 仍重現則需轉方案 B
- **方案 B：補 ingress cors-allow-origin 對齊 configmap（消除兩層 mismatch）**
  - 取捨：根治 VP-16009 兩層不一致；但 VP-16878 已 Done 下盲改是 over-engineering，且需先確認 prod 流量是否走此 ingress、必先 kubectl diff 防 cluster drift

### 風險與邊界案例
- NestJS 對非 allowlist origin 仍回 204 + 部分 CORS header，靜默缺 access-control-allow-origin → 驗證必 grep header 本身，不可只看 status code（memory iron rule）
- Ingress 只放 2 origin、configmap 放 4 origin：若 prod 走此 ingress，VI / wellness / america 前端可能在 ingress 層被擋即使 app 層放行；但若走外部 www.vibrant-america.com/lisapi gateway 則此層 bypass
- 若補 ingress：repo yaml 未必是 cluster source of truth（VP-16463 drift），盲 apply 可能踩錯；必先 kubectl get ... -o yaml 比對
- ALLOWED_ORIGINS=* 搭 credentials:true 時 NestJS reflect origin 而非回 literal *，要確認 prod ConfigMap 實際值（目前 configmap 仍是 4 origin 明列、非 *）—需 QA 確認 prod runtime 生效的是哪個值
- VP-16009 STM 顯示其 Decisions/Code Changes/Test Results 從未填寫（Dev Blocked 等 Kristine 提供 VI URL）→ 失敗 origin domain 仍是長期未解的關鍵未知數
