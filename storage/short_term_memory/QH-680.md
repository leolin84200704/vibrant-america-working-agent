### 風險與邊界案例
- 完整 consent flow 需 staging 已配好 Google/Microsoft OAuth app (client_id/secret) 與 redirect URI 白名單；若未配只能做 API-level smoke，無法點完整 Google/Microsoft 同意畫面
- Outlook 走 PKCE (commit 0c9d84d 移除 clientSecret)，驗證時不要誤判缺 clientSecret 為 bug
- token 加密存 v2_third_party_integration，驗 disconnect 後要確認 row 確實清除/失效 (live 查 DB，非 mock)
- Zoom OAuth connect 時只存 token、不主動 sync events (見 STM VP-16697)，驗證預期行為時勿期待自動拉 events
- public_booking 的 Google auth 只取 userinfo email/name，不做 calendar sync — 與 calendar-sync 的 Google OAuth 是不同 flow，勿混為一談
- 勿與 VP-16063/VP-16154 的 JWT→OAuth2 RS256 (RPC metadata 層認證) 混淆，那是不同概念
