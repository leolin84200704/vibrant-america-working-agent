## Retrospective
### [2026-04-20 21:30]

**1. 結果比較**
- 初始假設: crm.contacts 可以提供所有 customer 資料 → 錯誤，只有 53% (5,027/9,507)
- 初始假設: .env 的 gRPC URL 可以直接用 → 錯誤，CORE_SAMPLE_V2_RPC 是 Azure 內網
- 最終方案: gRPC GetCustomer @ 192.168.60.6:30276 成功率 99.8% (4,474/4,483)

**2. 失敗分析**
- Failure 1 (crm.contacts): 可預防 — 如果先查 knowledge/emr-integration.md 就知道 gRPC 是權威資料源
- Failure 2 (命名格式): 可預防 — 應先查 DB 現有資料的命名慣例再決定格式
- Failure 3 (gRPC endpoint): 完全可預防 — knowledge 裡有明確記載
- Failure 4 (NestJS CLI + gRPC): 新發現 — createApplicationContext 不支援 gRPC client，需改用 @grpc/grpc-js
- Failure 5 (不查 knowledge): 根本原因 — 所有其他 failure 的源頭

**3. 模式識別**
- **新 pattern**: Standalone gRPC script pattern — 用 @grpc/grpc-js + proto-loader + OAuth2 metadata
- **新 pattern**: NestJS createApplicationContext 的限制 — gRPC @Client 不初始化
- **已知 pattern**: gRPC endpoint 192.168.60.6:30276（已在 emr-integration.md，但執行時沒查）
- **已知 pattern**: Customer 資料的權威來源是 gRPC，不是 MySQL 表

**4. 教訓**
- 技術面: @grpc/grpc-js + proto-loader 可以從 standalone script 呼叫 gRPC，不需要 NestJS context
- 技術面: mysql2 連線需要 `timezone: '+00:00'` 才能正確讀 UTC datetime
- 流程面: 永遠先查 lis-code-agent knowledge，特別是 emr-integration.md 有 gRPC endpoint
- 流程面: 更新資料前先查 DB 現有命名慣例，不要自己猜

**5. 信心度: 2/5**
幾乎所有關鍵決策都是錯的，被 user 多次糾正。gRPC endpoint 是最基本的知識但沒有去查。

## Lessons Learned
### [2026-04-20 21:30]

1. **gRPC GetCustomer 是 customer 資料的唯一權威來源** — crm.contacts 只有部分資料，不可靠
2. **gRPC endpoint 已記載在 knowledge/emr-integration.md** — 永遠先查 knowledge，不要從 .env 猜
3. **Standalone gRPC script**: 用 `@grpc/grpc-js` + `@grpc/proto-loader` 載入 proto，直接建 client 連 `192.168.60.6:30276`
4. **OAuth2 metadata 必要**: authorization (Bearer token), x-request-id, internal-user-id, service-name
5. **NestJS createApplicationContext 不支援 gRPC @Client** — migration script 如果需要 gRPC，用方式 1（standalone）
6. **mysql2 timezone gotcha**: 連線加 `timezone: '+00:00'` 確保 UTC 正確解讀
7. **更新 DB 前先查現有命名慣例** — 不要假設格式，先 SELECT DISTINCT 確認
