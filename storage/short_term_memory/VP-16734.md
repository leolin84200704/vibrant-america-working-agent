## Lessons Learned
### [2026-05-26]

1. **「New EMR Integration」ticket 第一個 probe = 找 PENDING stub**：4 例連中 admin-portal 預建 stub，from-scratch new 反而是少數情境。Step 2 直接跑 `SELECT * FROM ehr_integrations WHERE customer_id=X`，看到 PENDING + business_* 全空 + requested_by 數字 → 就是 finalize 流程

2. **手動 stub finalize 的 sftp_folder_mapping 套 template 不套 VP-16424 慣例**：VP-16629（2026-05-19）加 `ehr_vendor_sftp_templates` + Cerbo approve hook 後，新標準是 `sftp_source_id=template.sftp_source_id`（Cerbo=3）+ `use_v2_pipeline=1`。VP-16424 的 NULL/0 是 hook 還沒導入前的舊寫法，不再沿用

3. **stub finalize without follow target 的 kit_delivery_option/kits_options**：Leo 偏好 `NO_DELIVERY/2`（非 LTM line 467 的 NON_BLOOD_ONLY/0）。兩者都對齊 ParseHL7.java 真實語意，但 VP-16424 + VP-16734 連續兩例 Leo 選 NO_DELIVERY/2，建議 LTM 寫成「無 follow target 時 → NO_DELIVERY/2 為預設」

4. **跨表 FK column 命名不要憑直覺**：`ehr_integration_status_history.integration_id`（不是 `ehr_integration_id`），先 `information_schema.COLUMNS` 確認再寫 query

5. **DB 寫入前一律四層防護**：read-only probe → SFTP/外部 source-of-truth 親查 → dry-run pre-check guard → commit run 包含 in-tx verify + fresh-connection 再驗

6. **report_option 通則 = PERSONALIZED**（Leo Step 6 指正）：= LTM Field Default / `getReportOption` fallback。same-clinic 有 LIVE 既有 → follow 既有值。**CLASSIC 不是預設**。
   - **我犯的歸納錯誤**：從單一 case（VP-16424 Leo 一次性指示採 stub default CLASSIC）反推出「stub 原本 CLASSIC 就保留」的假規則，還硬套 VP-16734 + 寫進 LTM。實際 grep VP-16423/16245/16424 後發現：CLASSIC 都是 (a) follow same-clinic 既有 或 (b) Leo 個案指示，從來不是預設
   - **教訓**：歸納 LTM 規則前先 grep 多個 case 看 CLASSIC/PERSONALIZED 各自的**真正原因**，不要從一個 case 反推普遍規則。Leo「看一下之前怎麼做的」就是要我做這個 cross-case check
   - 本 case DB 維持 CLASSIC（Leo 說原本就是不用特地改），未動
