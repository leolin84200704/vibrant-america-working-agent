## Ticket Analysis
### [2026-04-21 03:45]
**Ticket**: [BE] Update the supported EMR vendor list in Settings

**Priority**: P2 - Medium
**Story Points**: 6
**Related QA**: QH-4352

**需求摘要**:
Settings > EMR Integration 頁面的 EMR vendor 下拉選單從 API 取得，但目前回傳資料過時，包含了私有/測試/非公開的 vendor。需更新 API 只回傳公開可用的 vendor 清單。

**API Endpoint**: `GET /api/v1/integration-management/vendor-management/ehr-vendors`
**Repo**: lis-backend-emr-v2
**Service**: `src/modules/integration-management/vendor-management/services/ehr-vendor.service.ts` → `findAll()`
**Controller**: `src/modules/integration-management/vendor-management/controllers/ehr-vendor.controller.ts`
**Schema**: `prisma/schema.prisma` (EhrVendor model, lines 674-714)

**現狀分析**:
- `findAll()` 預設只過濾 `is_active = true`
- 沒有 public/private 區分機制
- 私有 vendor（VEJO, ZymeBalanz 等）和測試 vendor（MDHQTEST）若 is_active=true 則會被回傳
- Frontend 在清單最後附加 "Not on the list" 選項

**應排除的 Vendor（根據 Confluence 文件）**:
- Private: Unprescribed, VEJO, VEJOProgram, VEJOEcomm, THM, YHL, Nichols, ZymeBalanz
- Test: MDHQTEST
- Unknown: HealthMatters, OptimalDX
- Duplicate: PRAXISEMR（Praxis EMR 的重複）

**AC**:
1. API 回傳更新後的公開 EMR vendor 清單
2. 排除 private/test/unknown vendor
3. 向下相容，不破壞前端

## Approaches Considered
### [2026-04-21 03:45]

**方案 A（推薦）: 新增 `is_public` boolean 欄位**
- 在 ehr_vendors 表新增 `is_public` 欄位（default false）
- Prisma migration 更新 schema
- 批量更新現有 vendor 的 is_public 值
- findAll() 加入 is_public 過濾參數，Settings 頁面呼叫時傳 isPublic=true
- 優點：動態可維護、未來加 vendor 只需設 is_public=true
- 缺點：需要 DB migration

**方案 B: 新增 `visibility` enum 欄位 (PUBLIC/PRIVATE/TEST)**
- 更細粒度的分類
- 優點：區分 private vs test
- 缺點：較複雜，目前需求可能不需要

**方案 C: 在 service 層硬編碼排除清單**
- 快速但不符合 "dynamically maintainable" 的要求

### [2026-04-21 03:50] Debate Summary

**正方（支持方案 A: is_public boolean）**:
- 二元決策用 boolean 最簡單，YAGNI
- 向後相容：optional query param，不帶參數時行為不變
- 未來若需 enum，從 boolean 遷移成本極低
- 資料問題用資料解決，不該硬編碼

**反方（質疑方案 A）**:
1. 需確認 API 是否有其他消費者（admin panel、報表等）
2. `default(false)` 可能導致新 vendor 上線後忘記設 flag，dropdown 缺少選項且不報錯
3. Notion 與 DB 手動同步，長期會 drift
4. Boolean 缺乏擴展性（未來可能需要 internal/beta/deprecated）
5. 重要邊界案例：若 practice 已選了 private vendor，dropdown 過濾後會顯示什麼？

**綜合結論**:
- 方案 A（is_public boolean）仍然是最佳選擇，但需處理反方提出的風險
- `default` 值需要決定：true（新 vendor 自動公開）vs false（需手動開啟）
- 邊界案例（已選 private vendor 的 practice）需與前端/PM 確認
- 其他 API 消費者需確認

## Retrospective
### [2026-04-21 05:30]

**結果比較**:
- 初始假設 ChARM vendor code 是 `CHARMEHR`，實際是 `ChARM_EHR` — 若未查 DB 驗證會導致 migration 匹配失敗
- 初始假設 explore agent 查 migration scripts 就能得到完整 vendor list，實際 DB 有 33 個 vendor，migration scripts 只涵蓋約 20 個
- 初始 migration 策略是 default TRUE + 排除 private，改為更安全的 全部 FALSE + 設定 public，避免遺漏未知 vendor

**失敗分析**:
- 無重大失敗，但若未查 DB 驗證 vendor codes，ChARM_EHR 的 migration 會靜默失敗（UPDATE 0 rows）

**模式識別**:
- ehr_vendors 的 code 欄位有 mixed case（ElationEMR, OptimalDX, ChARM_EHR），不是全大寫
- repo 內的 migration scripts 不是 vendor 資料的完整來源，必須查實際 DB
- 新增 filter 欄位的安全模式：先全部設為排除值，再明確設定包含值

**信心度**: 4/5 — 分析和實作過程順暢，唯一需要 Leo 糾正的是查 DB 確認 vendor codes

## Lessons Learned
### [2026-04-21 05:30]

1. **ehr_vendors.code 是 mixed case** — legacy data 不遵守 DTO 的 uppercase 規則，寫 SQL 時必須用實際 DB 值
2. **查 DB 驗證 > 查 repo migration scripts** — migration scripts 只是歷史紀錄，不代表 DB 現狀
3. **Data migration 安全模式** — 新增 boolean filter 欄位時，先 UPDATE ALL = FALSE，再 UPDATE known = TRUE，比反向安全
4. **Vendor API 架構** — `EhrVendorService.findAll()` 只服務 HTTP endpoint，所有內部 vendor lookup（HL7、result、ChARM）都直接用 Prisma query，不經過 service method
5. **Notion 是 vendor public/private 的 source of truth** — Confluence 文件可能過時
