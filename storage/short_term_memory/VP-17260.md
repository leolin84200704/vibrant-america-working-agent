# VP-17260 - Work Loop Record

> Created: 2026-06-29 UTC

## Ticket Analysis
P1. Jason (Furthest Available = 14 天) 日曆顯示某 slot 可約但 book 報錯,~1-3 ticket/週。根因:max_advance 顯示用日期(整天可見)、預約驗證用 `now + N天` 時間戳 → 最後一天「比現在時刻晚」的時段顯示可約但 book 失敗。

## Decisions Made
- Leo 確認:date-based、**provider 時區**、最後一天整天可約、顯示=可約。
- Branch base: 從 `origin/stage_test` 開(已含 VP-17190/17222 provider-tz 遷移),用 **git worktree** `.worktrees/VP-17260`。

## Code Changes
- 新增純函式 `maxAdvanceCutoff(now, maxAdvanceDays, ianaTz)` in `shared/timezone.util.ts` = (now+N) 那天在 provider tz 的 23:59:59.999。
- `provider-availability.service.ts` validateAndPrepare (booking 驗證,涵蓋 meeting-request/public-booking/patient-event per VP-16850) + `user-settings.service.ts` validateBookingTime 都改用它。
- 顯示(L179 date-based)+ min-notice **不動**。
- commit on `feature/leo/VP-17260`, **PR #519** → stage_test。

## Test Results
- `timezone.util.spec.ts` +3 deterministic 測試(real date-fns-tz,DST 正確):end-of-day in tz、最後一天下午放行(regression)、隔天擋。7/7 pass。
- build 0 error、start:dev boot OK、無 CJK。
- 待 staging E2E:14 天 provider 點最後一天下午實際 book。

## Lessons Learned
- 「顯示 vs 可約」用同一個邊界定義(此處 per-day, provider tz)才不會 shown-but-unbookable。時間制 vs 日期制不一致是這類 bug 的典型根因。
- 抽純函式 (maxAdvanceCutoff) 讓 deterministic 單元測試可行,避開 `new Date()` 不可測問題。
