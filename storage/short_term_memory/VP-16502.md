## Lessons Learned

### 流程 (process)
- **PRD「To X」字面理解先 raise Open Question**：deployed prod 行為（譬如 VP-16391 reminder 發所有 participant）是 design intent 的硬證據；AC 跟 deployed test 衝突時不要直接改 test 配合 AC，先跟 PRD owner 確認語意
- **Simplify 範圍要 explicit + 系統掃**：Leo 限「calendar module」時用 Explore agent 派出 read-only scan 找 punch list，每個 finding 標 priority + 估改動範圍，讓 user 自選 high-priority
- **redundant else if simplify 的判別**：`if A {} else if X===false {} else if X !== true {}` 後兩條 body 一樣 = X===false 是 X !== true 的 subset，可合併

### 技術 (technical)
- **Cross-cutting helper 抽不乾淨時 in-place duplicate 比 over-DRY 好**：5 個 send method 每個 templateModel build 不同 fields（PRD 7 scenario × 2 recipient role 不互通），抽 helper 要 switch by `(notificationType, recipientType)` 變得醜；in-place 加 ~30-50 行 cross-recipient loop / method 讀起來更直白
- **Cross-recipient YAML mapping 模式**：每個 notificationType 下加 cross-recipient `recipientType` key，指向另一邊既有 Postmark id（不開新 template）— 用 in-file duplicate 比 YAML anchor 更易讀

### Repo pattern
- **Calendar email「actor doesn't get notified」反 pattern (pre-VP-16502)**：原本 createEvent → 只發 patient、createEventByPatient → 只發 provider/clinicadmin。VP-16502 後改成兩邊都發（gated by `isClinicianEvent`），non-clinician 行為不變
