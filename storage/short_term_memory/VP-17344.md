# VP-17344 - Work Loop Record

## Ticket Analysis

### [2026-07-08 10:30]
- Jira: P2 Story, Dev To Do, reporter Xiaoye Li, assignee Leo. Request originated from Yekai Liu (results team).
- Requirement: per-integration push level `per_report_ready` / `per_sample_type_ready`; default (whole-order) unchanged; v1 enabled for ONE VIP practice only; backend-config, no UI; reuse existing push pipeline; AC includes "incomplete-sample-type case handled (no stall)".
- Current trigger: `kafka-report-finished-listener.service.ts:262` — only `report_finished` / `redraw_report_finished` (lis-report) enqueue result generation.

### Event landscape (verified against ClickHouse `kafka.general_sample_events` @ 192.168.62.85:8123 + producer code)
- **`sample_type_all_finish`** (lis-result, NEW 2026-07-08, added by Yekai for this request): fires when ALL tests of one tube/sample type are finished, regardless of TNP count (superset of `sample_type_all_finish_with_tnp`, which still fires separately when ≥1 TNP). Payload: `sample_types:[type]`, addon `{sampleType, finishedTestIds, tnpTestIds, TNP}`. Has sample_id + accession_id + patient_id; **customer_id NULL / clinic_id 0** → must resolve via `grpcClient.getSampleRelevantInfo(sampleId)` (exists, returns customerId + clinicIds).
- **`new_report_status_updated`** (lis-report): fires on every per-report status change under a barcode (ready AND viewed). addon has generated/viewed/unviewed report name lists + counts; 6.7k partial (generated<total) events /14d. **sample_id=0**, accession-keyed, no customer → needs barcode→sample/customer resolution. Consumer must diff "which reports newly ready" itself (track pushed report names per accession+integration).
- **`personalized_report_ready`**: fires only when ALL reports under barcode done — NOT per-report; wrong trigger for per-report level.
- **`report_finished`** / `redraw_report_finished`: report-level whole-order, carries customer_id+clinic_id (current trigger, keep as default).
- Sample-level result events: `results_all_finished` (all ordered tests have results) → `sample_finished`/`_with_tnp` (all approved) → `report_finished` (report generated, last).
- `products_finished` family: per-product(panel) finish, products+test_ids arrays; ~99.1% unique per (sample,product) over 7d, ~0.9% fire 2–6x → consumers need idempotency.

### Producer facts (LIS-backend-results-core test-status.service.ts)
- `checkSampleTypeFinishStatus` (:1045) groups ordered non-calculation tests by sampleType, computes hasAllFinished; now emits `sample_type_all_finish` always + `_with_tnp` when TNP≥1. Redis lock dedup per (sampleId, sampleType).
- `products_finished` emitted from `checkSampleDifferentFinishStatus` when the just-approved batch completes a product.

### emr-v2 pipeline facts
- `GenerateResultHl7Input` has NO test/report filter → a "partial push" = trigger existing pipeline earlier; HL7 contains cumulative snapshot of currently-approved results. Sending method unchanged (matches Yekai's statement).
- `add_report` config (0=none, 1=classic, 2=personalized) attaches report PDF into HL7; file name `{barcode}.hl7` → later pushes overwrite earlier partial file on vendor SFTP (result-generation.service.ts:1491 comment).
- `ensureTransmissionRecord` reuses record per (sample_id, integration) within 24h → partial pushes need a scope key or multiple records collide.
- VP-17312 pipeline_location partition (onprem/cloud consumer groups) applies to any new event consumption too.

## Approaches Considered

### [2026-07-08 10:30] Draft design (pre-debate)
1. Schema: `ehr_integrations.result_push_level` ENUM('WHOLE_ORDER','PER_REPORT','PER_SAMPLE_TYPE') DEFAULT 'WHOLE_ORDER' (Prisma + manual migration on BOTH prod & staging MySQL).
2. Listener: accept 2 new event names, route by per-integration push level; resolve customer/clinic via getSampleRelevantInfo for lis-result events.
3. result_transmission_records: add push_scope/push_scope_key (e.g. 'SAMPLE_TYPE:EDTA', 'REPORT:WZDBS') for dedup + idempotency.
4. Keep final whole-order push at report_finished even for partial-level integrations (covers "no stall" AC + authoritative complete file). — needs PM/Leo confirm.
5. Redraw stays whole-order (redraw_report_finished) in v1.

### [2026-07-08 11:20] Debate (solo — subagent spawn failed twice with API 529; analyzed both sides by direct code reading)
**PRO**: single gate at listener:262 + eligibility query = contained change; getSampleRelevantInfo(sample_id)→customer/clinic exists (grpc-client.service.ts:269); GenerateResultHl7Input.add_report override ('0') exists → partial pushes can skip PDF; keeping final whole-order push trivially satisfies no-stall AC.
**CON (ranked)**:
1. PDF failure spiral: partial push with add_report>0 before report generated → downloadPdfFromApi (result-generation.service.ts:2243) retries 4x (~70s, 10/20/40s backoff) then throws → record ERROR + BullMQ retry x5. Mitigation: force add_report='0' on partial pushes (input override at :31), DB config only for final push.
2. getLatestApprovedTime (:1740) returns '' when finished_reports empty (:1755) → partial per-sample-type push before first report → empty report-time in HL7; vendor validation risk.
3. Volume: sample_type_all_finish ~4-5k/day + new_report_status_updated ~1.3k/day would each need gRPC/API resolution BEFORE eligibility; must gate cheaply (in-memory cache of partial-level integrations, v1 = 1 practice).
4. Monitoring/tooling assume 1 record per sample+integration: runbook §8 duplicate-check (GROUP BY sample HAVING >1) would false-alarm; fhir-result.service reads records; generation service must target exact record id, not findFirst by sample+integration.
5. {barcode}.hl7 filename overwrite on vendor SFTP: partial then final overwrite same name — acceptable because snapshots are cumulative (final ⊇ partial), but vendor polling behavior unknown (out of scope per ticket).
6. new_report_status_updated: viewed events dominate ready events; diff via push_scope_key records = naturally idempotent; concurrent same-accession events need unique-constraint-based dedup, not read-then-write.
7. OBR/OBX result status semantics for partial files (preliminary vs final) unverified — open question for PM/vendor.

## Decisions Made

### [2026-07-08 11:50] Leo confirmed scope (Step 4)
- "全部都先只有 final result ready 才 push, 我會手動調架構, 只要有功能, 現在要讓 end user 感受不到差異" → build both partial levels, ship with everything at WHOLE_ORDER default; Leo flips DB values manually later; zero behavior change on deploy is the hard requirement.
- Design defaults locked: partial pushes force add_report='0'; final whole-order push retained for partial-level integrations; redraw stays whole-order; REPORT scope never re-pushes (viewed events no-op forever), SAMPLE_TYPE scope re-pushes after 24h (redraw→re-finish).

## Lessons Learned
- Ship dormant features behind a DB-config gate checked via a short-TTL cache: zero-impact deploy, Ops-flippable, fail-safe on gate query error.
- Cross-team event gaps can dissolve mid-ticket: yesterday's "no plain sample_type finish event" became today's `sample_type_all_finish` — re-verify ground truth (ClickHouse) before designing around a gap.
- `new_report_status_updated` producer sends sample_id=0; any consumer needs barcode→sample resolution (`getSampleIdByBarcode`).
