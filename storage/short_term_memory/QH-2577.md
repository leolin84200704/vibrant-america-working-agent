### 建議方案（改前 / 改後為何有效 / 改什麼）
改前現狀: 之前推到 EMR 的 report 仍含 TNP markers，portal 已移除 → provider 端是舊版檔案。注意 source ticket VP-15050 已是 Done（2026-02），TNP-marker comment 是最後一則實質留言、隨後被標 Dev Complete→Done；QH-2577 只是 QA-Hub 鏡像 stub（auto-created，無描述，3 則全是 automation 留言），仍 To Do。因此第一件事是跟 Leo/PM 確認 QH-2577 是「真的要再推一次/驗證」還是「殘留鏡像可直接關掉」——不要假設一定要動 prod。

改後為何有效: 正規 generation flow 在 generation time 重抓 report status + PDF（getReportStatusFromApi line~1796/2244 + downloadPdfFromApi pdf-cache/download/{barCode}），只要來源 report 已是 marker-removed 版本，re-push 就會覆蓋 provider 端舊檔。

具體改什麼（確認要做之後）:
1. 解析 ID: 2511046917 是 10 碼字串，極可能是 accession_id 而非 numeric sample_id。先 SELECT sample_id, accession_id FROM lis_emr.sample WHERE accession_id='2511046917'（同 repush-samples-by-date-range.ts 的 query 形態）。
2. 用既有機制重推: gRPC ResultGenerationService.GenerateResultHl7({sample_id, send_result:true})，正確參考腳本是 scripts/repush-samples-by-date-range.ts（它就是 SELECT sample.sample_id 後 gRPC GenerateResultHl7 send_result:true）。**不要**參考 resend-vp15942-result.ts——那支是對既有 27MB 本地 HL7 檔做 SFTP 壓縮重送、無 gRPC、無 DB 解析，跟本案語意不同。
3. 必打 pusher pod: result-grpc.controller.ts:85-92 有 isPusher / POD_ROLE guard，非 pusher 直接回 fail。
4. branch bugfix/leo/QH-2577，若需新增一次性腳本仿 repush-by-date 的 gRPC 呼叫；不引入新 env var（temp 操作）。
5. 推完抓 vendor SFTP 端實際檔案人工檢查 TNP markers 已移除（live not mock / peer-observed），再回報。

注意: 工程師把 line 1162 註解解讀成「wellness PDF cache 未刷新 → 推舊版」是**誤讀**。該註解講的是舊 placeholder bug（gRPC 失敗時回傳合成 JB{sampleId} barcode + Unknown Patient，該合成 barcode 不在 cache → placeholder PDF），與 TNP-marker 舊版無因果。code 中無證據顯示 cache 是 TNP root cause。stale-version 根因仍未證實，必須先查實際資料（report approve 時間 / 上次推送內容 vs 現行 report）再下結論。
