# VP-17422 — Daily Appointment Report: 3 duplicate emails + "Mary Beth not receiving" (P1)

## Evidence (Postmark main server, tag=daily_appointments_report, 2026-07-15)
- 30 messages = **10 recipients x exactly 3 copies each** (9 clinical-team members practice
  150105 + ops hung.l@zymebalanz.com). All sent 03:01:19–22 EDT (= 00:01 PT, the
  `@Cron('1 0 * * 1-5', tz America/Los_Angeles)`; 07:01 UTC).
- Simultaneous triple → `generateAndSend()` executed **3 times**. Code sends exactly 1
  email/member + 1 ops per run (no loop dup), so 3 copies = 3 runs = the **3 AKS transv2
  prod replicas** (`lis-transv2-deployment-b487664fc` x3, all platform_type=cloud,
  aks_server_type=prod) each ran the cron. The best-effort daily lock failed to dedup.
- **acquireDailyLock** (daily-report.service.ts ~484): `SET daily_appointments_report:<date>
  NX EX 25h` on getRedisClient(); on ANY redis error the catch `return true` (fail-open =
  "send anyway"). If the App-Config-driven Azure redis was unreachable at 07:01 UTC (known
  fragility, cf INCIDENT-20260518 / VP-17421 reconnect), all 3 replicas fail-open → 3 sends.
- Only the daily report triples (consult reminders / kit emails don't) → it's the
  **producer** (per-replica cron + weak lock), NOT the shared notification-center consumer.

## Mary Beth ("not receiving") — CONTRADICTED
- Mary Augustine = mbaugustine@vibrant-america.com (calendar_id 30788, practice 150105,
  clinicadmin, active) IS in loadMembers and HAS an email. Postmark: all 3 of her 7/15
  copies Status=Sent, Events=Delivered (+Opened on 2). She received and opened them.
- So either "Mary Beth" is a DIFFERENT person (a team member not in v2_calendar practice
  150105 → loadMembers wouldn't include her) or a local mailbox/filter issue. Needs Leo/
  reporter clarification before any code change.

## Fix (proposed, NOT coded — design choice + Mary ambiguity → confirm first)
- #1 duplicates: replace the fail-open best-effort NX with a reliable single-owner guard.
  The calendar module already uses **Redlock** for its other crons — use it here so exactly
  one replica runs generateAndSend. (Fail-open is the actual defect: it converts a redis
  blip into N duplicate sends.) Alternatively fail-closed with the ops-heartbeat detecting a
  missed run. Recommend Redlock (matches existing pattern, exactly-once).
- #2 Mary Beth: no repro (she's Delivered+Opened). Confirm who "Mary Beth" is before acting.
