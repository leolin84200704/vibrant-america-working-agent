### 現行程式碼 (改前)
- `src/calendar/models/reminder/reminder.service.ts:229` `buildTemplateModel`:
  `const tz = recipient.timezone || event.timezone || 'America/Los_Angeles';`
- 這是 **per-recipient** (每個 recipient 用自己 `v2_calendar.timezone`)，VP-16391 建立、VP-16664 沿用。
- **關鍵 bug**: 沒有 `!== 'UTC'` 守衛（不像 event.service `resolveEventTimezone` 有擋 UTC）。
  prod 有 ~18k provider/clinicadmin `v2_calendar.timezone='UTC'`（VP-16202 legacy backfill 未填者保留 UTC）→ 這些 recipient 的 reminder 直接 render UTC。**這正是 PM 說 "currently display times in UTC" 的原因。**
- 資料來源 `v2_calendar.timezone`；新 setting 在 `v2_user_settings.timezone`（schema 已存在，default America/Los_Angeles，reminder 目前**未用**）— VP-16943 可能就是擴充/啟用這個。

## Approaches Considered

### [2026-06-09]
待與 Leo 討論後確定。核心待決：
1. **依賴**: VP-16943 未完成，need 確認 provider timezone 存哪（v2_user_settings.timezone? 新 column?）+ 讀取路徑（local PG / gRPC setting-service）。VP-16943 是 Fan Zhou 負責，需協調。
2. **語意衝突**: ticket 字面「用 provider's TZ format 信中所有時間」vs VP-16664 deliberately per-recipient（patient 看自己 TZ）。要釐清是 (a) 只改 provider-recipient 的信、還是 (b) 全 recipient 都改用 provider TZ（會 reverse VP-16664）。
3. **UTC-as-unset**: fallback 邏輯要把 'UTC'/null/空 都視為「未設定」→ PST，加 `!== 'UTC'` 守衛。
4. **PST 字面 vs IANA**: 維持 `'America/Los_Angeles'` IANA，由 getTimezoneAbbreviation DST-correct render PST/PDT，不要 hardcode "PST" 字串（VP-16664 教訓）。

## Decisions Made

### [2026-06-09] UTC sentinel 是核心問題；解法在「unset 表示法」，非換來源
- Leo 質疑：若後端補 `!== 'UTC'`，真的在 UTC 的 provider 登入後仍被強制 PST，永遠改不了。**正確。**
- 我曾下「換來源讀新欄位」結論 → **Leo 否決，ticket 沒這樣寫、來源讀哪個欄位本來就還沒定（取決於 VP-16943）。撤回此結論。**
- 真正的根因與 Leo 質疑一致：**只要 `'UTC'` 同時代表「未設」和「真 UTC」就一定有人被犧牲。** `!== 'UTC'` 守衛是錯的（犧牲真 UTC）。
- 正解 = 讓「未設定」變成可與真 UTC 區分的值（與來源無關）：
  - legacy 那批「填不出值的 UTC」應轉成 **NULL**（NULL=未設→PST；`'UTC'`=真 UTC→顯示 UTC）
  - VP-16944 登入抓真實時區寫回同欄位；reminder 端 `value ?? PST`，**不需 `!== 'UTC'` 守衛**
- 與 VP-16943 (Fan) 要鎖死的真正只有一件：**provider 設定「未設定」用什麼表示，且絕不能是 `'UTC'`**（傾向 NULL）。來源讀哪個欄位次要。

### [2026-06-09] 收斂：真正是「改同欄位(要migration) vs 開新欄位」的架構決策
- Leo 指出：`UTC→NULL`（render fallback PST）對現存 18k 來說，顯示結果跟 `UTC→PST` 完全一樣。**正確，我前面過度強調 NULL/PST 差異。**
- 我也修正自己前後不一致：先喊「不要動 DB」，後來的 NULL 方案其實還是在動 DB。
- **不可化約核心**：legacy「沒設」與「真 UTC」目前在 `v2_calendar.timezone` 都是字串 `'UTC'`。要讓 future 真 UTC 被尊重，**必須清掉 legacy 那批含糊 UTC**（清成 NULL 或 PST 顯示等價）。
- 真正只有兩種架構：
  - **(A) 沿用 `v2_calendar.timezone`** → 必須做 migration 清 legacy UTC（動 DB 無可避免）；之後 reminder 不加守衛、future 真 UTC 自動被尊重。
  - **(B) VP-16943 開獨立新欄位** → v2_calendar 不動，新欄位天生乾淨，reminder 改讀（= 先前被否決的換來源）。
- 兩者皆同：現存 18k 登入前都只能 PST（真實時區從未被記錄，無法還原）。
- 唯一實質差別：寫 PST 進 DB 會污染其他 consumer（event.service/availability 也讀此欄）；NULL/新欄位則保留「未知」訊號、不污染。
- **決策歸屬**：這是架構選擇，且綁 VP-16943 欄位設計 → 必須先跟 Fan/PM 對齊，不該替 Fan 假設。

### [2026-06-09] **最終決定：選 (A)，但未來再做（deferred，本次不動 code/DB）**
Leo 拍板走 **(A) 沿用 `v2_calendar.timezone` + migration 清 legacy UTC**。

未來實作具體步驟（記住，等 VP-16943 進度到位再做）：
1. **Schema**：`v2_calendar.timezone` 改 nullable（`String?`）。現有 reader 都 NULL-safe：event.service `resolveEventTimezone` 用 `providerTimezone &&` 會跳過 NULL；reminder 用 `||` 也會跳過。
2. **Migration**：`UPDATE v2_calendar SET timezone=NULL WHERE timezone='UTC'`（只清 legacy backfill 那批 ~18k）。
   - **時序鐵則**：必須在 VP-16944 登入抓真實時區（會寫入真 'UTC'）**上線前 / 之後不可再 blanket 重跑** —— 否則會把真 UTC 一起清掉。
   - 100% verify row count（batch DB verify 鐵律）。
3. **reminder.service.ts:229**：移除把 `'UTC'` 當 sentinel 的需求 —— migration 後欄位裡剩的 `'UTC'` 保證是真 UTC，故 **不加 `!== 'UTC'` 守衛**，NULL → fallback `'America/Los_Angeles'`，future 真 UTC 自動被尊重。
4. 維持 IANA fallback `'America/Los_Angeles'`，由 `getTimezoneAbbreviation` DST-correct 顯示 PST/PDT，不 hardcode "PST" 字串。

**仍未決（未來開工前要再跟 Leo/PM 確認）**：

- VP-16943（Fan）的 provider 設定到底寫不寫回 `v2_calendar.timezone`（決定 (A) 是否真的同欄位）。
