### 風險與邊界案例
- createEventByPatient 的 clinic-admin 代他人建立分支無單測，最易回歸（input.customer_id != token customer_id 的 403 邏輯 L1622）
- token gating：by-patient mutation 應只接受 customer token（PR #216 make_calendar_by_patient_only_accept_customer_token）；需驗 provider/admin token 被拒
- 150105 gate 邊界：非-150105 practice 必須走一般路徑而非 by-patient（update/delete spec 已有，create 缺）
- 下游回歸面：VP-16410 accession-claim sync、VP-16520/16521 rescheduleClinicalConsult、VP-16499 multi-clinician 都建在 by-patient 之上，改動可能破壞 create/update/delete 行為
- VP-16685 cloud migration 替換 on-prem 依賴可能影響 calendar 鏈路，驗證時機需與其協調
- 驗證須 live-not-mock：staging 跑真實三條 mutation，勿把 mock 單測當『線上驗證』
