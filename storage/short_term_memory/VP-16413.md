## Lessons Learned
### [2026-05-01 17:00]

1. **Clinical Consult email flow（practice_id=150105）的對稱結構**：
   - `createEvent` (staff books) → `sendAppointmentScheduledEmailForCreateEvent` → 收件人 only role=`patient`，template `appointment_scheduled_by_provider/patient` (prod 33802988)
   - `createEventByPatient` (provider as seeker) → `sendAppointmentScheduledEmailToNonPatients` → 收件人 only role=`provider`/`clinicadmin`，template `appointment_scheduled_by_patient/provider` (prod 34153520)
   - 兩段都 publish 同一個 Kafka `Appointment Created`（`general-sample-events`），但 email 是直接 transformer-v2 publish 到 `notification-email-template`（不走 setting-consumer）

2. **transformer-v2 calendar email Kafka 佈局**：
   - Appointment events: `Azure_kafka_general_events` broker → topic `general-sample-events`
   - Email payloads: `Azure_kafka_notification_url` broker → topic `notification-email-template`（注意 broker 跟 appointment 不同 namespace）
   - 兩個 broker 都 SASL plain + ssl + `$ConnectionString` 帳號

3. **Postmark Activity 查詢步驟**：UI → Server → Activity → filter recipient + template + date range，沒紀錄 = 訊息沒到 Postmark；有紀錄看 Status (Delivered/Bounced/Suppressed) 判斷下一步

4. **Event Hub retention 容量決定論**：高量 topic（如 general-sample-events）retention 可能短於 7 天；事後驗證得在事發後盡早做

5. **Silent .catch swallow 反 pattern**：`event.service.ts:1574` 是反例，await + .catch(log) 吞 exception 等於 fire-and-forget，前端不知道 email 失敗。後續做 audit log feature 時要避免重蹈

6. **kafkajs 連 Azure Event Hub**：`new Kafka({ brokers: ['*.servicebus.windows.net:9093'], ssl: true, sasl: { mechanism: 'plain', username: '$ConnectionString', password: connectionString } })`
