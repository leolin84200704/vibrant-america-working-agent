# VP-16516 - Work Loop Record

> Status: BLOCKED — paused pending PM confirmation (2026-06-06)

## Ticket Analysis
- [BE] Meeting Type-Scoped Availability (weekly hours + date overrides). Epic VP-15817. LIS-transformer-v2 calendar.
- 目的：每個 meeting type 有自己的可約時段（非共用 provider 一套）+ 「Apply default availability」複製 provider 班表。
- 改前：availability 是 provider-level 單一套（v2_schedule + v2_schedule_exception）；slot 引擎 provider-availability.service.ts 寫死讀這兩表。VP-16512 的 meeting-type availability 兩張表是空的、無 API、slot 沒讀。
- 改後：create/update 吃巢狀 weekly/date 陣列寫入；apply-default 複製 v2_schedule；slot 引擎加 optional meeting_type_id（沒給=行為不變）讀 meeting-type 表，沿用既有 date-override-weekly 合併邏輯（AC3）。

## 探查重點（provider-availability.service.ts 等）
- provider weekly = `v2_schedule`(calendar_id, weekday 0-6, start/end `@db.Time(6)`)；setter `schedule.service.ts updateWorkingHours`。
- date override = `v2_schedule_exception`(calendar_id, date, start/end Time?, is_available)；合併在 `provider-availability.service.ts buildAvailablePeriodsForDay` (~560-597)。
- slot 計算主方法 `getProviderAvailability(input)` + `calculateAvailableSlots`（provider-availability.service.ts），被 public-booking + meeting-request 呼叫。
- **格式差異**：provider 用 `@db.Time(6)`(Date)，meeting-type 表用 `String "HH:mm"` → 整合需轉換。
- 估 ~500-700 行，且 Part 2 動到**全 provider 共用的 slot 引擎**（hot path）。

## Decisions Made
- 範圍切兩塊：Part 1（meeting-type availability 管理 CRUD + apply-default + 驗證，自包零風險）；Part 2（slot 引擎加 behavior-preserving optional meeting_type_id，碰共用 hot path）。
- **2026-06-06 Leo 決定：先暫停，跟 PM 確認整個 epic 的「clinic-level → customer/provider-level」大方向是否確定再做**（因 Part 2 動到所有客戶都跑的 booking 引擎 + 舊 clinic-level 系統會被逐步取代）。已擬英文短訊給 Leo 貼 PM。
- **未動任何 code**（VP-16516 尚未開 branch / 無 commit）。

**Goal（目的）**：讓每個 meeting type 擁有自己的可約時段（weekly hours + date-specific overrides），取代目前全 provider 共用單一套 availability；並提供「Apply default availability」一鍵把 provider 預設班表（v2_schedule）複製進 meeting type。slot 計算最終要能依 meeting type 算可約時間（AC1–AC5）。

**Classification**：`calendar-clinical` | **Effort**：L | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
分兩片交付（B 方案），等 epic sync call 拍板大方向後執行。

改前現狀（已逐一驗證 file:line）：
- Schema 三表由 VP-16512 建好：v2_meeting_type(prisma/schema.prisma:530)、v2_meeting_type_weekly_availability(:562, start_time/end_time = String @db.VarChar(5) 存 "HH:mm")、v2_meeting_type_date_specific_hour(:577, 含 is_unavailable Boolean)。
- meeting-type.service.ts:176 getOne() 已 include weekly_availability + date_specific_hours；但 create(:124)/update(:186) 只逐欄位寫 scalar 欄位，完全沒寫子表 → 兩張表是空的。
- create-meeting-type.dto.ts / update-meeting-type.dto.ts 無 availability 欄位、無 nested class-validator。
- meeting-type.controller.ts 只有 CRUD + reorder + status，無 apply-default route（routes 分兩組 base path：:providerId/meeting-types 與裸 :id）。
- Slot 引擎 provider-availability.service.ts:146 getProviderAvailability 只讀 provider-level v2_schedule(:458)/v2_schedule_exception(:461)，無 meeting_type_id 概念；合併在 :560 buildAvailablePeriodsForDay。被 3 條路徑共用（hot path）：public-booking.service、calendar-sync.resolver、meeting-request.resolver。

Part 1（零 prod 風險，自包，可先交）改什麼：
- create/update DTO 加 weekly_availability?: {day_of_week:0-6, start_time:"HH:mm", end_time:"HH:mm"}[] 與 date_specific_hours?: {date, start_time?, end_time?, is_unavailable}[]，含 @ValidateNested + class-validator（day_of_week 0–6、HH:mm regex、end>start 用自訂 validator，AC5 回 400）。
- meeting-type.service.ts create/update 在 $transaction 內 create / replace（先 deleteMany 子表再 createMany）子表（cascade 已備但 update 要顯式 replace）。
- 新增 service applyDefaultAvailability(providerId/id, user)：讀該 provider 的 v2_schedule，用既有 formatScheduleTime(t:Date) helper（provider-availability.service.ts:100，把 @db.Time(6) Date 轉 "HH:mm"）轉成 meeting-type weekly rows 寫入（AC4）。
- controller 加 apply-default route，掛在 :providerId/meeting-types/:id 線上、宣告於動態 :id 之前避免 route 衝突。

Part 2（AC3，碰共用 hot path，需 behavior-preserving）改什麼：
- getProviderAvailability 加 optional meeting_type_id；未給 = 一行不變（沿用 v2_schedule）；給了才改讀 meeting-type 兩表。
- 需寫 "HH:mm" → Date adapter：buildAvailablePeriodsForDay 吃的是 Date（透過 combineDateTime，:562 簽名要求 start_time:Date/end_time:Date），不是字串 — 所以工程師「入口轉一次 HH:mm→Date」方向正確，且該 Date 的 UTC time-of-day 必須對齊 Prisma @db.Time(6) 的 mapping（getUTCHours/Minutes 取值，與 formatScheduleTime 互為反向）。
- 關鍵修正：兩表 polarity 相反 — provider 用 is_available，meeting-type 用 is_unavailable。adapter 餵進 builder 前必須翻轉 is_available = !is_unavailable，否則 unavailable 整天/挖洞邏輯會全反。
- 必須擴充既有 provider-availability.service.spec.ts（已含 VP-16850 end-to-end 案例），新增「給/不給 meeting_type_id」兩分支 + override 合併案例，跑真實 builder（非 mock getProviderAvailability）。

### 風險與邊界案例
- polarity 相反陷阱：v2_schedule_exception.is_available vs v2_meeting_type_date_specific_hour.is_unavailable，Part 2 adapter 漏翻轉會讓 unavailable 邏輯全反（concrete bug，工程師原分析未點出）
- getProviderAvailability 實際有 3 條 caller（public-booking.service、calendar-sync.resolver、meeting-request.resolver），不是 2 條；optional meeting_type_id 的 behavior-preserving 保證必須對三條都成立
- HH:mm→Date 時區陷阱：builder 用 combineDateTime + UTC time-of-day，adapter 產生的 Date 必須與 @db.Time(6) 的 Prisma mapping 完全一致，否則 slot 偏移（呼應 memory: end-to-end equivalence before swap）
- update 寫子表須用 delete+recreate（replace）語意而非 upsert，否則殘留舊 rows；且要在 $transaction 內，避免半寫
- AC3 override 語意未定：meeting-type is_unavailable=false + 自訂時段 = 「取代 weekly」還是「加回時段」？現有 builder 對 is_available exception 是 additive（push period），與「override 取代」可能不同 → 需 PM 確認（見 Q3）
- apply-default 來源語意：複製當下 v2_schedule snapshot，還是 live 連動？若 provider 之後改班表，meeting-type 是否同步？預設一次性 snapshot，需確認
- apply-default 對已有 availability 的 meeting type 是覆蓋還是合併？需明確（預設覆蓋）
- HH:mm regex + end>start 驗證須涵蓋跨日（end<start）、空陣列、重疊時段是否允許等邊界，AC5 回 400 with descriptive message
