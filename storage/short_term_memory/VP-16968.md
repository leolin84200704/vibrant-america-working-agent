## Ticket Analysis

### [2026-06-11 23:00]
Leo 兩個訴求：
1. 為什麼 emr-v2 用 API 新增 ehr_integrations 時，ordering_enabled/result_enabled/sftp_enabled 不跟著 integration_type 變？通過 ≠ integrate 成功。
2. order 流程完全改用 ehr_integrations，不要用 order_clients 比對。

調查結論：
- create (`integration-request.service.ts:183,208-211`)：三個 flag 純由 request body `technicalRequirements` 決定，與 integration_type 完全獨立，無推導、無 validation。預設全 false (DTO + Prisma 雙層)。
- 下游真正 gate 是 flag：order→ordering_enabled=true (`hl7-order.processor.ts:283`)；result 生成→result_enabled OR type∈{RESULT_ONLY,FULL} (`result-generation.service.ts`)；Kafka result→result_enabled AND type (`kafka-report-finished-listener.service.ts:343`)；sftp→sftp_enabled。
- order 真正解析來源仍是 order_clients：`customer-detail-fetcher.service.ts` fetchById/fetchByNpi 以 order_clients 當准入閘門 + 覆寫 kits_options/clinic_id/old_clinic_id；customer 主檔來自 gRPC GetCustomer。processor 的 ehr_integrations 查詢只做 logging（Java parity，配不到不擋）。
- order_clients 只在 2 檔用到：hl7-order.processor.ts(log only) + customer-detail-fetcher.service.ts(真解析)。caller：parser.service.ts、order-intake-enrichment.service.ts。
