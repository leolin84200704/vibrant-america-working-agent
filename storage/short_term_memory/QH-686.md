### 候選方案與正反取捨
- **方案 A — 純 QA 驗證票，對三個 query 跑 live e2e/smoke（auth guard、跨 clinic 擋、cancelled filter、空 date range），證據貼回票，不寫 prod code**
  - 取捨：零風險、最小動作、符合 Test link 與 Confirm-Before-Over-engineering；但若 PM 真正期待的是自動化測試或 endpoint 文件，交付不足需再補一輪
- **方案 B — QA 驗證 + 補自動化 e2e (*.e2e-spec.ts) 或 GraphQL endpoint 文件**
  - 取捨：留下可重複驗證資產、符合 QH-211/824/918/919 QA Hub 系列慣例；但範圍未經 PM 確認即擴大，cancelled 的 'Canceled by patient' filter 與 clinic_customers 無 role filter 有商業語意，自寫 doc 可能與 PM 認知不符而返工

### 風險與邊界案例
- 跨 clinic 越權：token clinic_id != 請求 clinicId 必須被 verifyClinicIdMatch 擋（回 Unauthorized），需 live 驗證非 mock
- getCanceledMeetingRequests 只回 future + cancellation_reason='Canceled by patient'：需驗證過去的、或其他 cancellation_reason 的 request 不會回傳（businesss-semantic 邊界）
- clinic_customers ResolveField 無 role filter — 回 clinic 全部 customer，需確認 PM 是否預期含所有 role（patient/staff）
- 空 calendar_ids array、空 date range、invalid clinicId（NaN）的錯誤處理 (BadRequestException)
- getClinicLocations 的 active_only filter 行為
- 驗證務必對 stage_test 真實 GraphQL endpoint 跑（live-not-mock），不要只跑 mock unit test 就宣稱已驗證
