## Ticket Analysis

### [2026-05-21] 問題重述（Leo 轉述客戶/vendor 回報）
- accession_id: `2604156666`
- vendor 從 5/19 開始下載的 HL7 結果 file，collection date 跑成 `"0001-01-01 06:59:56"`
- 期望值 `"2026-04-16 07:00"`
- "06:59:56" 看起來是 Pacific LMT (-07:52:58) 或 PT (-08:00) 時區套用到 UTC `0001-01-01T00:00:00Z` 後 vendor 端 display 出來的結果
- 時間吻合：5/18 (commit `c0852d0` `[INCIDENT-20260518] Use v2 gRPC as primary with v1 fallback for sample data`) deploy 後第一天 5/19 開始出問題

## Approaches Considered

### [2026-05-21] Hypothesis (待 raw gRPC 驗證)
**Root cause 候選**：v2 gRPC (Go @ `10.224.0.199:32100`) 對 missing `sample_collection_time` 回 Go time.Time 零值序列化 `"0001-01-01T00:00:00Z"`；v1 (Java @ `192.168.60.6:30276`) 回空字串 `""`。

`getValidCollectionTime` (`result-generation.service.ts:1876`) validation 三道：
1. empty/whitespace → fallback ✓
2. isNaN check → ✓
3. `grpcDate > earliestApprovedTime` → ✓（防 server 回 report time）

**沒擋**：`grpcDate < 合理下限`。0001 < 2026 通過 #3，0001 是 valid Date 通過 #2，非空通過 #1 → 直接寫進 OBR-7。

### [2026-05-21] Fix 方向（暫不執行，待診斷確認）
- **A. emr-v2 sanity check**：getValid* 加 `year < 2000 → fallback`。最小改動
- **B. v2 client mapper normalize**：grpc-client-v2.service.ts listSamples mapper 把 `0001-01-01...` → `""`，借既有 empty fallback
- **C. v2 Go server 端不 serialize zero time**：跨 repo，最乾淨但慢

## Lessons Learned

### [2026-05-22]

1. **v1 (`lis` @ `:30276` Java) 跟 v2 (`coresamples_service` @ `:32100` Go) 是平行 service，不是 upgrade 關係** — git blame c0852d0 + 看兩個 repo 的 commit 歷史就知道。誤當 upgrade 替換是這次 incident 根因。
2. **Go gRPC server marshalling 跟 Java/Node 不同**：Go `time.Time{}` zero → `"0001-01-01T00:00:00Z"`、Go `int64(0)` → `"0"`、Go `string("")` → `""`。Java/Node 對 missing field 通常回 null/undefined。跨 language gRPC primary swap 必須驗證 marshalling 邊界。
3. **替換 RPC primary 前必跑 end-to-end diff script** — 已寫進 user feedback memory `feedback_end_to_end_equivalence.md`
4. **outbound HL7 缺 schema validator**：date < 1900 / year=0001 應該在 SFTP push 前 reject。建議：generation pipeline 加 sanity check（不在本 hotfix scope）。
5. **多源 collection_time 問題**：emr_sample DB (`2026-04-16 08:00`) 跟 v1 gRPC (`2026-05-06 18:25`) 不一致 — emr-v2 內部 emr_sample 寫入 vs LIS coreSamples 即時返回 不同 source-of-truth。getValidCollectionTime 用 earliest-approval fallback 蓋過去 — 不算錯但讓 OBR-7 偏離 vendor 期望值（~1 天）。Follow-up 另開。
6. **service.spec.ts 不可信賴**：HEAD baseline 6/6 fail 累積多層 stale（DI / mock shape / payload casing / business behavior）— 沒人跑就沒人修。應該 CI gate 上把該檔 spec 列必過。
7. **host migration vs service migration 區辨**：
   - **Host migration**（VP-16685 cloud mirror）：同 port、同 package、同 backend、只換 IP — 安全。
   - **Service migration**（c0852d0 想做的）：換 port / 換 package / 換 backend language — 是兩個獨立 service，當全新 integration 處理，跑完整 contract 測試。

8. **v1 Java → emr-v2 port 時必查「Java 寫了哪些 DB column，emr-v2 漏了哪些」** — incident 期間 Leo 發現 `hl7_file_input.julien_barcode` + `sample_id_payment` 在 emr-v2 order processing 路徑上沒被 update（Java 版有）。Root cause：`order-finalizer.service.ts` 只在 transactionPay 成功路徑 set partialResult，其他路徑（replay / no-stax / not CUSTOMER_PAY / payment-fail）就空。Java EMR-Backend 用 `GenerateBarcodeForSampleID` RPC 拿 barcode 後一律寫入；emr-v2 的 v2 gRPC client wrapper 漏實作（proto `sample_service.proto:54` 已定義）。修法（PR #124）：補 wrapper + sendOrder 成功後**所有路徑** ensure populate (`partialResult.sample_id_payment ??= sampleId`, `partialResult.julien_barcode ??= await generateBarcodeForSampleID()`)。

9. **outbound HL7 file path 寫進 ephemeral container fs**：`testing.config.ts` 讀 `HL7_OUTPUT_BASE_DIR` env var、ConfigMap 設的是 `HL7_LOCAL_ROOT`（給 inbound order fetch 用，不是 outbound result）— env name mismatch 導致 fallback 到 `/app/test-data/hl7-output`，pod 重啟就沒了。修法（PR #124 + ConfigMap）：兩份 yaml 加 `HL7_OUTPUT_BASE_DIR=/EMR_storage/HL7Result_{prod|staging}` 指 PVC sub-dir。Code 不動（service 已會讀正確 env var）。
