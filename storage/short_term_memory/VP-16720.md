## Failures

### [2026-06-01] INSERT 新 row 漏借 same-customer NPI
**症狀**：3 個新建 Anna pair (2930/8003/36290) customer_npi 寫 null（理由：ticket 表沒列 NPI 欄）。Leo 指出 144510 既有 Anna row 的 customer_npi=1073000691 — 同 customer 跨 clinic NPI 應一致，3 個新 row 該借這個值。

**Root cause**：我的 sibling-borrow 邏輯只從 **same-clinic sibling** 取（borrow clinic_name / address / contact 等 clinic-level 欄位），沒考慮 **same-customer sibling**（不同 clinic 但同 customer_id）—— 那裡有 customer-level 欄位（customer_npi）。

**修法**：事後 `UPDATE customer_npi + effective_npi WHERE customer_id='43262' AND clinic_id IN (2930,8003,36290)`。3 row 補上。

**Preventable**：是。INSERT new pair 前應該分兩個 sibling lookup：
- same-clinic（任一）→ borrow clinic_name, address, contact_*
- same-customer 任一 LIVE row → borrow customer_npi, clinic_npi, effective_npi（如果有）

### [2026-06-01] order_clients 重複 INSERT（Anna 43262 ×4）
**症狀**：我 INSERT 24 order_clients（per pair），但 Anna 43262 跨 4 clinic 同 customer_id → 4 個重複 oc rows（ids 2303/2306/2309/2312）。

**Root cause**：[[VP-16766]] 是 single (cust, clinic) pair，沒呈現「跨 clinic 同 customer」場景，所以 STM 沒明確標 `order_clients` 是 **per-customer not per (cust, clinic) pair**。我直接按 pair × 1 INSERT 24 筆。

**Verify confirmed**：prod 「跨 2+ clinic 的 provider」全部都是 1 個 order_clients row（包含我 INSERT 前的 Anna 43262 應該也只有 1 row）—— 慣例明確。

**修法**：事後 deleteMany ids 2306/2309/2312，保留 2303。21 distinct customers / 21 oc rows ✓。

**Preventable**：是。pre-check 階段應該偵測 PAIRS 內重複 customer_id + 對 INSERT 邏輯 dedupe by customer。

## Retrospective

### [2026-06-01]
**1. 結果**：24 (clinic, cust) pair 全部 FULL_INTEGRATION + LIVE + ordering/result enabled。但 oc 多建 3 筆需修正。

**2. 失敗分析**：oc dedupe rule 沒在 pre-check 處理 → INSERT 24 含 3 重複 → 事後刪。如果 INSERT 階段有 unique constraint on (customer_id, emr_name) 會在 tx 內 throw 自動 rollback，但 prod 沒這 constraint。pre-check 應該 dedupe PAIRS by customer_id 再算 oc INSERT 計畫。

**3. 模式識別**：
- **FTP vendor = Optimantra 模式**：vendor 共用 folder + sftp_folder_mapping 1 筆共用 + sftp_ordering_path=null（走 oc.remote_folder_path）。跟 MDHQ 的 client-specific folder 不同。
- **「Expand to Bidirectional」 ≠ from-scratch**：主流程是 UPDATE 既有 RESULT_ONLY 升級，順帶補 missing pair + missing order_clients。要用 mixed UPDATE+INSERT 而非純 INSERT script。
- **跨 clinic 同 customer 的 oc dedupe**：1 customer = 1 oc row 是 prod 慣例（驗證確認）。

**4. 教訓**：
- 技術：FTP follows Optimantra-class vendor-shared folder pattern; oc per-customer not per-pair.
- 流程：呈報方案要每 ticket 列完整 A/B/C，不沿用前 ticket 編號；「commit」一詞要明確區分 git vs DB transaction。
- 工具：single $transaction 適合 mixed UPDATE+INSERT；pre-check 要 dedupe 結構性重複（avoid post-hoc cleanup）。

**5. 信心度：4/5**（核心邏輯成功 + atomic + verify 全做；但 oc dedupe 失誤事後修補扣 1 分）

## Lessons Learned

### [2026-06-01]
1. **`order_clients` 是 per-customer，不是 per (customer, clinic) pair**：跨 clinic 同 customer_id 的 provider（譬如 Anna Emanuel 43262 在 4 clinic）在 `order_clients` 應該只 1 row。Prod 慣例驗證：「跨 2+ clinic 的 provider」全部 oc 都是 1 row per customer。批次 INSERT 前要 dedupe PAIRS by customer_id 算實際 oc INSERT 計畫。
2. **FOLLOWTHATPATIENT vendor 標準範式**（VP-16720 + 21 LIVE 既有）：
   - `ehr_integrations`：integration_type=FULL_INTEGRATION, status=LIVE, msh06=Practice ID, sftp_result_path=`/Prod/FollowThatPatient/Results/`, sftp_archive_path=`""`, **sftp_ordering_path=null**, report_option=CLASSIC, **kit_delivery_option=NO_DELIVERY**, legacy_emr_service=FOLLOWTHATPATIENT
   - `order_clients`：kits_options=**2**（NO_DELIVERY↔2）, emr_name=`FOLLOWTHATPATIENT`, remote_folder_path=`/Prod/FollowThatPatient/Order/`
   - `sftp_folder_mapping`：**不新增** — 全 FTP 共用 id=251（server=/Prod/FollowThatPatient/Order/, local=/FOLLOWTHATPATIENT/Prod/Order/, use_v2_pipeline=1）
3. **「Expand to Bidirectional」 ticket 是 mixed UPDATE+INSERT，不是純 INSERT**：主流程是把既有 RESULT_ONLY 升級成 FULL_INTEGRATION（UPDATE integration_type + ordering_enabled），附帶補 ticket 列出但 prod 沒既有的 missing pair（INSERT）+ 所有 customer 的 order_clients（很可能既有 RESULT_ONLY 都沒 oc）。`insert-ehr-integration.ts` script 是 from-scratch INSERT 設計，不適合 expand 流程，single Prisma $transaction 才能 atomic 混合操作。
4. **呈報多方案要每 ticket 列完整 A/B/C，不沿用前 ticket 編號**：直接寫「方案 B（推薦）」沒列 A/C 對照會讓 user 困惑「為什麼是 B」。
5. **`--commit` flag 命名要避開 git 雙重含義**：spec script 用 `--commit` 控制 DB transaction 落地（rollback vs commit），但「commit」在 user 心目中常指 git。下次用 `--apply` / `--write` / `--no-dry-run` 更清楚。
