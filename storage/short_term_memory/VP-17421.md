## Root cause (first pass 2026-07-15 — producer attribution SUPERSEDED by the 15:30 entry above)

> Resolution note (dream 2026-07-15): the Bull-replay MECHANISM below is correct, but the
> producer is NOT Portal-Calendar (its pods can't reach redis — no REDIS_URL). Actual
> producer = LIS-transformer `src/calendar/email/reminders.processor.ts`, Bull queues on
> on-prem redis 192.168.60.9:4646, consumed by the stprod pod. See [2026-07-15 15:30] above.

**Producer = legacy `Vibrant-America/Portal-Calendar`** (on-prem `default` ns, 3 pods; reads
MySQL **crm** DB — the same legacy calendar service behind /v1/portal/calendar and VP-16499
specialty seed). It schedules consult reminders as **Bull delayed jobs** on queues
`reminder_24h/48h/15m` (Redis-backed).

Mechanism:
- `email.service.ts sendCustomerEventReminderEmail` enqueues a job with
  `delay=(event.from-now)-reminder_time`, `jobId=event.id`, `attempts:5`, **no
  removeOnComplete** (jobs accumulate in Redis). delay<0 guard exists only at ENQUEUE.
- `reminders.processor.ts sendReminderEmail` (fire time) checks ONLY
  `calendar.findFirst({deleted:0,id})` — **no check that event.from is still in the
  future**. So any stale/overdue delayed job that fires sends a reminder for a past event.
- On **pod restart 2026-07-15 17:59:24Z** (old pod had run since 2026-03-10, 126d), Bull
  replayed overdue delayed jobs → **338 reminders burst at 18:01:16Z** (Tag
  `CustomerEventReminder`, template 33802989, subject "See You in N Hours at Your Clinical
  Consult", From notification@vibrant-america.com) to **149 distinct customer emails**,
  including March-2025 events (Jessica Kaylor). Timing (restart+~2min) is the smoking gun.

## Evidence
- Postmark (token from noti/notification-center pod POSTMARK_KEY): 338 capital-"Hours"
  reminders on 2026-07-15 only (0 on 7/08-7/14), 149 distinct recipients, Tag
  CustomerEventReminder.
- calendar_prod v2_reminder_audit_log: healthy transv2 dispatcher (Tag calendar_prod,
  "in in N hours") writes a row per send and drips per 2-min cron; only 1 row in the
  18:00-18:02Z window vs 338 Postmark sends → burst did NOT come from prod dispatcher.
  0 Kaylor rows in calendar_prod or calendar_dev_new.
- Ruled out: AKS transv2 `-st` (reads calendar_dev_new, 0 reminder rows in 3d) and on-prem
  lis-trans prod/st (platform_type=local → gated, no sends).
- portal-calendar pod startTime 2026-07-15T17:59:23-24Z (on-prem via ssh leo@192.168.60.5).

## Fix (proposed, NOT applied — pending Leo; legacy service mid-migration to transv2)
1. **Code**: in `reminders.processor.ts sendReminderEmail`, after loading the event, skip
   if the event is not still upcoming (e.g. `event.from <= now` or reminder window passed).
   This single guard suppresses all past-event sends regardless of why a job is overdue.
2. **Containment**: clean the stale delayed-job backlog in the portal-calendar Bull/Redis
   queues; add `removeOnComplete/removeOnFail` so jobs don't accumulate. Recurs on EVERY
   restart until done.
3. Note: Portal-Calendar is legacy, being replaced by transformer-v2 (VP-16499). Whether to
   patch legacy vs accelerate migration = Leo/PM call.
