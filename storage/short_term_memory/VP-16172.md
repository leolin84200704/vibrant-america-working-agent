## [2026-06-06 batch 分工辯證 Analysis]

**Goal（目的）**：Epic VP-16163 將 EHR integration 從 provider-level 提升到 practice-level + per-provider membership。本 ticket 負責 server-side RBAC（只有 practice admin / clinicadmin 能對 integration settings 做寫操作，AC1），並在 integration request 提交時建立 practice_integrations + 初始 membership，再加上 membership CRUD、auto-enroll toggle、vendor_provider_fields config 等端點（AC2-6）。Admin 判定權威來源是 gRPC getCustomer 回的 clinics[].customer_roles 含 'clinicadmin'（Kristine 5/4 確認）。

**Classification**：`practice-integration-epic` | **Effort**：L | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：lis-backend-emr-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
改前現狀：(1) RBAC 完全缺席 — integration-request.controller.ts:51 只掛 @UseGuards(JwtAuthGuard)，所有寫端點散落 user.isAdmin / user.internal_user_role（Vibrant 內部 admin），practice admin（clinicadmin）檢查 0 處（已實證）。(2) getCustomer 的 mapper（檔案實為 src/modules/grpc/services/grpc-client.service.ts，_invokeGetCustomer line 380-414，非工程師說的 auto-integrate/ 路徑）完全沒 map clinics / customer_roles，CustomerDetails interface（line 22-41）也沒這兩個欄位 — 雖 proto 有定義。(3) AC2-6 依賴的三表（practice_integrations / provider_integration_memberships / vendor_provider_fields）VP-16164 已於 5/27 落 prod/staging DB（523 practice + 761 membership，100% 驗證）並 raw-SQL 完成，但 schema.prisma model 尚未進 codebase（本 branch 實證 grep 不到 model、prisma/migrations 也無此 migration），故無 generated Prisma client 可寫。

改後為何有效：RBAC 收斂用 gRPC getCustomer 當權威來源（符合 rules「GetCustomer 是 customer 資料權威」），per-clinic 判定 customer_roles.includes('clinicadmin') 避免跨 practice 越權；沿用既有 validateClinicAccess pattern 讓改動最小、可獨立交付且高風險可單獨回滾。

具體改什麼（採方案 A — AC1 先獨立交付）：
- c1: src/modules/grpc/services/grpc-client.service.ts — CustomerDetails interface 補 clinics: Array<{clinicId; customerRoles: string[]}>；_invokeGetCustomer 的 resolve mapper（line 404-412）補 clinics/customer_roles map（response.clinics ?? response.clinics_）。只有一個 mapper（tryCloudThenOnPrem 共用 _invokeGetCustomer），不需改兩處。加單測。
- c2: 不要做成純 route guard。既有 mutation 端點的 target clinic 來自 DB-loaded integrationRequest.clinic_id（line 391/433/480/...），不是 request param；只有 create 端點 clinic 來自 dto.clinicId（line 192）。因此新增一個 validatePracticeAdminAccess(user, clinicId) helper（仿既有 validateClinicAccess），在各寫端點 load record 後呼叫；create 端點用 dto.clinicId。helper 內 call getCustomer(user.customerId) → find clinic → 檢查 customer_roles。fail-closed（403）為預設（RBAC 場景 fail-open 是安全漏洞，覆寫 ticket-routing「graceful fallback」通則）。
- 前置（解封）：把 VP-16164 已落 DB 的三表 model 補進 schema.prisma 並 prisma generate（依 patterns 兩 schema）。AC2-6 須以 VP-16164 v2 實際 DB 欄位為準（SFTP/MSH/option 等多在 membership 層，非 practice 層；status enum practice=pending_approval|in_progress|live_and_testing|active|rejected|suspended，membership=pending_approval|active|suspended|removed；added_by 含 practice_admin），不可照 ticket/PRD 字面。
- AC2: integration-request.service create 流程 INSERT practice_integrations + 初始 membership。AC3/AC5: 新 GET vendor_provider_fields by vendor + Add Provider 驗證 vendor 必填欄位。AC4: PATCH auto-enroll 更新 practice_integrations.auto_enroll_new_providers。AC6: Remove Provider 設 membership.status='removed'（soft delete）。審計沿用既有 ehr_integration_status_history pattern。

### 候選方案與正反取捨
- **方案 A：AC1（RBAC）先獨立交付，AC2-6 等 schema model 進 codebase 再做**
  - 取捨：立即交付 403 防護、reusable helper、高風險可單獨回滾；但 RBAC 上線前須確認 FE/token 已帶 clinicadmin，否則可能誤擋既有 prod provider，且 AC2-6 端點若回頭改 helper 簽章可能兩次 review
- **方案 B：等三表 model 進 codebase 後一次做 AC1-6**

- **純 NestJS route guard（@RequirePracticeAdmin 從 request 取 clinicId）**
  - 取捨：工程師原提案；對 create 端點（dto.clinicId）可行，但對 :id mutation 端點不可行 — target clinic 來自 DB-loaded record，route guard 在 handler 前拿不到，故改用 load 後呼叫的 validatePracticeAdminAccess helper 較貼合既有 pattern

### 風險與邊界案例
- RBAC 上線可能誤擋既有 prod provider：若 FE（VP-16170）/token 尚未帶 clinicadmin，原本能操作的 provider 會突然 403。需確認 rollout 順序，必要時 feature flag 或先擋部分端點
- getCustomer 無對應 clinic（provider 不屬該 clinic）或 rpc 失敗：RBAC 須 fail-closed（403），不可 silent fallback 放行
- 工程師的檔案路徑錯誤：mapper 在 src/modules/grpc/services/grpc-client.service.ts（非 auto-integrate/）；review/實作時勿照錯路徑改
