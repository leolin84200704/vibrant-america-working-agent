## Lessons Learned

### [2026-05-06 11:40] 技術
- **`kit_delivery_option` 正確對應規則**（per ParseHL7.java:930 switch）：
  - kits_options=0（only non-blood）→ NON_BLOOD_ONLY
  - kits_options=1（full kits, both ship）→ BOTH_BLOOD_AND_NON_BLOOD
  - kits_options=2（both supplied by client）→ NO_DELIVERY
- **lis-backend-emr-v2 不 consume `ehr_integrations.kit_delivery_option`**：runtime 行為靠 EMR-Backend Java 看 `order_clients.kits_options`。ehr_integrations 是 informational。
- **改 schema.prisma `@default` 不需要 prisma migration**：lis-emr-v2 透過 PrismaClient 寫，application-side default 已 cover；DB column-level DEFAULT 留之後 baseline 時統一處理。
- **Cross-table UPDATE LEFT JOIN 在 MySQL Prisma**：用 `prisma.$executeRawUnsafe` 包在 `prisma.$transaction` 中可正確 rollback。注意 transaction `timeout` 要拉長（default 5s 對全表更新可能不夠），這次用 `{ timeout: 60_000 }`。

### [2026-05-06 11:40] 流程
- **LTM「規則表」要 cross-check code source**：LTM 是 cached 結論，可能寫錯（VP-16423 釐清表就錯了）。引用 LTM 規則表前先 grep 對應程式碼確認；發現有錯時當下修正 LTM。
- **PM ticket + 整理性質改動混合**：Leo 看到資料品質問題時可能順手提「全表整理」，agent 要主動 surface「PM 原 request 還在嗎」+ 同步呈報兩件事的數字。
- **Apply prod 不可逆改動前必跑 dry-run + 顯示數字**：即使 Leo 說「直接做」，呈報 dry-run 結果（多少 row 變化、target row 預期狀態）+ pre-check sanity guard 是 Safety First 必要程序。

### [2026-05-06 11:40] 工具
- **`prisma.$transaction(async (tx) => { ... }, { timeout: 60_000 })`**：interactive transaction 帶 timeout 設定，default 5s 對全表更新可能不夠，全表 update 拉到 60s 以上。
