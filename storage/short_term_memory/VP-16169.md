### 候選方案與正反取捨

- **Method C: split into two tickets -- one for the send/resolver/log service layer (do now), one for trigger wiring + idempotency + reminder (after deps).**
  - 取捨：Cleanest tracking and clean AC mapping per ticket, but adds Jira overhead and needs PM to re-scope the story; only worth it if PM wants strict AC-to-ticket traceability.

### 風險與邊界案例
- Template-ID send path mismatch: existing service cannot send by Postmark template ID (fixed email_template enum) -- building the new path on the wrong assumption (direct Postmark vs utility endpoint) wastes the whole ticket. Resolve Q1 BEFORE coding.
- Idempotency race (AC6): without notified_practice_at guard, retries / concurrent provisional creation double-send Trigger A. Guard must be a transactional check-and-set, not read-then-write.
- Multi-admin fan-out (AC5): if resolvePracticeAdmins returns 0 admins, decide fallback (skip vs route to internal team) -- silent zero-recipient send must be logged, not swallowed.
- Provider-on-#6 gating (AC2): only include provider if active Portal account exists -- needs a real Portal-account lookup, not just provider record existence; verify against live data not mock.
- Business-days math for Trigger C: '3 business days' must exclude weekends/holidays; naive +3 days will fire early. Confirm holiday calendar source.
- Env-specific template IDs (AC8): staging vs prod template IDs differ (45072742 vs 45021853 etc.) -- a hardcoded ID will send wrong template in one env. Drive by NODE_ENV/config.
- Provider ID vs Practice ID: recipient resolution must resolve practice admins via the practice, not the provider id (memory iron rule).
- AC7 banner table ownership unclear -- if it is portal-DB-owned, emr-v2 cannot write it and that AC belongs elsewhere; do not create a phantom table in emr-v2 schema.
- Fire-and-forget pattern (.catch at integration-request.service.ts:288) means send failures never surface to caller -- AC10 'permanent failures surface in logs' relies entirely on the audit log row + logger.error; verify a failed Postmark send actually writes a 'failed' row.
