**B. Thinnest Slice — Notification infra + recipient resolver**（推薦）
範圍：
1. 新 `NotificationService` (or 加到既有 `emailNotificationService`)：包 Postmark template send + retry + audit log
2. `RecipientResolver`：
   - `resolveInternalTeam(practice_id)` → Sales (RSM/CAM/SD) + emr.integrations@vibrant-wellness.com
   - `resolvePracticeAdmins(practice_id)` → 透過 getCustomer rpc 取 clinics.customer_roles==='clinicadmin'
   - `resolveProviderEmail(provider_id)` → portal user lookup (only if Portal account exists)
3. Template alias 常數 + env-conditional template ID (Template 4/5/6 prod vs staging)
4. Payload mapping function (provider_name / npi / practice_name / emr_vendor / quarantine_id / integration_level / CTA URLs)
5. Audit log table: `notification_send_log` (entity_id, entity_type, template_id, recipients, postmark_msg_id, status, error, sent_at)
6. **Stub publish methods**: `notifyProvisionalProviderCreated(provisionalProviderId)` 跟 `notifyMembershipActivated(membershipId)` — 簽章 ready 但內部走 audit + Postmark，等 VP-16168 + VP-16164 落地後 caller 接上
7. In-app notification record write (portal notifications table — 寫入 schema TBD)
8. Idempotency guard: 用 `notification_send_log` `WHERE entity_id + template_id NOT EXISTS` 而非 `provisional_providers.notified_*`（避免依賴未 ship schema）

範圍**外**：
- Trigger A 真正 hook 進 VP-16168 (沒 owner code)
- Trigger B 監聽 `provider_integration_memberships.status` change（schema 未 ship）
- Trigger C 3-business-day cron（需要 notified_* timestamp on provisional_providers，VP-16164 schema 範圍）

時程：1.5-2 天，5/22 可交付這層
