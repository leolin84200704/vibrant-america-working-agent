### [2026-06-08] Explore 驗 fee 路徑：fee=maxPrice(priceVa,priceVw)，但 swap 必須在 price-lookup 之前

Explore agent 翻 lis-backend-emr-v2 結果：
- `getLabProcessingFee` 在 `order-test-client.service.ts:87-115`，呼叫點 `parser.service.ts:453-460`，fee 寫入 `parser.service.ts:469`（OrderFrontend.lab_processing_fee）。
- pricing 來源 `order-mapping-cache.service.ts:200` 用 `getLegacyPackagePriceMapping?currency=usd`。
- fee 取值用 `parser.service.ts:549 maxPrice(priceVa, priceVw)` = 取兩者較大。GZ priceVa=-1 → fee=priceVw。**故上游把 485 priceVw 改 630 → maxPrice(-1,630)=630=+$80，方案成立。**
- **關鍵時序限制**：price 由 emrcode 查表發生在 OBR 解析期 `obr-parser.service.ts:222`（`emrCodeToPackagePriceMap.get(groupCode.toLowerCase())`），orderItems 從此 cached PackagePrice 組出（`parser.service.ts:378-391`），fee 此刻就鎖定，且 getLabProcessingFee 只收 price 不收 emrcode。
  - → **swap 必須在 line 222 查表之前發生**。既定 swap 點（ClearSky remap ~line 99）在 222 之前，故在那裡 swap 正確：查表會用 GZ-NY 的 package → 帶上游改過的 priceVw。
  - → 若 swap 放在 222 之後或下游 parser.service.ts，fee 仍用舊 463 價格 → 靜默少收 $80（VP-16175 類靜默金額錯誤）。
- **結論**：emr-v2 不需寫任何 fee code，前提是 swap 接在 emrcode→price 查表之前。
- **impl 待驗（非阻塞）**：(a) ClearSky remap block 改的欄位是否正是 line 222 查表用的 groupCode/emrcode；(b) line 93-99 與 206-226 確在同一 parseObr flow 且前者先執行。
