## Decisions Made
### [2026-05-28]
**1. 4 個 RPC ticket 對 EMR repo runtime no-op（含 proto 殘留 disclosure）**
- **Runtime 層面**：grpc-client.service.ts 只 `new` 5 個 client（Sample / Customer / Patient / TestResultInfoGrpc / ReferenceRange）。整個 `src/` 對 `ShippingService` / `AuditLogService` / `IssueService` 字串 grep 零匹配。`loadSync` 載入的只有現役 5 個 proto。4 個 on-prem port 整 repo 完全沒出現。
- **但 proto file 存在 repo 裡（dead artifact）**：`shipping.proto`（src/proto、src/proto-v2、src/proto-v2/proto/）、`auditlog.proto`（src/proto/）、`issue.proto`+`UserResolveIssue.proto`（src/proto-v2/）— 沒被任何 TS 載入。Dashboard 連 proto 都沒。可能是上游 service repo 同步過來預備、或半成品。不影響 runtime 但要 disclose 避免後續 grep proto 誤判。
- 出現的 `Shipping` / `AuditLog` string match 全是無關：`Shipping address` 是 postal address log、`createAuditLog` 是內部 DB function 寫 audit table、`international_shipping_fee` 是 order DTO field
- VP-16685 STM 也記載：那 4 個 blocked RPC 是 LIS-transformer-v2 (Calendar) 使用，不是 EMR
- 跨 repo 動 LIS-transformer-v2 超出 EMR-side scope，當 Leo 不在不擅自跨 repo

**2. Kafka 不遷雲端**
- empirical evidence：AKS↔on-prem Kafka cluster 完整可用（已驗證 metadata + 真實 consume）
- 既有 AKS 服務（ehr-chat、ehr-workflow）ConfigMap 明寫 `KAFKA_BROKERS=192.168.60.9-11:9095` → 跨網路用 on-prem Kafka 是現役 production pattern
- 遷 Event Hub 需要：(a) KafkaJS code 加 SASL_SSL（目前 plain）、(b) Event Hub 端 producer 也遷（EMR 只是 consumer，producer 在 result pipeline，若不遷 → cloud topic 空訊息）、(c) 跨 cluster consumer-group migration coordination

## Addendum — Cloud↔on-prem Kafka 是 dual-published（2026-05-28 深夜更正）
**Leo 質疑後重新驗證的結論**：

用 kcat `-o s@<ms>` timestamp-seek 對齊兩邊到同一 4hr window：
- 同 4hr window：cloud 22 條 `report_finished` / on-prem 37 條，**event_id 交集 8 / sample_id 交集 8 / accession_id 交集 8 / patient_id 交集 8**
- 8 個 shared sample 的 event_id + timestamp millisecond exact 完全一致 → **同一邏輯事件雙寫到兩個 cluster**

**翻案的證據**：
| sample_id | event_id | event_time | 兩邊 byte-identical? |
|---|---|---|---|
| 2563137 | 6db020da-6bf8-4baf-b71d-eed8ffa97cf4 | 00:09:26.740Z | ✅ |
| 2553923 | fdf4f345-e589-46ab-b7fd-0a92daf3b709 | 01:25:52.366Z | ✅ |
| 2557892 | e6512a98-a742-43ff-8811-e43b50c4a923 | 01:24:55.400Z | ✅ |
| 2542983 | 54da288d-fef7-429a-9202-b1b2ed91f7fb | 01:27:46.463Z | ✅ |
| 2558698 | 9044422d-35b7-4732-aa35-f36aecf146ff | 01:23:53.390Z | ✅ |

**Cloud topic mapping**：on-prem topic `lis-general-events` = cloud Event Hub `general-sample-events`（namespace `general-events`, host `general-events.servicebus.windows.net:9093`）。名字差距大、容易誤判。

**結論調整**：
- 不是「不能遷 Kafka」，是「不急遷」 — dual-published 表示遷與不遷都會收到一樣事件
- 真要遷只需 code 加 SASL_SSL（KafkaJS 加 `ssl: true` + `sasl: {mechanism: 'plain', username: '$ConnectionString', password: <conn string>}`）+ 換 broker host + 換 topic name；**不用 producer 端協調**（已 dual-publish）
- consumer group offset 跨 cluster 不能 carry，但 EMR `fromBeginning: false` (latest only) 影響很小（切換瞬間漏的訊息 < 1 秒 production）

## Lessons Learned
### [2026-05-28]
1. **AKS↔on-prem Kafka over VPN 是 LIS 既有 production pattern**：ehr-chat / ehr-workflow ConfigMap 明寫 on-prem brokers + 我從 AKS pod kcat 實測 metadata + consume 都通。EMR 上 AKS 後 Kafka 配置不用動，KAFKA_BROKERS 留在 192.168.60.9-11:9095 即可。但 cloud Event Hub 也有 dual-published 等價 stream（見 Addendum），所以遷雲也可行 — 不再是「絕對不該遷」。
2. **多 partition / 流量懸殊 topic 跨 cluster 比對必須用 timestamp seek 對齊**：`kcat -o -N -c N -e`（最新 N 條）對 partition 不均的 topic 是 false negative 製造機 — 各 partition 從各自的 latest 往前抓 N，覆蓋的 wall-clock window 可能差數小時甚至數天。正確：`kcat -o s@<ms>` timestamp seek 強制兩邊從同一 wall-clock 起點抓。我這次連翻三次結論（dual→獨立→獨立→dual）就是因為前兩次 sample 用「最新 N」沒對齊窗口。
2. **Kafka 沒有 gRPC 那種 per-call fallback**：consumer 是長連接訂閱，連哪個 cluster 就只收哪邊事件。"cloud primary + on-prem fallback" 對 Kafka 不成立 — 要嘛 producer + consumer 同步遷，要嘛 dual-consume 自己 dedupe（complex）。
3. **Verification 流程從淺到深**：nc TCP（最淺）→ grpcurl/kcat handshake → metadata fetch（含 advertised.listeners 解析）→ 真實 RPC call / Kafka consume（最深，含 application protocol）。`nc OK` 不等於「能用」，至少要走到 metadata fetch 才算驗到 protocol layer。
4. **跨 repo scope 紅線**：Leo 給 autonomy 不等於授權跨 repo。同 repo 內可判斷推進，跨 repo（如 LIS-transformer-v2）需明確指令。
5. **`.svc.cluster.local` 從 AKS 外 NXDOMAIN**：cluster-internal DNS 不會被外部解析。任何 outbound dependency 用這格式的，consumer pod 必須在同一 cluster；on-prem EMR 期間用不到 — 自然 fallback 模式是強制條件。
