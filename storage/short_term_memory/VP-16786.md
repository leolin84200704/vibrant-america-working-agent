# VP-16786 - [BE] Migrate DASHBOARD_RPC to cloud interface

## Ticket Analysis

**Goal（目的）**：把 transformer-v2 的 dashboard gRPC client 指向 cloud in-cluster service (lis-dashboard-prod-rpc-service.default.svc.cluster.local:5800)，作為 EMR/Calendar 上 AKS 遷移 (epic VP-16776) 的一環。AC：(1) 所有 Calendar/EMR code path 連 cloud；(2) 移除 on-prem reference 192.168.60.6:31987；(3) dashboard 相關流程 regression 通過。Consumer 是 transformer-v2 的 src/dashboard/，不是 lis-backend-emr-v2（後者對此 RPC 無 client code）。

**Classification**：`rpc-cloud-migration` | **Effort**：S | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
關鍵修正（已實測，推翻 engineer 的「改前現狀」前提）：(1) on-prem endpoint 192.168.60.6:31987 在整個 repo grep 零匹配——src/ 與 yaml/ 都沒有。(2) yaml/prod-env.yml:6 的 DASHBOARD_RPC 值「已經」是 cloud endpoint `lis-dashboard-prod-rpc-service.default.svc.cluster.local:5800`。所以 prod runtime 上 dashboard 其實已連 cloud，AC1/AC2 在 prod config 層面實質已達成；本 ticket 主要是「在 code 形式上對齊 sibling 的雙-client cloud pattern + 補齊 st 環境 config + 驗證」。\n\n改前現狀：dashboard.service.ts:42-43 @Client(microserviceOptions_dashboard)→client5→getService('DashBoardService') (L87)，6 處 call site：userTimelineFetch×2 (L361-362)、eventFlag (L526)、eventUnFlag (L582)、batchArchiveEvents (L635)、batchUnarchiveEvents (L688)。microserviceOptions_dashboard 讀 process.env.DASHBOARD_RPC。microserviceOptions_dashboard_cloud (grpc.option.ts:252) 讀 DASHBOARD_RPC_CLOUD，已建好但 grep 全 repo 零引用（未接線）。DASHBOARD_RPC_CLOUD 不存在於任何 yaml。\n\n改後為何有效：比照已上線 sibling pattern（customer.service.ts:42-81 + 11 個 calendar service 都用 @Client(*_cloud) + withCloudFallback）。cloud 健康走 cloud；transient gRPC error (DEADLINE_EXCEEDED/UNAVAILABLE/UNKNOWN/INTERNAL) 自動退 on-prem；business error 直接 propagate（不浪費 latency）。GRPC_CLOUD_FALLBACK_ENABLED=false 是 prod kill switch。fallback 形式與 epic 一致、最低風險，且即使 transformer-v2 在 on-prem 期間（.svc.cluster.local 從 AKS 外 NXDOMAIN）也不斷。\n\n具體改什麼（全在 LIS-transformer-v2）：\n1. src/dashboard/dashboard.service.ts：新增 @Client(microserviceOptions_dashboard_cloud) private client5Cloud + onModuleInit 加 if(this.client5Cloud) this.dashboardServiceCloud = client5Cloud.getService('DashBoardService')（比照 customer.service.ts:60-62 null-guard）；6 個 call site 改成 this.dashboardServiceCloud ? await withCloudFallback('opName', ()=>cloud, ()=>onprem, {warn}) : onprem。注意 userTimelineFetch 兩個並行呼叫 (Promise.all L361-362) 各自包 fallback。\n2. import { microserviceOptions_dashboard_cloud } from '../grpc.option' 與 withCloudFallback from '../calendar/shared/with-cloud-fallback.util'。grpc.option.ts:252 已存在，無需改。\n3. Config（iron rule：新 process.env.X 同 PR 補 config）：yaml/prod-env.yml 與 st 對應 configmap 新增 DASHBOARD_RPC_CLOUD（prod 值同現有 cloud endpoint）；yaml/prod.yml + yaml/st.yml 的 Deployment env 區段新增 DASHBOARD_RPC_CLOUD 的 configMapKeyRef（比照現有 DASHBOARD_RPC 區塊 L102-106）。注意 yaml/st-env.yml 目前連 DASHBOARD_RPC 都沒設——需確認 st 是否有 dashboard service，否則 st 走 on-prem fallback。\n4. Branch feature/leo/VP-16786，base = stage_test（目前 repo 在 feature/leo/VP-16513，切前確認 clean）。切 branch 後跑兩個 prisma generate（calendar PostgreSQL + prisma2 MySQL），npm run build / start:dev 必須過。\n5. dashboard 目前無 spec 檔——若新增最小 spec，mock client5Cloud 驗 cloud-primary 與 fallback 兩分支（但這是 mock，不可當「線上驗證」）。\n6. 順手評估同檔 client3 (ShippingService L86) → 屬 VP-16784 範疇，可同 PR 一起遷以省事。

### 候選方案與正反取捨
- **方案 B 硬切：直接把 client5 換成 microserviceOptions_dashboard_cloud 並從所有 config 移除 DASHBOARD_RPC（字面滿足 AC2）**
  - 取捨：無 fallback，cloud 不通即 timeline/flag/archive 全掛；dashboard image 是 :latest mutable tag（STM 5/28 已標 drift 風險）；與 11 個 sibling service 的雙-client pattern 不一致。風險最高，不建議。
- **方案 C 幾乎 no-op：prod config 既已指 cloud，僅補 st config + 加最小驗證、code 不動**
  - 取捨：改動最小但留下 code 與 sibling pattern 不一致（無 cloud client/fallback），未來四張 RPC 票各走各的 style；且 epic 既定就是雙-client，QA reviewer 可能要求對齊。除非 PM 確認只要 config 達標。
- **方案 D 四票合一 PR (16784/85/86/87)：同 repo 同 pattern dashboard/shipping/issue/audit 一起遷**
  - 取捨：省 review/部署 overhead、pattern 一致；但 blast radius 大、單 PR 失敗影響四個 RPC、regression 範圍變大。建議拆成同 branch 系列或逐張 PR 由 Leo 決定。

### 風險與邊界案例
- 重大前提修正：engineer 宣稱 on-prem DASHBOARD_RPC=192.168.60.6:31987 且 cloud env 不存在——實際 grep 證實 31987 全 repo 零匹配，且 yaml/prod-env.yml 的 DASHBOARD_RPC 已是 cloud endpoint。任何方案描述須以此為準，別在 PR 宣稱「移除了 31987」（根本不存在）。
- AC2「fully removed on-prem reference」與保留 fallback 字面衝突——但 sibling customer/clinic 全保留 fallback 且已上線，需 PM 確認字面 vs 實質（與 VP-16685 同口徑即可）。
- dashboard image lis-dashboard-prod:latest 是 mutable tag——proto/contract 可能漂移。驗證須對實際 cloud service 跑真實 RPC（grpcurl/dist service），非只 mock spec（記憶 verified_means_live_not_mock）。
- st 環境 config 缺口：yaml/st-env.yml 連 DASHBOARD_RPC 都沒設定。若 st 無 dashboard cloud service，DASHBOARD_RPC_CLOUD 也會缺→client5Cloud 為 undefined→自動走 onprem 分支（onprem 在 st 也可能未設）。需確認 st 是否有可達 dashboard endpoint，否則 st regression 會失敗或靜默走空。
- userTimelineFetch 是 Promise.all 兩個並行呼叫——包 fallback 時兩個各自獨立 fallback，勿共用一個 try。
- DashBoardService 是 camelCase service 名 + snake_case proto field（keepCase:true）——cloud client 必須用同一 dashboard.proto（grpc.option 已指同 protoPath），確保 marshalling 等價（記憶 end_to_end_equivalence）。
- regression（AC3）須 peer-observed：不只看 transformer-v2 log，要確認 dashboard service 端真的收到並回正確 timeline/flag 結果。
