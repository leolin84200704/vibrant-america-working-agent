## Approaches Considered

### [2026-05-18]
Three interpretations of "provide timezone to FE for emails":
- **A**. Postmark `templateModel` adds `timezone_abbrev` variable per template
- **B**. FE-composed email endpoint missing `timezone` field
- **C**. Some admin/preview UI mislabeled as "email"

Section A — Postmark template model — was the only actionable gap explore agent surfaced (`reminder.service.ts` / `event.service.ts` already TZ-aware on dates but template_model had no TZ field).

Initial proposal: add paired `*_timezone` field per time field in YAML. Leo rejected: "不能改 yaml file 啊, email template端什麼都沒改, 重點是那些field要填入timezone" → **embed TZ abbrev inside existing time field VALUES**, YAML schema unchanged.

Second design decision: per-recipient TZ format or single eventTimezone for all? After framing as "time/timezone correctness" Leo chose **per-recipient** (A): each recipient sees time in their own calendar TZ, with matching TZ abbrev embedded.

Third refinement (during Step 6 review): **only time-bearing strings carry TZ abbrev; pure dates do not.** `consult_date: "05/12/2026"` (no TZ), `consult_time: "09:30 AM PDT"`, `appointment_date_time: "05/12/2026, 09:30 AM PDT"`.

## Decisions Made

### [2026-05-18]
- TZ format: short abbrev (`PDT` / `EDT`) via `Intl.DateTimeFormat` with `timeZoneName: 'short'`, DST-correct per date.
- Embed in field value, NOT new YAML keys.
- Per-recipient: each recipient's `v2_calendar.timezone` drives both the time format AND the abbrev.
- Date fields stay as `MM/dd/yyyy` (no TZ); time and dateTime fields get TZ appended.
- All 23 template variants in scope (full sweep), including pre-existing buggy paths in `meeting-request.service.ts`.

## Failures

### [2026-05-18] First-pass YAML schema change
I added `*_timezone` paired fields to both YAML files at Step 5 start, planning to use them in templateModel. Leo immediately reverted with "不能改 yaml file 啊". Root cause: I conflated "BE config describing template variables" with "Postmark template body in Postmark cloud" — Leo treats the YAML AS the template-end contract; adding fields there is an API change even if Postmark template body doesn't reference them. Lesson: when spec says "embed timezone", default to enriching existing values, NOT adding paired fields.

### [2026-05-18] Initial date-also-with-TZ assumption
At Step 6 review I included `consult_date: "05/12/2026 PDT"`. Leo corrected to "date does not need TZ; only time-bearing strings do". My reasoning was "dates cross midnight differently in different TZs so date should carry TZ" — technically true but display-copy convention does not write dates with TZ. Lesson: distinguish display-copy convention from technical correctness; the latter does not always require the former.

## Lessons Learned

### [2026-05-18]

1. **When PM says "fill X with Y", default to enriching X's value, NOT adding Y as a paired field.** VP-16664 spec was "each time field needs timezone" — my first read added `*_timezone` paired YAML fields. Correct read: `consult_time: "10:30 AM PDT"` (TZ in value), not `consult_time + consult_time_timezone` (paired field). YAML field schemas represent the template's external contract; adding fields = changing the contract.

2. **Display-copy convention ≠ technical correctness.** A calendar date `"05/12/2026"` could mean different absolute days in different TZs, but display copy doesn't usually write "May 12, 2026 PDT". Only time-bearing strings (`"10:30 AM PDT"`, `"05/12/2026, 10:30 AM PDT"`) need TZ qualifier. Don't over-apply TZ embedding "for technical correctness" if the convention doesn't ask for it.

3. **Per-recipient TZ format is the default for recipient-targeted emails.** Single-TZ-for-all (provider TZ) makes recipient mental-convert. `reminder.service.ts` (VP-16391) already did per-recipient; `event.service.ts` did not. When discovering an inconsistency between two builders in the same flow, refactor to whichever pattern matches the user-facing correctness invariant — not whichever is easier.

4. **"BE config YAML" and "cloud template body" are the same surface to a non-developer PM.** Leo conflates YAML field schema with Postmark template body because both define the "external contract" of an email template. When changing email template field schemas, get explicit OK even if no body change is needed.

5. **Pre-existing bugs in the same flow should be fixed in the same ticket** when (a) they share the correctness invariant the ticket is enforcing, AND (b) leaving them creates inconsistency (e.g. some builders TZ-aware, one not). VP-16612's `meeting-request.service.ts:715-725` toLocaleString bug was orthogonal scope but fixing it alongside VP-16664 prevented a future "why is this one builder different" investigation.
