## Ticket Analysis

### [2026-07-08] 起因
Leo: 「看一下 Jenkinsfile 為什麼 staging 沒有雲上的。應該要上雲」。
Root cause（改前現狀）: Jenkinsfile `staging` block（原 L275-335）只做 internal registry push + ssh 到 on-prem cluster `kubectl set image lis-emr-v2-deployment -n default`。VP-17291 Phase A 的 ACR push + AKS apply 只加在 `main` block（`if (params.DEPLOY_TO_AKS)`），staging 從未擴充。AKS ingress 只有 `/v1/lis/emr-service(/|$)(.*)`（prod）；`/v1/lis/emr-service-staging` 實測 404。staging 唯一 public 入口 = legacy `www.vibrant-america.com/lisapi/v1/lis/emr-service-staging/api/v1/…`（無 rewrite，直達 on-prem pod，2026-07-08 實測與 NodePort 31320 同 pod）。

## Approaches Considered

### [2026-07-08]
鏡射 prod Phase A（web-only endpoints 上雲、pipeline 留 on-prem）＝選定方案。獨立 `-staging` 物件（deployment/service/ingress/PVC 全分開）而非改 prod ingress 加 path — staging deploy 永遠碰不到 prod 物件。
Ingress path 安全性：prod regex `(/|$)` guard 使 `emr-service` 不會吞 `emr-service-staging`；nginx 長 prefix 優先。staging ingress 不掛 cert-manager annotations（同 host 兩個 ACME-managed ingress 會搶同一 order），直接 reuse `tls-secret`。

## Decisions Made

- POD_ROLE=web + ENABLE_KAFKA_CONSUMER=false（direct env 蓋 envFrom）— staging pipeline 留 on-prem，零重複處理
- envFrom `lis-emr-v2-config`（staging config；AKS default ns 為 source of truth，deploy 時 export+清 managedFields 後 apply 進 emr-v2 ns，手法同 main block）
- 獨立 PVC `emr-pv-claim-staging` 20Gi azurefile-csi
- 新 param `DEPLOY_STAGING_TO_AKS`（default true）
- PR target `staging`（emr-v2 慣例；Jenkinsfile staging build 讀自己 branch 的檔）；merge 後同步 main（PR 內註明）
