## Ticket Analysis

**Goal（目的）**：為 coreSamples gRPC service 的 sample-attribute 檢查 RPC 補一份 API contract 文件,讓 QA/前端/整合方知道如何呼叫、輸入輸出語意與邊界行為。關鍵是要清楚標註 BatchCheckSampleAttributes 已是 no-op stub,應改用 NewBatchCheckSampleAttributes。純文件任務,非 code change。

**Classification**：`transv2-rewrite-doc` | **Effort**：S | **Confidence**：5/5 | **Ticket fetched**：是
**Affected repos**：LIS-backend-v2-coreSamples

### 建議方案（改前 / 改後為何有效 / 改什麼）
目的:替已上線的三個 RPC 補可分享的 API 文件。改前現狀:文件缺失,repo 內只有 handler 上的 godoc @Summary/@Router 註解,無 swagger/markdown 慣例;整合方無從得知 BatchCheckSampleAttributes 是空 stub。改後為何有效:把 contract + deprecation 寫清楚可避免誤用 no-op stub。具體寫什麼(已逐一對照原始碼驗證,定義在 LIS-backend-v2-coreSamples/proto/sample_service.proto:62-64 與 messages 158-207):

(1) CheckSampleAttributes — handler/sample_handler.go:1513 + service/sample_service.go:1199。輸入 sample_attribute_name(enum: sample_id=0 / accession_id=1)+ sample_attribute_value(string);輸出 sample_attribute_value_exists(bool)。邊界:值先 TrimSpace;空字串回 false(不報錯);sample_id 非整數→common.Error log + 回 false;DB 錯誤才回 error;未知 enum 回 error。

(2) BatchCheckSampleAttributes — handler/sample_handler.go:1550 是 `return nil` 空 stub,永遠回空結果。DEPRECATED,請改用 New 版。注意其 proto response(BatchCheckSamplesAttributesResponse, proto:187-189)用 `repeated int64`,與 New 版的 `repeated string` 不同 — 文件需註明(此處更正工程師「三個都是 string」的說法,但因是 no-op 不影響功能)。

(3) NewBatchCheckSampleAttributes — handler:1563 + service/sample_service.go:2315。輸入 repeated string;輸出 valid_sample_attribute_list / invalid_sample_attribute_value_list(皆 string)。行為:>500(MAX_SINGLE_BATCH_SIZE)自動分塊,CHUNK_SIZE=200(service:2310-2311);查詢時 trim+去重,但 response 保留原始順序與重複;空字串/非整數歸 invalid;空輸入回兩個空 list;無效 enum name 回 error。

落點:沿用團隊 API doc 慣例 = Confluence structured markdown(對齊 user memory 的 API doc format 偏好),不要塞進 repo(repo 無文件慣例,自創 docs/ 結構等於 over-engineer)。先確認 Confluence space/parent(建議沿用同性質 QH-824 Calendar doc 的落點/模板)。附 grpcurl 範例 payload。三個 RPC 全寫(因 New 版才是實際使用版本)。

此為文件任務:不需 branch / start:dev / prisma generate;不要在此 ticket 改動 BatchCheckSampleAttributes stub(另開 ticket)。

### 候選方案與正反取捨
- **倉庫內補強(README 區段或 docs/)與 code 同源、PR 可一起 review**
  - 取捨：repo 無文件慣例,新增 docs 結構=自創 pattern;QA ticket 通常要可分享連結而非 repo 檔。不採用。
- **只寫 ticket 標題的兩個 RPC(Check + Batch)**
  - 取捨：會漏掉實際在用的 New 版,讀者拿到 Batch stub 反而誤用。應三個都寫。

### 風險與邊界案例
- BatchCheckSampleAttributes 是 no-op:文件若不顯著標註 DEPRECATED,整合方會誤用空 stub 拿到空結果
- Batch(int64)vs New(string)response 型別不同:文件需明確區分,否則 client 反序列化會出錯
- NewBatch response 保留原始重複與順序、但內部查詢去重:文件需講清 caller 不能假設 response 已去重
- 分塊行為(>500 走 chunk=200):大量輸入時 valid/invalid 順序為 chunk 串接,跨 chunk 仍對應原始輸入順序
- sample_id 非整數與空字串都歸 invalid(NewBatch)/ 回 false(Check):兩個 RPC 對壞輸入行為不同,文件勿混淆
- 驗證方式:用 grpcurl 對 stage 實際打三個 RPC(live-not-mock),涵蓋 空值/非數字/重複值/>500 筆 邊界,確認文件描述與真實回應一致
