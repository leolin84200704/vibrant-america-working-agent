## Approaches Considered

### [2026-05-18 22:11] 三種 race 假說

**假說 A：approve 和 flush 並發**（與 Yekai 3/22 解釋一致）
- approve 流程：`SREM Redis` → `UPDATE DB` (test-approve.service.ts:199-216)
- flush 流程：`SELECT DB (snapshot)` → `SADD Redis` (grpc-result.controller.ts:387-760 `/fix-missing-approval-pending`)
- Race window: T1 approve SREM → T2 flush SELECT 看到舊 DB → T3 approve UPDATE DB → T4 flush SADD 把舊資料塞回去
- 前提：5/16 那次有人手動 call flush endpoint（待向 Zixi 確認頻率）

**假說 B：approve SREM 本身有 transient failure**
- 若 Redis 暫時 unavailable / network blip，SREM 失敗但 DB UPDATE 成功，Redis 永遠殘留
- 沒看到 retry / outbox pattern
- 解釋為何沒有 flush 也會發生

**假說 C：多個 approve 路徑某個沒 SREM**
- 是否還有別的入口（Kafka consumer、Bull queue、其他 gRPC method）會 approve test 但忘記清 Redis？
- 待 grep approve flow 全部入口

## SOP: 處理 "Ghost Pending" Ticket（如 PO-222）

### 一、從 ticket 抓出需要的欄位

最少需要兩個：**barcode** + **test name (assay key)**

範例 PO-222：
- Barcode: `2605156049`
- Test name: `B12`

附加（驗證 / 確認用）：sample ID、collection date、instrument。

### 二、認識三組 Redis Key

| Key | 類型 | 內容 | 由誰維護 |
|---|---|---|---|
| `pending_tests` | Set | 所有 pending tag 字串（master 索引） | results-core: `redis.service.ts:78` SADD / `:118` SREM |
| `pending::{barcode}::{instrument}::{sampleType}::{recvDate}::{sampleId}::{collDate}::{testName}::{instrument}` | Set | 該 tag 下的 `testId`（如 `"69"`） | 同上, `:79` SADD / `:109` SREM |
| `lis::full::pending_tests` | String (JSON array) | Portal 直接讀的前端 cache（含 `Julien_Barcode`/`Test_Names` 等欄位） | producer 不在 results-core，由其他 service 定期 regenerate（約 60s） |

⚠️ **Key 在所有環境都不加 env suffix**（`getEnvKey('pending_tests')` 因 baseKey 不含 `lis::pending` 字串，`replace()` 不生效）。也就是 prod/staging/dev Redis 主機**必須分開**才不會相互污染。

### 三、定位「該清哪台 Redis」

對照 ticket sample 是 prod / staging：
- **Prod**：Azure Cloud Redis（記憶記 `reference_azure_mysql.md` 同套件，但這裡是 Redis；確切 host/port 請查 Consul 或 K8s secret `REDIS_ADDR`/`REDIS_PORT`）
- **本機 dev**：腳本預設 `192.168.60.9:6390`

部署模式判定見 `src/env.utils.ts` → `isDeployedLocal()`（`DEPLOYMENT_ENV === '60.6'`）。

### 四、Inspect（清之前先確認 stale）

```bash
# 1. master set 找 barcode 對應的 tag
redis-cli SMEMBERS pending_tests | grep "::2605156049::"

# 2. 確認 tag 本身的成員（應該是 testId）
redis-cli SMEMBERS "pending::2605156049::Roche::Serum::2026-05-16::2559040::2026-05-15::B12::Roche"
# => "69"

# 3. 反查 frontend cache 是否也有殘留
redis-cli GET "lis::full::pending_tests" | jq '.[] | select(.Julien_Barcode == "2605156049")'
```

### 五、清除（兩種方式，**強烈建議用方式 A**）

#### 方式 A：跑 `remove-from-pending-complete.js` 腳本（推薦）
```bash
cd /Users/hung.l/src/LIS-backend-results-core
REDIS_HOST=<host> REDIS_PORT=<port> \
  node scripts/remove-from-pending-complete.js 2605156049 B12
```
腳本會自動：
1. `SMEMBERS pending_tests` 撈所有 tag，filter `parts[1]===barcode && parts[7]===testName`
2. 對每個 match 的 tag：`SREM tag testId` → `DEL tag` → `SREM pending_tests tag`
3. `GET lis::full::pending_tests` → filter 掉該筆 → `SET` 回去
4. 自動 verify

#### 方式 B：手動 SREM（如 Yekai 凌晨做的）
只在腳本不可用或 ticket 給的 key/member 非常明確時使用。
```bash
redis-cli SREM "pending::2605156049::Roche::Serum::2026-05-16::2559040::2026-05-15::B12::Roche" "69"
redis-cli SREM pending_tests "pending::2605156049::Roche::Serum::2026-05-16::2559040::2026-05-15::B12::Roche"
# 還要記得清 frontend cache，不然 portal 還是看得到
```

#### ❌ 不建議方式 C：`/fix-missing-approval-pending` endpoint
- 這是「**重建** approval pending list」的 admin endpoint（grpc-result.controller.ts:387）
- 它跟 `pending_tests` 是**不同的 Set**（針對 `lis::approve::pending*` 那組）
- 用錯地方反而會把資料 SADD 回去

### 六、驗證

1. 等 ~60s 給 frontend cache regenerator 跑一輪（如果有用方式 A，cache 已被腳本同步清掉，理論上可以馬上看到）

   - 清掉的 tag + testId
   - 驗證結果
