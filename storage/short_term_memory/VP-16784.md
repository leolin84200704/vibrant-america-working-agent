## Ticket Analysis

**Goal（目的）**：把 SHIPPING_RPC gRPC client 從 on-prem (192.168.60.6:31865) 指向 cloud in-cluster gRPC service (lis-shipping-service-grpc.shipping.svc.cluster.local:63142)，作為 parent VP-16685 / epic VP-16776 (Cloud Migration - EMR) 的一片。真正的 code 在 LIS-transformer-v2（不是 emr-v2，emr-v2 對這 RPC 是 no-op），需把既有 cloud-primary + on-prem fallback wiring 從 calendar 模組擴展到 8 個 trans/dashboard shipping caller。

**Classification**：`rpc-cloud-migration` | **Effort**：M | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
改前現狀：LIS-transformer-v2 src/grpc.option.ts:53 microserviceOptions_ship (process.env.SHIPPING_RPC) 被 8 個 service 注入 (dashboard/dashboard.service.ts, trans/utility.service.ts, trans/setting.service.ts, trans/services/{PNS,patientProfile,service,utility.api}.service.ts, trans/merchandise.service.ts)，全部 getService('ShippingService') 共 8 個 call site。VP-16685 已建好 microserviceOptions_ship_cloud (grpc.option.ts:237, process.env.SHIPPING_RPC_CLOUD) + 通用 helper src/calendar/shared/with-cloud-fallback.util.ts (cloud-primary, 只在 transient gRPC error 退 on-prem, kill switch GRPC_CLOUD_FALLBACK_ENABLED=false)，但只接進 12 個 calendar service；trans/dashboard 的 8 個 shipping caller 完全沒接 cloud。yaml/{prod,st}.yml 只有 SHIPPING_RPC env，無 SHIPPING_RPC_CLOUD。cloud endpoint .svc.cluster.local 從 AKS 外 NXDOMAIN（STM VP-16784-87 實證），且 VP-16685 STM 記載 4 RPC cloud endpoint 仍 blocked on Xiaoye Li。改後為何有效：重用 production-proven withCloudFallback —— SHIPPING_RPC_CLOUD 未設或 endpoint 不通時自然走 on-prem fallback，等同現狀零影響，可在 endpoint + EMR-on-AKS 都 ready 前安全 deploy（graceful）。具體改什麼：(1) 把 withCloudFallback util 提到共用位置（如 src/shared/with-cloud-fallback.util.ts）供 calendar+trans 共用，或讓 trans 直接 import 既有 calendar 路徑（先做最小改動可暫 import 現址）；(2) 8 個 shipping caller 各加第二 @Client(microserviceOptions_ship_cloud) + getService('ShippingService')，每個 shipping call 包 withCloudFallback('shipping.<method>', cloudCall, onpremCall, logger)；(3) 同 PR 更新 yaml/prod.yml 與 yaml/st.yml 加 SHIPPING_RPC_CLOUD env entry（config-yaml coupling 鐵律；prod 真值另在 Azure ConfigMap）；(4) 不要硬移除 SHIPPING_RPC（先和 Leo/PM 確認 AC2 接受先保留 on-prem 為 fallback，待 in-cluster 部署+endpoint byte-compare 驗證後另開 cleanup ticket 移除）；(5) 從 stage_test 開 branch feature/leo/VP-16784（注意目前 working tree 在 feature/leo/VP-16513，務必先 checkout stage_test 再開 branch），切 branch 後跑 npx prisma generate 兩份 schema (prisma + prisma2)，npm run start:dev + npm run build 必過。驗證見 risksEdgeCases。

### 候選方案與正反取捨
- **方案 B — 直接硬切：把 8 個 caller 換成 microserviceOptions_ship_cloud 並移除 on-prem SHIPPING_RPC（滿足 AC2 字面）**
  - 取捨：違反 VP-16685 fallback 鐵律；endpoint 未 ready + .svc.cluster.local 從 AKS 外/on-prem 期間 NXDOMAIN → 立即全鏈 shipping 失效、不可逆風險高。否決。
- **方案 C — 把 VP-16784/85/86/87 (SHIPPING/ISSUE/DASHBOARD/AUDIT) 併進同一 branch 一起做**
  - 取捨：caller 大量重疊（utility.service.ts / services/service.ts 同檔同時注入 ship+issue+audit），分開改會反覆動同檔、易 merge conflict；併做更省工。但 4 個是獨立 Story、各有 QA test ticket，併做會模糊驗收邊界，且 audit cloud option 尚未建（grpc.option.ts 只有 ship/issue/dashboard cloud，無 audit cloud）。需先問 PM 是否接受併 PR。

### 風險與邊界案例
- proto byte-equivalence 未驗：VP-16685 STM 只 byte-compare 了 lis package 4 RPC，shipping.proto 兩端未直接驗（reflection 兩端 disabled）。shipping image lis-shipping:f7df5d9c 是 SHA-pinned 但需用 grpcurl -import-path protos -proto shipping.proto 對 cloud+on-prem 跑同一 input byte-compare response（含 timestamp/zero-value 隱形欄位）。
- 驗證只能從 AKS 內 pod 做：.svc.cluster.local cluster-internal DNS 從外部/Mac NXDOMAIN，必須在同 cluster pod (netshoot/grpcurl) 內驗，不能用本機 mock 當『線上驗證』(verified-means-live 鐵律)。
- Endpoint 切換是 peer-observed：實際切到 cloud 流量後要驗 cloud shipping service 端確實收到 request（peer-observed state，不只 my-side log）。
- withCloudFallback 只對 transient code (DEADLINE_EXCEEDED/UNAVAILABLE/UNKNOWN/INTERNAL) 退 on-prem；business error (NOT_FOUND 等) 不退 → 確認 shipping RPC 的錯誤碼語意與此一致，否則 cloud 真錯會被當 business error 直接拋。
- 回歸面廣：8 caller 涵蓋 dashboard / PNS portal / patient profile / merchandise / utility / setting，shipping label/tracking 是 patient-facing；deploy 後（fallback path）需確認行為等同現狀，再切流量。
- audit cloud option 尚未建（sibling VP-16787 缺 microserviceOptions_audit_cloud）—若併 PR 需補建。
