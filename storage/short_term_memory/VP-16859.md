## Ticket Analysis

### [2026-06-02 21:13]
- **Title**: [BE] Create Compliant Deep Link for Result-Ready Emails (Story, status: Dev In Progress, assignee Leo)
- **PRD**: Confluence VW pageId 2467725313「Compliant Deep Link in Provider Result-Ready Emails」
- **背景**: 之前為 HIPAA 把 provider result-ready email 的 PHI 全移除，導致 provider 無法分辨哪封信對應哪個 report，client 抱怨上升。
- **目標**: 「View Report」按鈕點下去直接到對應 report，但 email 內容/subject/URL 都不能含 PHI。
- **解法（PRD §6）**: 把 href 換成 opaque token deep link `https://portal.vibrant-wellness.com/r/{token}`。token 是 CSPRNG 22+ char base62，server 端對應到 report。
- **Compliance（不可談判）**: C1 URL 不含 18 HIPAA identifiers；C2 必須登入才能開（轉寄/截圖無法用）；C3 token lifetime 30–45 天 OK；C4 subject/body 維持 PHI-free。
- **Token storage（PRD §6.3）**: `report_email_tokens` 表 (token, report_id, recipient_user_id, expires_at, created_at)，TTL 45 天，one token→one report，解析時 authed user_id 必須 == recipient_user_id。
- **解析流程（§6.2）**: GET /r/{token} → 已登入則解析+驗證 recipient → redirect report；未登入 → login?next=/r/{token}；非本人 → generic access-denied；過期/無效 → generic error。
- **Non-goals (V1)**: digest、custom label、子帳號、改 email body/subject、SMS/push、subject 加 ref code（defer V2）。
- **Logging（§8）**: 記 token_id/recipient_user_id/outcome，無 PHI。

### 探查發現（兩個 explore agent）
**發信端 (LIS-setting-consumer)**
- `consume_result_ready()` controller line 883 起 provider/customer 分支，確有寄信：組 Postmark `email_message` → `sendNotification` → Kafka topic `Notification-Email-Template`。3 個 TemplateId：prod beta 35058386 / 非prod beta 38859237 / 一般 33002864（controller line 1319-1414）。
- **「View Report」href 寫死在 Postmark template，TemplateModel 沒有 href 欄位** → 要 tokenize 必須(a)在 TemplateModel 加 link 欄位 (b)改 3 個 Postmark template。
- 可用 identifier：accession_id / sample_id / patient_id / customer_id。**沒有結構化 report_id**；`finished_reports` 只是 report name 字串陣列。
- **沒有 customer_id → portal user_id mapping**；`getCustomerEmailBoth` 只回 email 無 user_id。
- setting-consumer 兩個 DB：primary=主 LIS、prisma2=`lis_frontend_service`(trans DB)。無既有 token/deep-link 表。token 產生既有用 uuid v4。

**解析端**
- va-portal 是純 Vue SPA 無後端；report route = `/patient-profile/:patientId/:accessionId/:sampleId`。
- 既有「email token→login」pattern 在 **LIS-transformer**：`invitation_history` 表 (email, clinic_id, customer_id, invitation_email_payload JSON 含 JWT)，URL `portal.vibrant-wellness.com/#/login?token=<JWT>`（trans.service.ts ~18893-18938）。
- 候選解析後端：LIS-backend-results-web（NestJS + JwtAuthGuard 現成）或 LIS-transformer（有 Prisma + portal trans-service API + invitation token 邏輯）。無現成 `/r/{token}` resolver。
- provider portal 身份系統用 `customer_id`（= invitation_history.customer_id）。

## Approaches Considered

### [2026-06-02 21:13]
**核心 scope 問題**: 無法只在 setting-consumer 完成。需要 3 塊：
1. Token 發行 + 寫表（發信當下）
2. Postmark template href 改成 `/r/{token}`（外部 template，需協調誰改）
3. `/r/{token}` 解析 endpoint + auth + redirect（需要有 server，va-portal 沒後端）

**Approach A — 集中在 LIS-transformer**: setting-consumer 寄信前 call transformer API 取 token（或 transformer 統一發信）；token 表 + resolver 都在 transformer（已有 invitation token pattern + portal API + DB）。一致性高。
**Approach B — 分散**: setting-consumer 自己寫 token 表（prisma2 trans DB）+ 產 href；resolver 放 LIS-backend-results-web 讀同表。耦合兩 repo + 跨 DB。
**待確認**: report_id 用什麼（建議存 patientId+accessionId+sampleId 供 redirect）；recipient_user_id == customer_id？；Postmark template 誰改。
