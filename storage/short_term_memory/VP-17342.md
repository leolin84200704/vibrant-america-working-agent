# VP-17342 — [EMR-v2] ResultGenerationProcessor ignores {success:false} (silent result drop)

## Work record

### [2026-07-08 15:20] Fix implemented — PR #242 (base=staging)

Filed as a follow-up from VP-17312 Stage A (a staging schema drift made getIntegrationDataById throw; the result-gen job completed "successfully" with zero deliveries). Leo prioritized it as the pre-condition for widening result traffic to the cloud pod.

**Root cause**: `generateResultHl7` (result-generation.service.ts:171) catches ALL errors and RETURNS `{success:false}` instead of throwing. Its catch only writes an ERROR record when one already exists (`if (existingRecord)`); a failure before `ensureResultTransmissionRecord` (e.g. `prepareResultGenerationData` throws) leaves NO row. `ResultGenerationProcessor.doProcess` (result-generation.processor.ts:92-110) never checked `result.success` → logged success, called `markGenerationSuccess`, returned `{success:true}` → BullMQ marks completed. No retry, no artifact, no alert. `markGenerationFailure`/`markPermanentFailure` use `updateMany` (no-op on 0 rows) so they wouldn't create an artifact either.

**Fix** (feature/leo/VP-17342, from origin/staging, commit fb5e5dc):
1. Processor: `if (!result.success) throw` — routes into existing catch → markGenerationFailure + re-throw → BullMQ retry/backoff; onFailed logs 🚨 FINAL ATTEMPT + markPermanentFailure on exhaustion.
2. Service catch: added `else` (no existingRecord) branch emitting a distinct greppable `[RESULT_SILENT_DROP_GUARD] ... sample_id=... integration=... error=...` structured error log so pre-record failures are observable.
3. New unit test `result-generation.processor.spec.ts`: success:false → throws + markGenerationFailure called + markGenerationSuccess NOT called; success:true → marks success. 2/2 pass. `nest build` clean. No CJK.

**PR**: https://github.com/Vibrant-America/lis-backend-emr-v2/pull/242 (base staging). Agent does not merge — Leo decides.

**Not in scope** (deliberately minimal): did not restructure the service to create a skeleton record before prepare (bigger lifecycle change, would interact with ensureResultTransmissionRecord find-or-create). The structured log + BullMQ retry converts the silent drop into a loud, retried, logged failure — satisfies the ticket AC. Terminal-vs-retriable classification not added (all failures retry 5x then permanent-fail, matching existing thrown-error behavior).
