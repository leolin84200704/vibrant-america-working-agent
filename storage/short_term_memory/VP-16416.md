**問題本質**:
- Clinical Team (practice_id=150105) 要求所有 provider-scheduled appointments 預設使用 "Clinical Consult"
- 原預設為 "General"；該 type 被刪除後 fallback 到 "Follow-up"（非預期）
- "Clinical Consult" type 已由 clinical team 在 setting 中建立並承諾保留

**AC**:
- AC1: 新 provider-scheduled appointments 預設 "Clinical Consult"
- AC2: 不影響其他 practice
- AC3: 不更動歷史 events
- AC4: 若 "Clinical Consult" 不可用，graceful 處理（**no silent fallback**）

**Repo**: LIS-transformer-v2

**核心程式碼路徑（已調查）**:

## Approaches Considered
### [2026-05-04] 草擬

**A. Practice-150105-aware default in `getDefaultPracticeEventType()` + AC4 throw**
- 在 service 內：if `practiceId === CLINICIAN_PRACTICE_ID (150105)` → 用 name="Clinical Consult" 查詢；否則維持原 "General" 邏輯
- AC4: clinical team practice 找不到 "Clinical Consult" 且非 cancelled → **throw 明確 error**（不 fallback 到 first-active）。其他 practice 仍可 fallback 維持向後相容
- 優點: 最小改動；AC2 自動滿足；複用既有 CLINICIAN_PRACTICE_ID 常數；fallback 行為解耦（其他 practice 邏輯不變）
- 風險: hardcoding practice ID 不優雅，但已是 codebase 既有 pattern

**B. 通用 per-practice "default type" config（DB schema 改動）**
- 加 `v2_practice_event_type.is_default` BOOLEAN
- prisma migration + 資料 seed：practice 150105 設 Clinical Consult，其他 practice 設 General
- `getDefaultPracticeEventType()` 改為先查 `is_default=true`
- 優點: 通用、乾淨、未來可由 admin UI 切換 default
- 缺點: over-engineering for 1 practice；需 migration、seed、跨 service 影響面大；ticket 沒要求 admin 介面

**C. 環境變數 / config map**
- e.g. `CLINIC_DEFAULT_EVENT_TYPE_BY_PRACTICE = { 150105: 'Clinical Consult' }`
- 優點: 比 hardcode 略好擴展
- 缺點: 仍 special-case；config 維護成本

**初判**: A 最符合 ticket 範圍（針對 clinical team only），AC4 graceful 用 throw 處理。寧可先做最小改動。

## Lessons Learned
### [2026-05-04]

1. **AC 字面「no silent fallback」≠ 一定要 throw**：PM 寫 AC 時通常希望「不要 silent」即可，warn log + 沿用既有 fallback path 也算 graceful。先給 default 解讀讓 PM 推翻，比預設 strict 解讀少一輪 review。

2. **Step 4 呈報格式**：列細節問題時同步給出我的 default 偏好，例如「以下細節我採 default A/B/C，若 Leo 不同意可調整」— 比純列選項更高效。

3. **getDefaultPracticeEventType 在 transformer-v2 有兩份重複實作**：`event.service.ts:2950` 與 `meeting-request.service.ts:905`。改 default 邏輯時要同步檢查兩處，否則 meeting-request flow 會與 event flow 行為不一致。

4. **special-case-by-practice-ID 是 codebase pattern**：`this.CLINICIAN_PRACTICE_ID = 150105` 在 EventService 內已是 readonly field，散落多處（reminder/calendar/email-template/isClinicianEvent/createEventByPatient 寫死等）。對單一 practice 的 customize 直接擴展此 pattern 即可，無需立刻通用化。
