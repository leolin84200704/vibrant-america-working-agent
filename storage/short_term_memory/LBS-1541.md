## Code path (exploration)

**Injection point**: `src/modules/hl7-order-processing/services/parser.service.ts:assembleOrderFrontend()` after BestDeal expand（line ~440），在 `getLabProcessingFee` 之前 push 進 `orderItems`。

**Package ID lookup**: `OrderMappingCacheService.testIdToPackagePriceMap` / `groupIdToPackagePriceMap` / `emrCodeToPackagePriceMap`（key 是 `uniqueemrcode` lowercased = "wheat zoomer" 等）。

**新 item shape**：
```typescript

  item_id: totalIgTestId,
  item_type: ItemType.PACKAGE_PRICE_ID,
  original_price: 0,
  total: 0,
}
```

**現有 code 無 add_on / free-test / complimentary 邏輯** — 完全新功能。

## Approaches Considered

### Approach A: Hardcoded ID list（lookup at module init）
- 啟動時用 `uniqueemrcode` 字串 lookup 21 個 package ID + Total Ig ID
- Cache 進 `OrderMappingCacheService` 新 fields: `addOnTriggerPackageIds: Set<number>`、`totalIgPackagePriceId: number | null`
- 在 parser.ts 注入點 check `bestDealTests` ∩ `addOnTriggerPackageIds` 是否非空 → 加 Total Ig $0 item
- 優點：純 code，無 DB schema 變動；運維簡單
- 缺點：21 個 trigger package 是 hardcode（將來想加減要改 code + deploy）

### Approach B: Config-driven via DB / env
- 新 DB table `free_addon_rules`（trigger_package_id, addon_test_id, customer_id_or_null）或 JSON env var
- 啟動 cache 從 DB load
- 優點：PM/lab 可自助加減無需 code change
- 缺點：需新 schema，scope creep；first version 不必要

### Approach C: Push 給 upstream `BestDeal` API
- 跟 PM 提議：upstream order API 自己根據 trigger packages add `Total Ig` add-on
- 不在 EMR repo 寫死，跟 VA portal 一樣 server-side rule（如果 VA 是這樣做的話）
- 優點：DRY — VA + EMR 共用 upstream logic
- 缺點：跨 team coordination；不清楚 upstream API 是否已支援

### 建議方案（改前 / 改後為何有效 / 改什麼）
改前現狀：進單組裝在 lis-backend-emr-v2 的 src/modules/hl7-order-processing/services/parser.service.ts:assembleOrderFrontend()（line 310 起，1:1 port 自 EMR-Backend ParseHL7.java）。orderItems 由 BestDeal 輸出（left_over_test_id_list line 371-392 + suggest_bundle_list line 394-437）組成。Package 身分查詢走 OrderMappingCacheService（order-mapping-cache.service.ts:29-33,116），含 testIdToPackagePriceMap / groupIdToPackagePriceMap / emrCodeToPackagePriceMap（key = pp.uniqueemrcode.toLowerCase()）。OrderItem（order.dto.ts:37,56）支援 original_price/total，可帶 0。無任何 add-on 邏輯。

重要更正（對 engineer 分析）：engineer/STM 假設可用 emrCodeToPackagePriceMap 以 "wheat zoomer" 等名稱反查 ID——這是錯的。實際 uniqueemrcode 值是 "varequistionNNN"（test-panel-mapping.service.ts:357 VAREQUISTION${orderTypeId}、spec 中 varequistion200/332），非顯示名稱。20 個 trigger 必須由 PM 給明確 package/test ID，或對 name 欄位 match，不能靠 uniqueemrcode by-name 自行推導。此點加強 Dev Blocked 的必要性。

### 風險與邊界案例
- lab fee 互動：getLabProcessingFee（line 453）是遠端 HTTP call，會把整個 orderItems（含 $0 add-on item）送上游，lab fee 由 Order API server-side 算。$0 item 是否被誤算進 lab fee 無法本地確認——需測或問 PM VA 如何提交 add-on（pre/post fee-calc），必要時把 add-on 排除在 getLabProcessingFee payload 之外

- 若 trigger package 進來時走的是 left_over vs suggest_bundle 不同分支，注入點需放在兩分支都跑完之後（line 437 後）才能涵蓋所有 path——已選正確位置

- result HL7 回傳：Total Ig 結果是否要回送 EMR 或只做 lab 不回傳，影響 result 模組是否需配套改動（test-panel-mapping）
