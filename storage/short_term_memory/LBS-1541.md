## Ticket Analysis

**Title**: EMR add free add on (Total Immunoglobulins)
**Reporter**: Zhenhe Zhang（內部，非 Zendesk app）
**Priority**: P2 Medium, no due date
**Project**: LIS Backend Service (LBS, Service Desk)

**Requirement (per description)**：當 EMR-originated order 含以下 21 個 package 任一個時，**自動加 "Total Immunoglobulins" 作為 $0 free add-on**，"like VA"（如 VA consumer portal 已實作的邏輯）：

```
Wheat Zoomer, Tickborne Diseases 2.0/1.0, Food Sensitivity Complete IgA+IgG,
Lectin Zoomer, Corn Zoomer, Dairy Zoomer, Peanut Zoomer, Egg Zoomer, Soy Zoomer,
Food Sensitivity Profile 1 IgA+IgG, Lyme+TBRF, Coinfections 1/2,
Opportunistic Infections, Neural Zoomer Plus, Viral Infections,
Candida+IBS Profile, Autoimmune Zoomer, Food Bundle, Food Zoomer
```

## Code path (exploration)

**Injection point**: `src/modules/hl7-order-processing/services/parser.service.ts:assembleOrderFrontend()` after BestDeal expand（line ~440），在 `getLabProcessingFee` 之前 push 進 `orderItems`。

**Package ID lookup**: `OrderMappingCacheService.testIdToPackagePriceMap` / `groupIdToPackagePriceMap` / `emrCodeToPackagePriceMap`（key 是 `uniqueemrcode` lowercased = "wheat zoomer" 等）。

**新 item shape**：
```typescript
{
  item_id: totalIgTestId,
  item_type: ItemType.PACKAGE_PRICE_ID,
  original_price: 0,
  total: 0,
}
```

**現有 code 無 add_on / free-test / complimentary 邏輯** — 完全新功能。

## Approaches Considered

### Approach A: Hardcoded ID list（lookup at module init）
- 啟動時用 `uniqueemrcode` 字串 lookup 21 個 package ID + Total Ig ID
- Cache 進 `OrderMappingCacheService` 新 fields: `addOnTriggerPackageIds: Set<number>`、`totalIgPackagePriceId: number | null`
- 在 parser.ts 注入點 check `bestDealTests` ∩ `addOnTriggerPackageIds` 是否非空 → 加 Total Ig $0 item
- 優點：純 code，無 DB schema 變動；運維簡單
- 缺點：21 個 trigger package 是 hardcode（將來想加減要改 code + deploy）

### Approach B: Config-driven via DB / env
- 新 DB table `free_addon_rules`（trigger_package_id, addon_test_id, customer_id_or_null）或 JSON env var
- 啟動 cache 從 DB load
- 優點：PM/lab 可自助加減無需 code change
- 缺點：需新 schema，scope creep；first version 不必要

### Approach C: Push 給 upstream `BestDeal` API
- 跟 PM 提議：upstream order API 自己根據 trigger packages add `Total Ig` add-on
- 不在 EMR repo 寫死，跟 VA portal 一樣 server-side rule（如果 VA 是這樣做的話）
- 優點：DRY — VA + EMR 共用 upstream logic
- 缺點：跨 team coordination；不清楚 upstream API 是否已支援

7. **HL7 報告層面**：Total Ig 加進去後，發回 EMR 的 result HL7 是不是要把 Total Ig 結果也帶回？還是只是 lab process 多做這項、不回傳給 EMR？
