### 建議方案（改前 / 改後為何有效 / 改什麼）
重要修正：本工作在 LIS-transformer-v2（不是 grounding cwd 的 lis-backend-emr-v2；後者的 src/modules/customer-portal/dashboard 是無關的另一個模組，endpoint 完全不同）。已逐項核對工程師的 code 描述，皆與 LIS-transformer-v2 src/dashboard/ 相符。

目的：補齊 VP-13758「Test」缺口、完成 QH-918 驗收。
改前現狀：模組已 refactor 完成且 merge——dashboard.controller.ts(464行)/dashboard.service.ts(718行)/dashboard.dto.ts(344行)/dashboard.module.ts(37行)，含 JSDoc + 5 個 @ApiOperation。Controller-level @UseGuards(UnifiedAuthGuard)。共 5 個 endpoint（工程師誤寫成「4 個」）：GET /user/timeline、PUT /user/event/flag、PUT /user/event/unflag、PUT /user/events/archive、PUT /user/events/unarchive。Service 靠 client5.getService('DashBoardService').userTimelineFetch + Redis(getRedisClient) + UtilityAPIService（from ../trans/services/utility.api.service）。全 repo 無任何 dashboard*.spec.ts。
改後為何有效：以 unit test 固化現有行為，不依賴任何 blocked sibling（thinnest independent slice，符合 Leo 偏好）；mock gRPC/Redis 可離線跑、CI 友善、純新增不動 prod 邏輯。
具體改什麼（純新增）：1) branch feature/leo/QH-918，PR base stage_test（非 staging/main，遵 no-direct-push iron rule）。2) 切 branch 後先對兩個 prisma schema 跑 npx prisma generate，確保 start:dev 與 test 可過（iron rule）。3) 抄 src/mypractice/clinic-charting.controller.spec.ts 與 *.service.spec.ts 的 Test.createTestingModule + provider mock 範式，新增 src/dashboard/dashboard.service.spec.ts、dashboard.controller.spec.ts。4) mock client5.DashBoardService（userTimelineFetch）、getRedisClient、UtilityAPIService —— mock 的是 service 介面而非 endpoint，故 VP-16786 DASHBOARD_RPC 遷雲後測試不需改（勿 hardcode 192.168.60.6:31987 之類 endpoint）。5) 覆蓋 5 個 endpoint 的 happy path + 邊界：timeline 分頁/filter、flag/unflag、archive/unarchive 空陣列與權限/role 分支。6) 不新增 process.env.X，故不涉及 lis-emr-v2-config*.yaml 同步。7) start:dev + npm test 必須過才交付。

注意：unit test（mock）不能宣稱為「線上驗證」(verified-means-live iron rule)；真實 staging 行為驗證需走 e2e，但 staging getTimeline 正卡 VP-16759（且該 bug 本身目前是 Dev Blocked，等 Kristine 提供 API/payload/token 才能 repro），故 e2e 應另開 ticket 或待 VP-16759 解除後再補，不要 gate 住 QH-918。

### 候選方案與正反取捨
- **方案 A：service + controller unit test（Jest，mock gRPC/Redis/UtilityAPIService）**
  - 取捨：直接命中 Test 缺口、有 mypractice 範式可抄、不依賴 blocked sibling、純新增低風險；但 mock 樣板量大、驗不到上游/guard 真實行為（如 VP-16759 的 403）
- **方案 B：打真實 staging endpoint 做 e2e / 手動驗收**
  - 取捨：能抓 A 抓不到的 auth/上游真實行為、對 QA 最有說服力；但依賴 staging 健康，getTimeline 正卡 VP-16759 403 且該 bug 自身 Dev Blocked，環境不穩會 flaky，需先排障
- **方案 C（建議）：先交付 A 的 unit test，待 VP-16759 解後另補一條 getTimeline happy-path e2e**
  - 取捨：立即可獨立交付、固化行為；e2e 留待依賴解除，避免被 blocked sibling 拖住
