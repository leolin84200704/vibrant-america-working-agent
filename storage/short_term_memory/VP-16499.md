## Approaches Considered

### [2026-05-22] Code 現狀 (Explore agent 報告)
- **無**現成 reschedule availability query — 目前 reschedule 走 `updateMeetingRequestForMe()` mutation，client 自己挑時段
- New Appointment 用 `ProviderAvailabilityService.getProviderAvailability(provider_id)` (meeting-request/provider-availability.service.ts:121) — single provider only
- Clinical Educators 在 `v2_calendar` 表 (`practice_id=150105`, `role!='patient'`, `is_active=true`) — `clinic.service.ts:115 getDetailedCustomersForClinic()`
- OPD timing 已實作 (v2_schedule + v2_schedule_exception)
- **Specialties** 不存在 codebase — 是 gap，需新 schema 或 gRPC
- "1st available" logic 不存在
- 無 cache 層；live Prisma query → cancellation 即時生效

### [2026-05-22] 推薦方案：minimal wrapper + 分階段
1. 新增 GraphQL Query `getRescheduleAvailability(input)` in `meeting-request.resolver.ts`
2. Input: `event_id` (UUID), `scope` (default 'single'), `date_range_start/end`
3. Service:
   - `'single'`: 用 event.provider_calendar_id 跑既有 `getProviderAvailability`
   - `'all-clinicians'`: `Promise.all` over `getDetailedCustomersForClinic(150105)` × `getProviderAvailability`
4. `getProviderAvailability` 需擴充加 `excludeEventId` 參數（避免重排當前 event 時看不到自己原時段）
5. "1st available" = `slots[0]`（既有 ascending sort）
6. Specialties 暫回空陣列 + API doc 標 pending；或先確認 FE 是否真的會用

### [2026-05-22] 正反辯論結論
**正方** (PRO)：cost/benefit 最佳；重用既有 service 安全；7-10 clinician × Promise.all OK；backward compat 由 default scope 保證
**反方** (CON) 提出致命 issues：
- 🔴 `excludeEventId` 必須加 — 否則重排當前 event 時，X 自己佔的 slot 被誤判 unavailable
- 🟡 API contract 必須先跟 Jiaming 對齊 specialties / "1st available" 細節
- 🟡 hardcode 150105 vs 接受 practice_id 參數
- 🟢 N+1 對 7-10 clinician 可接受；若擴張到 30+ 需重寫接受 clinicianIds[] 一次 batch

## Lessons Learned

### [2026-05-26]
1. **Calendar DB 多 schema**：同一 host `lis-postgresql.postgres.database.azure.com:5432`，prod=`calendar_prod`、staging=`calendar_dev_new`，table 結構相同。任何 schema 變更兩邊都要做。
2. **Idempotent migration `current_schema()` 陷阱**：`IF NOT EXISTS (SELECT FROM information_schema.columns WHERE table_name=...)` 不加 `table_schema = current_schema()` 會跨 schema 誤判 → 第二個 schema 永遠 skip。
3. **psql 沒裝**：用 `prisma db execute` 或 `new PrismaClient({ datasourceUrl })` + `$executeRawUnsafe` 跑 raw SQL；後者要自己 split statements、保留 `$$` DO block。
4. **`npm run lint` 帶 `--fix`** — 會 auto-reformat 整個 repo。跑之前要警覺，否則污染 branch 100+ unrelated files。改用 `npx eslint <files>` 只檢查自己的 file。
5. **legacy portal-calendar specialty**：source of truth 是 Java service `/clinicians/first-available`，Bearer JWT auth。practice 150105 的 9 個 clinician 每人 ~48-50 個 specialty，且目前**全部 universally held**（filter 機制 work 但 prod data 無區分效果）。
6. **availability query 是 UX provider 不是 conflict guard** — server-side `validateSlotAvailability` 已 hard-block double-booking；availability query 額外存在是為了讓 FE 主動列可選時段。
