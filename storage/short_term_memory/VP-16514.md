## Ticket Analysis

### [2026-06-04]
- **Title**: [BE] Meeting Type Status Toggle & Reorder Endpoints (Story, P2, Dev To Do, assignee Leo)
- **Epic**: VP-15817 EHR Calendar - Multi-Type Availability. **Repo: LIS-transformer-v2 `src/calendar/`**（calendar postgres, DATABASE_URL_CALENDAR, Prisma）。
- **Tech (ticket)**: `PATCH /api/meeting-types/{id}/status`(is_active bool) / `PATCH /api/meeting-types/reorder`(ordered id 陣列→display_order) / inactive 排除於 public booking。
- **AC**: AC1 toggle active/inactive 不刪除；AC2 inactive 不出現在 public 查詢；AC3 reorder 持久化 display_order，public 依 display_order, name 排序；AC4 只有 owning provider 能 toggle/reorder。
- Links: QH-5236 (QA), PRD VW/2205483009, Figma。

### 探查發現（LIS-transformer-v2/src/calendar）
- **Model `v2_practice_event_type`**（prisma/schema.prisma:507-524）= meeting type。欄位：id, practice_id, name(VarChar30), color_hex, is_canceled(soft delete), created_by, updated_by, timestamps。`@@unique([practice_id,name])`。**沒有 is_active、沒有 display_order** → 要加欄位 + migration。owner 是 **practice 層級**（無 per-provider 欄位，只有 practice_id + created_by）。
- **Public 查詢**：`public-booking.service.ts:1020-1048 getAppointmentTypes()` → where `{practice_id?, is_canceled:false}`, orderBy `name asc, id asc`。route `GET /api/booking/appointment-types`（public, 無 auth, 30/min）。要改成加 `is_active:true` + orderBy `display_order asc, name asc`。
- **無既有 meeting-types admin controller** → 要新建 authenticated controller（toggle + reorder）。
- **Auth**：`src/calendar/guard/auth.guard.ts` AuthGuard + @CurrentUser；ClinicUserPayload(user_id, customer_id, clinic_id, user_roles…)。ownership check 無現成 helper。
- **v2_calendar**（schema:306+）：`@@unique([practice_id, calendar_owner_id])`。
- **Migration**：Prisma migrate + 手動 SQL 慣例 `prisma/manual-migrations/YYYYMMDD_desc.sql`。

## Approaches Considered

### [2026-06-04] 待 Step 4 跟 Leo 確認的點
- **欄位**：加 `is_active Boolean @default(true)` + `display_order Int @default(?)` 到 v2_practice_event_type，prisma migration + backfill 既有 row 的 display_order（依 name/id 排序）。
- **ownership (AC4)**：meeting type 是 practice 層級（無 per-provider）→「owning provider」應解讀為「該 provider 屬於這個 practice」= 驗 `event_type.practice_id == 登入者的 practice_id`。**待確認 JWT(clinic_id/customer_id) 如何對應 practice_id**。

- **reorder**：id 陣列 scope 到登入者 practice，display_order=陣列 index，驗所有 id 屬該 practice 否則拒。

### 建議方案（改前 / 改後為何有效 / 改什麼）
採方案 A（最小可獨立交付片）+ 強制 PM 確認 public 歸屬，工程師分析整體正確、code 已驗證屬實，但有一處關鍵修正。

改前現狀：toggleStatus() / reorder() 已在 LIS-transformer-v2 commit 8220f9e（branch feature/leo/VP-16513，與 VP-16513 同 PR）。schema v2_meeting_type 在 prisma/schema.prisma:530（display_order:543、is_active、@@index([provider_id,is_active,is_deleted]):558，VP-16512 建）。controller (src/calendar/models/meeting-type/meeting-type.controller.ts:62/71) 掛 PATCH api/meeting-types/reorder + PATCH api/meeting-types/:id/status，皆 @UseGuards(AuthGuard)，路徑完全符合 ticket spec。授權走 assertProviderAccess（owning provider customer_id===provider_id 或同 practice clinicadmin，service:70）。reorder 用 $transaction 把 display_order=陣列 index，並驗所有 id 同一 provider + 非 deleted（service:259-287）。DTO validation 完備（ReorderMeetingTypesDto: IsArray/ArrayNotEmpty/IsInt each；ToggleStatusDto: IsBoolean）。public 缺口確認：getAppointmentTypes()（public-booking.service.ts:1020）查舊表 v2_practice_event_type、where {is_canceled:false}、orderBy [name,id]；全 public_booking 目錄 grep 不到 v2_meeting_type。故 AC1/AC4 provider 端成立，AC2/AC3 public 端未生效。

關鍵修正（推翻工程師假設）：工程師說「AC2/AC3 public 缺口由 VP-16516 承接」——此為推測，不成立。我已抓 VP-16516 全文：它是 Dev Blocked，描述純粹是 per-meeting-type weekly hours / date overrides / "apply default availability"，ACs 完全沒提 public booking query / inactive 排除 / display_order 排序，也沒提把 public booking 切到 v2_meeting_type。所以 public 反映目前是 orphan，無任何 ticket 明確承接。這正是必須跟 PM 敲定的事，不能擅自塞給 VP-16516。

具體要改什麼（本張幾乎無新 code）：(1) 補 unit test——src/calendar/models/meeting-type 下無任何 *.spec.ts（已確認），需 cover toggleStatus（is_active 寫入、權限拒絕）+ reorder（display_order=index、跨 provider 拒絕、找不到/deleted id 拒絕），prefer live-not-mock 對真實 dist/test DB 驗 row。(2) 跟 PM 確認 AC2/AC3 public 反映的歸屬與時程（見 clarifyingQuestions），確認後在 ticket/PR #485 留言並附 public-booking.service.ts:1020 行號。(3) merge 前在 feature branch 跑 npx prisma generate（兩 schema：prisma/schema.prisma + prisma2/schema2.prisma）、start:dev 必過、跑新 unit test，再 PR 進 stage_test（不直接 push）。

### 風險與邊界案例
- AC2/AC3 public 反映目前無 owner ticket（VP-16516 ACs 不含 public booking，已抓全文確認）→ 若 QH-5236（is tested by）照 AC 字面驗，會判 fail；不先敲定歸屬會變孤兒需求。
- 部分排序（partial reorder）：reorder 只送 provider 5 個 type 中的 2 個時，未送的 3 個保留舊 display_order，可能與新 index 0/1 碰撞造成排序歧義。需確認前端是否總是送完整清單，或 service 是否該拒絕部分清單。
- display_order 在跨 type 衝突時 orderBy [display_order,name] 的 tie-break 行為（同序時退回 name），QA 需驗排序穩定且符合設計預期。
- AC4 授權為放寬版：同 practice clinicadmin 也能 toggle/reorder（service:70），但 ticket 字面寫 'Only owning provider'。需 PM 確認接受放寬，否則 QA 會判 AC4 不符。
- reorder 用 $transaction array of updates——若某 id 在交易中被併發 soft-delete，update where{id} 仍會更新到 deleted row（findMany 過濾 is_deleted 與 update 之間有 TOCTOU 窗口）。低風險但 QA 可注意。
- toggleStatus 停用 type 後，已存在的未來預約（appointment）是否仍有效？ticket 說 'without deleting' 歷史預約，需驗停用不影響既有 booking、只影響新 public 查詢。
- includeInactive query param 是 string（controller:43），需驗 'false'/'0'/'' 等字串被正確解讀，避免 truthy string bug。
