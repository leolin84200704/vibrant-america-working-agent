## Decisions Made

### [2026-06-22] Shortcut ordering 設計定案（與 Leo 對話確立）
**情境**: FTP order 的 OBR-4.1 送 shortcut **顯示名稱**（如 `Food Sensitivity Test`），無 VACP/VATEST/VAREQUISTION/discountpanel/數字 code（真實樣本確認）。目前 classifyBatteryId 最後 else `parseInt`=NaN → ERROR(emr_code_not_found)。要新增 shortcut 解析分支。

**定案流程**:
1. 在 classifyBatteryId 既有 4 種 prefix + 數字都不符後、回 ERROR 之前，新增 shortcut 分支。
2. clinic/customer **不需重查** — parser 上游已用 ORC-12.1 NPI 走 gRPC getCustomerByNPINumber + ehr_integrations winner 解析出唯一 customer(customer_id+clinic_id)，已傳進 classifyBatteryId 的 `customer` 參數。
3. call Get Shortcuts API：`GET /v1/pricing/item/catalog/shortcut?clinic_id=&customer_id=&account_type=clinic`，用 (customer_id, clinic_id) **成對**。同 NPI 跨多 clinic 時 iterate 全部既有 ehr_integrations row（**winner clinic 排第一**以保持與整單 customer 歸屬一致），找到 name match 就停。
4. 名稱比對需正規化（trim + case-insensitive；API 回 `(MALE)` vs sheet `(Male)`、有 `&`）；對不到 → 落回 emr_code_not_found（不可默默下錯）。
5. **tests + bundles 都要展開**（Leo 確認）：
   - `tests[].test_id` → testIdList（testIdToPackagePriceMap 定價）
   - `bundles[].bundle_type='discountpanel'` → discountPanels bucket（mapDiscountPanelId + officialBundleIdToBundleMap）
   - `bundle_type='custompanel'` → customBundles bucket（customOldOrderTypeId...lookupCustomBundle，VACP 同路）
   - `bundle_type='xchoosey'` → XChooseYList（meta_data.selected_test_ids + supplement_test_ids）— 最複雜，實作時需驗證對應
   - 只取 test_id 會漏單：實測「Food Sens & Wheat」tests=0/bundles=1 → 空單；「Total Wellness M/F」也含 bundle
6. `is_dbs` 要帶進 order（影響 kit/sample；9 Shortcuts 表頭註明 blood collection no DBS）

**已驗證關鍵事實**:
- customer_id **會**過濾結果（2930+43262→10、+999999→36、150053+43262→只1筆不完整、不帶 customer_id→`invalid customer_id`）→ 必須用真實成對 (customer_id, clinic_id)。
- is_practice=true 的 shortcut 由 clinic_id 驅動；shortcut_id 與 xlsx 完全吻合。
- OAuth: client_credentials @ `/v1/oauth2/token`（emr-v2 OAUTH2_CLIENT_ID/SECRET）→ JWT 1h；靜態 VIBRANT_API_TOKEN 對 shortcut API 回 401。

## Failures

### [2026-06-22] 重大查詢 bug — Prisma $queryRaw IN() 用 join 字串
- 錯誤寫法 `WHERE clinic_id IN (${CLINICS.join(',') as any})` → Prisma 把整串當**單一 bound param** → SQL 變 `clinic_id IN (?)` param='2930,8003,...' → MySQL 字串轉 int 只取開頭 → **只比對到 2930**。
- 後果: 兩支 check script (_vp17076-check.ts / _vp17076-exist.ts) 全程只看到 2930，誤判「19 clinic 都不存在 / 需新建」。Leo 自己跑 SELECT * 抓到一堆既有 row 才發現。
- Root cause: 沿用 scripts/check-vp16329.ts 的 hardcode 單值模式，改成 array 時沒用 `Prisma.join()`。
- 正解: `import { Prisma }` + `IN (${Prisma.join(CLINICS)})`，或對信任的整數陣列直接字串內插建 SQL。
- 教訓: 多值 IN 查詢務必**先驗證回傳筆數合理**（20 clinic 只回 1 筆就該起疑），不能直接拿來下「不存在」結論。對應 [[feedback_batch_db_verify]] / [[feedback_join_scope_reverse_audit]]。
