# VP-16166 - [BE] Quarantine Data Model & Service

## Ticket Analysis

**Goal（目的）**：當整合中介層收到無法辨識 provider 的 HL7 order 時，不再只 log 丟棄，而是把整封原始 HL7 完整寫進新的 quarantined_orders 表（永不遺失訊息），由 UniMod admin 審核 resolve（link 既有 provider / 建新 provider / reject）。配合 7 天自動過期（含 48hr 警示信）與 dead-letter dual-write 做災難復原。本票是 PRD EHR Integration Enhancements V2 (pageId 2316173315) Feature 1 的 quarantine 後半段。

**Classification**：`practice-integration-epic` | **Effort**：M | **Confidence**：4/5 | **Ticket fetched**：是
**Affected repos**：lis-backend-emr-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
採「最小可獨立交付片（方案 A）」，但須先等 Leo/Kristine 確認 cutover 時程才動工（見 blockerReason）。

改前現狀：src/modules/hl7-order-processing/processors/hl7-order.processor.ts 在 NPI/MSH.4 都 resolve 不到時只 logger.log 「falling through to parser (Java parity)」(line 76-87)，order 不會被 quarantine。resolveIntegration() (line 281-312) 只實作 Step1(customer_npi) + Step2(clinic_id=MSH.4 + customer_id=CLINIC_LEVEL_MARKER)；quarantined_orders 表不存在。intake 走 BullMQ + Redis（queue process-hl7-file），非 Kafka。

改後為何有效：quarantine 寫入用 fire-and-forget（try/catch 吞例外 + warn log，比照同檔 archiveLocalFile line 234-237 的 non-fatal pattern），不進 BullMQ worker 成敗判定 → 天然滿足 AC2 non-blocking，且絕不破壞 VP-16463 確立的 Java-parity fall-through（NPI miss 不可 fail out）。資料表是純新增的影子表、無 read path，與 VP-16164 dual-schema 策略一致，可先於 read-path 切換獨立交付。

具體改什麼：
1) prisma/schema.prisma 新增 quarantined_orders model：raw_hl7_message TEXT(完整)、sending_facility(MSH.4)、provider_npi、provider_name、vendor_internal_id、patient_name、patient_dob、order_id、matched_practice_id(nullable)、provisional_provider_id(nullable)、quarantine_reason enum(unknown_provider|unknown_origin)、status enum(open|in_review|resolved|expired)、resolution_action enum(link_existing_provider|created_new_provider|rejected, nullable)、resolved_by、resolved_at、internal_notes、created_at、updated_at。**修正：用真正 Prisma enum**（見 risksEdgeCases，schema 現有 35 個 enum，VP-16629 起的 EHR-V2 新表如 EhrIntegrationStatus 都用真 enum；engineer 與 VP-16164 STM 的「String+JSDoc 慣例」已過時，仍須 Leo 拍板）。
2) prisma/migrations/<date>_quarantine/migration.sql：raw-SQL CREATE TABLE（沿用 VP-16164/VP-16629 manual migration pattern）。
3) 新 QuarantineService.write()（async）+ QuarantineExpiryService（@Cron，沿用 scheduled-reports 範例：open>7d→expired，剩 48hr 寄警示信，email 走 VP-16169 管道）。
4) hl7-order.processor.ts line 76-82 else 分支新增 `void this.quarantineService.write(...).catch(err=>logger.warn)`；reason = MSH.4 命中 practice 但無 provider→unknown_provider，否則 unknown_origin。**不改** fall-through 本身。
5) AC6 dead-letter dual-write 與 resolution_logs 表待 PM 確認落地方式（見 clarifyingQuestions）；建議切成 sub-task 不擋本片交付。

### 候選方案與正反取捨
- **方案 B：一次做完全部 AC（含 dead-letter queue + auto-reprocess hook + 接 read path 到 practice_integrations）**
  - 取捨：scope 過大且卡 dependency：dead-letter 需新 BullMQ queue/落地策略、auto-reprocess 是 PRD 未定 Open Q1、read-path 依賴 VP-16164 切換 + VP-16168，無法獨立交付，違反 Leo 偏好的最小切片。
- **Prisma enum vs String+JSDoc：沿用 VP-16164 STM 記的 String 慣例**
  - 取捨：經查 schema 實際有 35 個 enum，VP-16629 之後的 EHR-V2 新表（EhrIntegrationStatus/Type/Origin/ConnectionType）都用真 Prisma enum。沿用 String 會與同 epic 新表不一致、失去型別保護；但 enum 改值需 migration、彈性較低。建議用真 enum 並請 Leo 確認。

### 風險與邊界案例
- AC2 non-blocking 的真正風險：fire-and-forget 必須確保 quarantineService.write 內部任何 throw（DB 連線失敗、TEXT 過長）都被 .catch 吞掉，否則 unhandled rejection 或誤 propagate 回 worker 會重蹈 VP-16463 的 fail-out。驗證要實際模擬 DB 寫入失敗，確認 worker 仍把 order fall-through 成功。
- AC3 raw HL7 完整儲存：raw_hl7_message 須 TEXT/LONGTEXT，不可截斷；要測超大 HL7（多 OBR/OBX）不被 MySQL 預設欄位長度截掉，且驗證 round-trip 內容逐 byte 相同（PHI 完整性）。
- Prisma enum 慣例不一致（已查證）：engineer 與 VP-16164 STM 都聲稱 schema 無 prisma enum，實際有 35 個且 EHR-V2 新表用真 enum — 此為對 engineer 分析的明確修正。
- queue 技術：AC6 dead-letter 的 intake 是 BullMQ+Redis（process-hl7-file queue）非 Kafka；DLQ 若用 BullMQ failed-job 機制 vs 另建表，行為差異大，不可假設。
- 重複 quarantine：同一封 HL7 重送（remote delete 失敗或 re-enqueue）可能重複寫入 → 需以 order_id 或 hl7_file_input id 做 dedupe/unique，否則 admin queue 充斥重複。
- expiry cron 多 pod 競態：intake pod 可能多副本，@Cron 在每個 pod 都跑會重複寄 48hr 信/重複 expire → 需確認 cron 只在單一 pod 跑（比照 scheduled-reports 的 pod gating）。
- 48hr 警示信收件人：AC5 「assigned admin or default group」未指定 default group 是誰，未定義會靜默不寄。
- matched_practice_id/provisional_provider_id FK 指向 VP-16164/VP-16168 的表，這兩票 read-path 尚未切/表尚未建 → 本票先存 nullable 值不加硬 FK constraint 以免 cross-ticket 卡住。
