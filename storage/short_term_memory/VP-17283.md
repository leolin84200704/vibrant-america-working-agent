## Ticket Analysis

Origin: design meeting (recording API.m4a, 45min). Transcribed via whisper large-v3-turbo.
Goal: turn the API-exposed order path into a PID-based flow that delegates patient/product
eligibility judgment to the Order Eligibility Check API and places orders via the new
Place Order API. HL7/SFTP path stays unchanged.

Key reference specs (Confluence, space LIS):
- Order Eligibility Check API: pageId 2517139459 (POST .../v2/portal/order/orders/eligibility-check)
  - input: patient_id, customer_id, clinic_id, line_items[], source (enum incl "API","EMR")
  - returns {eligible, failures[]}; runs 10 checks server-side by patient_id:
    patient/customer exist, patient-in-clinic, profile-complete (only source=="API": DOB/gender/address),
    ProZ follow-up gating (item 800/831), Skincare follow-up (801-807), genetic-test-once,
    Gut Zoomer NY / non-NY (both directions), PFSA male-only.
- Place Order API: pageId 1881997337 (POST .../v2/portal/order/orders) — new OrderRequest schema
  (line_items w/ pricing, charge_method, payments, fees). Its own flow also runs eligibility.
- Current Order Intake doc (Leo wrote, to rewrite): pageId 2485977089 (VP-16934, demographics-based).

Both new APIs on host api.vibrant-wellness.com (prod) / .../order-staging/ (staging).

## Approaches Considered

Chosen: modify existing /order-intake API path (NOT a new endpoint).
- Input: patient_id (no demographics/address — caller sends none).
- customer_id + clinic_id from JWT (UserPayload.customer_id/clinic_id confirmed present:
  src/modules/hl7-order-processing/services/token-helper.service.ts:25-26; auth.service.ts).
- source always "API".
- Flow: BestDeal (local) -> line_items -> eligibility-check -> if eligible Place Order API else return failures[] as actionable errors.
- NY: NO local auto-swap on API path (eligibility rejects mismatch; surface clear msg). gz-ny-routing swap/fee/email stays HL7-only.
