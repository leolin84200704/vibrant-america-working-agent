**需求本質**：
為已確認 (Confirmed) 的 lab consult appointment，在 48hrs / 24hrs / 15mins 前送 reminder email 給 provider，使用 FE 建立的 Postmark template 33802989（server 8340335）。

**技術細節**：
1. Scheduled job/cron/trigger 在 `appointment_date_time` 前 48hrs / 24hrs / 15mins 觸發
2. 整合 Postmark template (33802989)
3. Edge cases:
   - Cancelled 不送
   - Rescheduled 要重算觸發時間（不基於舊時間）
   - 不重複送（同 appointment 同 reminder type 只送一次）
4. 套用既有 email retry 機制
5. Log send attempts + outcomes

**Acceptance Criteria**:
- AC1: 48/24/15min 前精準觸發
- AC2: 用 FE 設定的 Postmark template
- AC3: Template variables 從 appointment 正確填入
- AC4: appointment_date_time 用 provider 本地時區格式化
- AC5: Cancelled 不送
- AC6: Rescheduled → 重算（不基於舊時間）
- AC7: 不重複送

## Approaches Considered
### [2026-04-30]

**Explore agent 回報關鍵發現**:
- **無既有 calendar reminder 機制**，需新蓋
- **參考實作**: `LIS-setting-consumer/src/setting-consumer/pns.service.ts:12647-12790` FedEx Pickup Reminder（Kafka event + Redis lock + Bull queue + Postmark）
- **Lab consult appointment 資料模型**:
  - 主表 `v2_event` (`LIS-transformer-v2/prisma/schema.prisma:164+`)
  - status enum: scheduled/confirmed/checkedin/canceled/noshow（line 237）
  - `is_canceled` boolean
  - `start_time` (Timestamptz), `timezone` (string)
  - participant status enum: no_response/accepted/tentative/declined
- **Postmark 整合**: `LIS-transformer-v2/src/calendar/models/notification/email-template-config.service.ts:39-100`（YAML 驅動 template ID 管理）
- **既有 Kafka topics** (`appointment-event.service.ts:88-99`): general-sample-events / lis-general-events，event_type: Appointment Created/Updated/Cancelled/Deleted
- **既有 email flow**: LIS-transformer-v2 publish Kafka → LIS-setting-consumer 消費 → Bull queue → gRPC NotificationService → Postmark
- **Provider timezone**: `v2_calendar.timezone` (default America/Los_Angeles)，已用 date-fns-tz `toZonedTime()`
- **Cron 基建**: `@nestjs/schedule` 已 import，`LIS-transformer/src/schedule/schedule.service.ts` 有 @Cron 範例（但跟 calendar 無關）

**三個候選方案**:

**A. Bull Delayed Job 模式（主動排程）**
- Appointment Confirmed Kafka event → LIS-setting-consumer 計算 3 個觸發時間 → 加 3 個 Bull delayed jobs (jobId = `lab_consult_reminder_{eventId}_{type}`)
- Cancel event → `queue.removeJobs(jobId pattern)`
- Update/Reschedule event → 移除舊 jobs 重排
- 防重複: jobId unique，重排自然取代
- 優點: FedEx pattern 類似，精準觸發無 polling 開銷；Bull 已有 retry 機制
- 風險: 15min reminder 需 confirmed → 排程 → Bull 接 → 若 confirm 距 appointment <15min 要 fail-safe；Bull job 若被誤刪需 fallback；Redis 持久化必須開（既有設定有開）

**B. Cron 定期掃 DB（被動掃描）**
- 新 @Cron 每 1 分鐘掃 `v2_event` where status=confirmed AND is_canceled=false AND lab_consult AND start_time in [now+48h-1m, now+48h+1m] (24h, 15m 同理)
- DB 加 3 欄位 `reminder_48h_sent_at` / `reminder_24h_sent_at` / `reminder_15m_sent_at` 防重複
- Reschedule: 若 start_time 改 → 對應欄位 reset (UPDATE 時 trigger 或 service 邏輯)
- 優點: 邏輯直觀、可審計、Reschedule 處理簡單（只看 start_time，不用追 job）
- 風險: 1 分鐘 cron 精準度只到 ±30s（若 SLA 嚴格需 30s cron）；schema migration；DB scan 開銷（appointment 量級需評估）；服務 downtime 期間若超過 cron 窗口會漏發

**C. 混合（Kafka event 觸發 Bull delayed + DB 防重複欄位）**
- 採方案 A 的排程機制 + 方案 B 的 DB 欄位（防 Job 重複觸發）
- Bull job 觸發時 SET column = NOW() 加 SELECT FOR UPDATE skip locked 防雙觸
- 優點: 雙重保險、可審計、與 FedEx pattern 一致
- 缺點: 多了一個 schema migration（但只 3 timestamp 欄位，低風險）

**初判推薦: 方案 C** — 跟既有 FedEx 模式 align，加 DB 欄位提供可審計性同時防重複（AC7），reschedule 重排 + reset 欄位處理 AC6，cancel 移除 jobs 處理 AC5。

### [2026-05-01] — Reality Check 後調整為 C3
**Reality check finding**:
- transformer-v2 **無** Bull infrastructure（package.json 確認無 `@nestjs/bull` / `bull` / `bullmq` / `@nestjs/schedule`）
- `src/calendar/models/schedule/schedule.module.ts` 是 calendar 自定義 module，**不是** NestJS 排程框架
- Bull 只在 LIS-setting-consumer 有
- transformer-v2 有 `ioredis`（line 68）+ `kafkajs` ✓
- 既有 `EmailService.sendEmail()` 已支援 `delay` 參數（payload `Delay` field），透過 Kafka topic `notification-email-template` 走下游

**C 的兩個 sub-option**:
- C1: transformer-v2 引入 `@nestjs/bull` + `bull` + Redis BullModule 整合（新大 dep + 測試複雜度高）
- C2: 整套 reminder 邏輯放 setting-consumer（跨 repo + 動 16K 行 controller 周邊）
- **C3 (採用)**: 引入 `@nestjs/schedule` + cron 30s 掃 `v2_event_reminder_log`，atomic CAS 防重複；audit/防重等價於 C，dep 最輕，測試簡單

**最終方案: C3**
- Schema: 新表 `v2_event_reminder_log` (PostgreSQL `prisma/schema.prisma`)
- Trigger: `@nestjs/schedule` `@Cron` 30s（暫定，待 Leo 最終決定頻率）
- Dispatcher: cron handler 找 `WHERE status='scheduled' AND scheduled_for <= NOW() + 30s` → atomic UPDATE → 'sent' → 呼叫 `EmailService.sendEmail()` 走既有 Kafka path
- Reschedule: event.service.ts 偵測 `start_time` 改 → UPDATE 舊 row `status='superseded'` + INSERT 新 scheduled rows
- Cancel: UPDATE all event_id rows `status='superseded'`
- Audit: log row 保留 `bull_job_id` 欄位改名 `dispatch_id`（或乾脆移除，C3 不需要）
