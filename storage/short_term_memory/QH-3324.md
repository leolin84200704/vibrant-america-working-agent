## Ticket Analysis

**Goal（目的）**：驗證 Clinical Consult（practice 150105）的活動建立已放寬「必須同時有 Provider+Patient」的限制，改為「至少一個參與者即可」，以支援 block-out time / staff meeting 等場景。本 ticket 是 QA Task（透過 is tested by 掛在已 Done 的 VP-15374 下），本質是對 Clinical 入口 createEventByPatient 跑 VP-15374 的 4 條 AC 驗證，預期 0 code change。

**Classification**：`calendar-clinical` | **Effort**：S | **Confidence**：5/5 | **Ticket fetched**：是
**Affected repos**：LIS-transformer-v2

### 建議方案（改前 / 改後為何有效 / 改什麼）
改前現狀：放寬邏輯已實作完成且涵蓋 Clinical 路徑。(1) src/calendar/models/event/event.model.ts:22-34 AtLeastOneParticipantConstraint（customer_id OR clinician_calendar_id 任一即過）。(2) src/calendar/models/event/event.service.ts:1565-1574 createEventByPatient（Clinical 專屬入口，CLINICIAN_PRACTICE_ID=150105，resolver event.resolver.ts:248）已做 hasCustomerId||hasClinicianCalendarId||hasParticipants，零參與者才 throw BadRequestException。

改後為何有效：VP-15374 的 4 條 AC（只 Provider OK / 只 Patient OK / 零參與者報錯 / 兩者皆有照舊）在 Clinical 路徑全部命中，無需新 code。

具體改什麼：採方案 A（QA 驗證，0 code change）。產出驗證報告，用 staging GraphQL 真打 createEventByPatient（非 mock，符合 verified-means-live 鐵則）。驗證 5 個案例：(1) 只 customer_id 無 clinician → 應成功；(2) 只 clinician_calendar_id（role 須 clinicadmin，service:1665）無 patient → 應成功；(3) 零參與者 → 應 400；(4) customer_id+clinician 皆有 → 照舊成功；(5) participants 陣列路徑（含 ownership 1608 / practice 校驗 1601）。

對工程師判斷的修正：(a) 工程師把 service:1587 find((p)=>p.role) 當可疑 bug，實際上 role 為 optional，此行只是「挑第一個有 role 的 participant 當 creator」，無 role 則 fallback participants[0]（1615）。註解「find patient role」是 stale comment，非行為錯誤，不影響任何 AC，至多算 comment nit，不應因此走方案 B。(b) 工程師低估了一點：VP-15374 ACs 是為 Practice/Providers 的 createEvent 寫的，Clinical 入口 createEventByPatient 有額外更嚴格子規則（clinician 須 clinicadmin、calendar 須在 150105、ownership 校驗），所以 AC1/AC2 在 Clinical 不能假設等價，必須逐一 live 驗。

僅當方案 A 驗出缺口才走方案 B（補 event.service.spec.ts 的 createEventByPatient 三條 AC case + 視情況修），branch bugfix/leo/QH-3324；若純加 test 則 feature/leo/QH-3324。無 schema/migration、無新 process.env、不動 config yaml。注意：目前在 feature/leo/VP-16513，base 須重切自 stage_test，切完跑 prisma generate（兩個 schema）+ start:dev/test 必須過。

### 候選方案與正反取捨
- **方案 A：當 QA 驗證任務，先不寫 code，用 staging GraphQL 跑 5 個案例**
  - 取捨：零 prod 風險、符合 Confirm-Before-Over-engineering；但若 PM 期待產出可回歸的自動化覆蓋則需再追方案 B
- **方案 B：補 event.service.spec.ts 對 createEventByPatient 的 AC unit test（目前 spec 對此入口 0 覆蓋）**
  - 取捨：留下回歸保護、符合 test-before-push；但 createEventByPatient 無現成 spec scaffold，須自建 mock，工時較大，且無明確 PM AC 要求會被視為超範圍

### 風險與邊界案例
- createEventByPatient 對 clinician_calendar_id 強制 role==='clinicadmin'（service:1665）——若 QA 場景是『只放一個一般 provider、無 patient』，會被擋下並非 bug，但要確認這符合 PM 對『只 Provider』的定義
- participants 陣列路徑的 ownership 校驗（service:1608 calendar_owner_id !== customerId && !isClinicAdmin）與 practice 校驗（1601 practice_id !== 150105）會在跨 practice / 非本人 calendar 時 throw，需納入負向測試
- service:1587 find((p)=>p.role) 註解寫 find patient role 但實際只看 role truthy——非行為 bug，但建議 QA 順手回報讓 owner 改 comment 避免日後誤讀
- spec 目前對 createEventByPatient 零覆蓋，若日後有人改 1565-1616 區塊（QH-2648/QH-1159/QH-1591 同屬 Calendar Clinical 群）無單元測試保護，merge 衝突或 regression 不易被發現
- staging 上需有 practice 150105 的測試 calendar（含 clinicadmin role 與 patient role 各一）才能完整跑 AC，環境資料不齊會誤判 fail
