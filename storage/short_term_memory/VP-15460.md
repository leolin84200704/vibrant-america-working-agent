### [2026-04-28~29] Hotfixes (post-merge)
- **lis-emr-v2 #88** — `redlock@4` is CommonJS-only with no `.default`; `import Redlock from 'redlock'` compiled to `redlock_1.default` (undefined) → `TypeError: redlock_1.default is not a constructor` at startup. Fix: `const Redlock = require('redlock')`.
- **lis-emr-v2 #90** — `lock.release()` is the redlock@5 API; `redlock@4` Lock prototype only has `unlock()`. Every cron tick logged `lock.release is not a function` (cosmetic — TTL still expires the lock). Fix: `lock.unlock()`.

### [2026-04-28~29] Tangential EMR-Backend fixes (incident-discovered)
- **#158** — `hl7_file_input.emr_code_not_found` (VARCHAR 255) and `customer_not_found` (VARCHAR 45) overflow when stack-trace-style strings are written. UPDATE throws `Data truncation: Data too long for column` and bubbles as job-level PersistenceException. Added `truncate(s, max)` helper + applied at 3 call sites.
- **#159** — Practice Fusion sends HL7 v2.5 `OML_O21` (Laboratory Order) but `HL7_package.parseOrderFromMessage` blindly casts to `ORM_O01` → `ClassCastException` → file stuck after 5 retries. Quick fix: `normalizeIncomingMessageType` rewrites MSH-9 from `OML^O21[^OML_O21]` to `ORM^O01^ORM_O01` so HAPI parses into the existing model; downstream parser unchanged. OML-only segments (SPM/SFT/UAC) silently ignored by HAPI non-strict parser.

## Failures

### [2026-04-27] Wrong proto file edited initially
Edited `src/proto/customer.proto` (`package lis`, legacy LIS host) before realizing v2 RPC lives in `src/proto-v2/customer.proto` (`package coresamples_service`, coreSamples host). Reverted both `proto/` + `dist/proto/` and applied to `proto-v2/`. Detection trigger: reading `src/config/grpc.config.ts`. Lesson already in `long-term-memory/patterns.md` (under "lis-backend-emr-v2 雙 proto 樹").

### [2026-04-28] redlock CommonJS interop (#88)
Production NestFactory crash at startup: `TypeError: redlock_1.default is not a constructor`. Root cause: `redlock@4` is plain CommonJS (`module.exports = Redlock`, no `.default`); this repo's `tsconfig.json` only sets `allowSyntheticDefaultImports`, not `esModuleInterop`, so `import Redlock from 'redlock'` compiled to `redlock_1.default` (undefined). Should have caught this at code review by recognizing redlock's package age + checking `tsconfig`.

### [2026-04-28] redlock Lock API confusion (#90)
Picked `lock.release()` from redlock@5 docs while installing redlock@4. The two versions have different Lock prototypes (`unlock` vs `release`). Cosmetic in production (TTL covered the leak) but log noise + would have been a real bug if TTL was raised.

### [2026-04-27 → 28] Cwd persistence in Bash tool calls
After `cd /Users/hung.l/src/EMR-Backend && gh pr view 156`, subsequent Bash calls without explicit `cd` defaulted to EMR-Backend. Created `bugfix/leo/VP-15460-redlock-import` in the wrong repo, had to clean up. Lesson: always explicit `cd` in cross-repo flows.

### [2026-04-28] Migration not applied automatically
Agent committed migration SQL to repo and assumed release pipeline would `prisma migrate deploy` it. Leo had to remind: "你 sftp_folder_mapping 的改動還沒真的上傳到 database". Then `prisma migrate deploy` failed with P3005 (DB never baselined for prisma migrations) → fell back to `prisma db execute --file <sql>` (raw SQL apply). Then "192.168.60.11:3306 也要 apply" — second DB. Lesson: this repo has two MySQL instances + Prisma is not the migration source-of-truth in prod.

## Retrospective

### Result vs initial assumptions
- **Initial plan (2026-04-23)** assumed adding `customer_name` to `ehr_integrations` was needed — verified on resume that it already exists, so plan was outdated
- **Estimated scope** included full payment + sendOrder pipeline; reality showed Java porting (ChargeClient, OrderTestClient, OBR dispatch, JWT generation) is 2-3 days minimum, justifying the Stage 3a/3b split
- **Multi-pod safety** — initially debated whether redlock was overkill for 2 pods; in production logs the lock contention is visible and working as intended (one pod skips per tick) — design decision validated

### Failure analysis (preventable?)
| Failure | Preventable? | How |
|---------|--------------|-----|
| Wrong proto file (Stage 0) | Yes | Always read `grpc.config.ts` before editing any `.proto` |
| redlock CommonJS interop | Yes | Run `node -e "new Pkg(...)"` smoke test for any new dependency before deploy, not just `tsc` |
| redlock Lock API | Yes | Read the actual installed package version's source/types before writing wrappers, not the latest docs |
| Wrong-repo branch creation | Yes | Always `pwd` + `git branch --show-current` before `git checkout -b` in cross-repo flows |
| Migration auto-apply assumption | Yes | Ask user explicitly about migration apply path; don't assume "release pipeline handles it" |

### Pattern recognition
- **VP-16280, VP-16329, VP-15460** all hit "agent assumed something the codebase had drifted past" — outdated STM, outdated docs, or stale memory of repo structure. Mitigation: read current code, don't trust prior STM blindly.
- **Multi-stage tickets benefit from per-stage commit + Leo review** (Stage 1 → review → Stage 2 → review → Stage 3a → review). Saved at least one wrong direction (would have built worker on top of broken proto sync if I'd skipped Stage 1 review).
