## Ticket Analysis
### [2026-04-29 17:35]
**Ticket**: VP-16361 [BE] Support rescheduling clinician-initiated lab consult appointments
**Status**: Dev To Do, Assignee: Leo, Reporter: Kristine, Priority: P2, Due 2026-05-01
**Linked**: PH-759 (parent), VP-16359 (FE temp hide, done), VP-16360 (FE unhide, blocked by this), QH-4898 (QA)

**問題本質**:
- Provider Portal 嘗試 reschedule 由 clinician 代建的 lab consult appointment 時，後端 API 報錯。
- Lab consult appointment 走 `updateEventByPatient` GraphQL mutation（位於 `LIS-transformer-v2`，clinician practice_id = 150105）。
- 設計上該 endpoint 是「provider 自己當 patient 去 consult clinician」，所以 creator role 限定為 'patient'，creator owner 必須等於登入 provider 的 customer_id。

**拒絕點 (root cause)**:
- `src/calendar/models/event/event.service.ts:1619-1622`
  ```ts
  if (creatorCalendar.role !== 'patient') {
    throw new ForbiddenException('This endpoint only works with events created by patient role calendars');
  }
  ```
- 此外 `:1629-1631` 的 ownership check 也會擋：clinician-initiated 的 creator 是 clinician calendar，owner 是 clinician 的 customer_id，不會等於 provider 的 customerId。

**影響範圍**:
- `src/calendar/models/event/event.service.ts` — `updateEventByPatient`（reschedule 入口）
- 可能也要看 `deleteEventByPatient`（同樣有 role==='patient' check at line 1775）—— ticket 沒提，先不動
- Notification/Kafka publish 流程：要加 originating clinician 的通知對象
- Test: `event.service.spec.ts`

## Approaches Considered
### [2026-04-29 17:35]
**A. 放寬 creator role + 改 ownership check（PM 隱含預期）**
- 允許 `creatorCalendar.role` 為 `'patient'` 或 `'clinician'`
- Ownership 檢查改為「user 是 event 的 assigned provider」(透過 v2_event_participant 找 provider 角色 participant，比對 calendar_owner_id)
- Notification: clinician-initiated 時加 originating clinician(creator calendar owner) email
- 優點: 最小改動，符合 AC1/AC2/AC4/AC5；前端不必改
- 風險: ownership 邏輯改寫要小心，要確保 provider-initiated case 不退化（patient creator 仍需 owner == provider customer_id）

**B. 新增專用 endpoint `updateClinicianInitiatedEventByProvider`**
- 隔離 clinician-initiated reschedule 路徑
- 優點: 語意清楚、低風險
- 缺點: 違反 AC2（同 endpoint 行為一致）；前端要改；endpoint 蔓生

**C. 改用 meeting-request 系列 (`updateMeetingRequestForMe`)**
- 已支援 reschedule 且無 creator-role 限制
- 缺點: lab consult 不一定是 meeting request 模型；要先確認資料結構是否一致；風險最大（架構級改動）

**初判推薦: A**
