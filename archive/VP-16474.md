## Ticket Analysis

### [2026-05-14]
- Title: [BE] Persist and return provider name on Clinical Consult create/fetch endpoints
- Blocks: VP-16458 (Provider Name missing on Confirmation modal — Bug)
- Reporter: Kristine Legaspi (PM)
- Original ask: add `provider_name` field to create payload, persist on event record, return on fetch endpoints (AC1-AC6)
- 6 acceptance criteria centered on a new `v2_event.provider_name` column

## Approaches Considered

### [2026-05-14] Options explored before pivot
1. **Persist on `v2_event`**: add `provider_name VARCHAR(255)?` column, accept in InputType, store, return. Straight reading of ticket AC. Migration + new field everywhere.
2. **Derive from `v2_calendar` JOIN**: skip column. `creator_calendar_id` → `v2_calendar.calendar_owner_firstname/lastname` → join + concat. `createEventByPatient` already sets creator=clinician when patient passes `clinician_calendar_id`, so creator IS the provider for the Consult flow.
3. **FE passes provider_name → BE writes into `v2_calendar`**: hybrid. Accepts payload, but the persistence target is the calendar row, not the event.
4. **Stuff into existing `notes` free-form field, FE parses it back out** ← chosen.

## Decisions Made

### [2026-05-14] Encode in existing `notes` field — no BE change
The current `notes` payload already carries semi-structured context:

    [Accession Id 2605046944: Gut Zoomer] [Phone: 123-123-1231] [Email: test@123.com] [Meeting Type: Zoom Meeting]

FE can extend the same `[Key: Value]` envelope with a provider entry (e.g. `[Provider: Tara Calmes-Norgaard]`) on create, and parse it back when rendering the Confirmation modal. The Event GraphQL type already returns `notes` on every fetch — no resolver change needed.

**Outcome**:
- ❌ No Prisma migration
- ❌ No new field on `Event` ObjectType
- ❌ No `CreateEventInput` / `CreateEventByPatientInput` / `UpdateEventInput` change
- ❌ No service-layer change
- ❌ No BE PR for VP-16474
- ✅ FE owns the encoding contract end-to-end

## Retrospective

### [2026-05-14]

**Initial assumption vs reality**:
- Assumption: ticket needs BE schema + persistence + return changes per AC1-AC6.
- Reality: cheapest cross-team solution is "FE encode in notes, FE decode on read" — BE doesn't move at all.

**Why this is fine**:
- `notes` already accepts arbitrary string, already returned on every Event fetch.
- Provider name has no internal BE consumer right now (no rule / search / report keys on it). It's purely display data for FE's Confirmation modal.
- No migration risk, no API contract churn, no rollback footprint.

**When this pivot is NOT fine** (record for future):
- If BE needs to reason about provider_name (search, filter, report, audit) → must store properly.
- If multiple FE surfaces will read it and parse the same envelope → fragile; structure should be DB-owned.
- If the envelope format isn't already a documented convention → adds a covert FE-FE contract that the next dev will trip over.

**Confidence**: 5 / 5 (no code shipped, no risk).

## Lessons Learned

### [2026-05-14]
- **Before designing a new column/field, check if an existing free-form field already carries the data shape**. `notes` here had no schema, FE already encoded other key/value pairs in it, and adding provider was a strict extension — zero BE cost.
- **The cheapest correct answer can be "do nothing in BE"**. Worth surfacing as an option in the Step 4 user-discussion phase, not just the implementation paths.
- Recorded the generalization in LTM `patterns.md` under "Before adding a new field, check if existing free-form field covers the use case".
