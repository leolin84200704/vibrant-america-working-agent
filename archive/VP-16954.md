## Ticket Analysis

### [2026-06-10]
- Bug：Accession **2502136775** 排程 consult 顯示「已使用」無法用，但 Unimod Panel reset 回「No claim existed; no action taken」。ticket 自述 = consultation scheduling status 與 Unimod claim records 不同步。
- 機制（VP-16410，clinic 150105）：accession 1:1 claim，三個 FE/後端 gate 全讀 `v2_event_accession_claim`：`getClaimedAccessionIds`(list 來 disable)、`isClaimable`(單筆檢查)、`claimForEvent`(createEvent 時 claim)。reset 也讀同表。
- accession 存兩處：`v2_event.accession_ids String[]`（事件資料）+ `v2_event_accession_claim`（1:1）。

### calendar_prod 實查 (2026-06-10，純讀)
- `v2_event_accession_claim` WHERE accession_id='2502136775' → **0 rows**
- `v2_event` WHERE '2502136775'=ANY(accession_ids) → **0 rows**（取消/未取消都無）
- `v2_event_accession_audit_log` → 只有 1 筆 `released_by_reset`（event_id NULL，reason「manual reset (no claim found, no-op)」，actor user:150459，2026-06-10 15:59）。**無任何 claimed/rejected/backfilled** → 此 accession 從未進過 calendar claim 系統。

### [2026-06-10] 更正後定論：used 來源 = claim 表本身（FE isAccessionClaimable）
- FE `va-portal/.../ScheduleMeeting/related-test-dialog/SearchOrders.vue:289-296`：「The Accession ID selected has been used」**只**由 `checkIsAccessionClaimable(accessionId)` 回 `claimable=false` 觸發。catch(L307-309) 只 console.error、**不**設 used。
- `isAccessionClaimable` = `v2_event_accession_claim.findUnique({accession_id})`，與 `resetEventAccession` 讀**同一張表**。
- 故 used 來源 = claim 表，**非**上游 order 系統。沒有 consult_used/consult_scheduled 之類旗標。
- **calendar_prod 實查（含變體/孤兒檢查，純讀）**：
  - claim 表 `LIKE '%2502136775%'` → 0 rows（排除前後空白/格式變體）
  - audit 只有 1 筆 reset no-op（accession len=10 乾淨）；**無 claimed/backfilled/rejected/released_by_event**
  - 全表無 orphan claim（event_id 都對得到 v2_event）→ claim 系統健康
- audit log append-only：若曾透過 service claim 過必留 `claimed` row → 完全沒有 → **此 accession 從未透過應用程式被 claim**。
- **∴ `isAccessionClaimable('2502136775')` 現在必回 claimable=true → 可預約。後端從未把它記為 used。**

### [2026-06-10] provider 當初為何看到 used（無法從後端重現）
