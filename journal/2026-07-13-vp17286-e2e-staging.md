## Discovery path worth remembering
- Test patients WITHOUT VPN/DB: portal staging API `orderTest/searchPatient?inputPatient=`
  (base `https://api.vibrant-wellness.com/v1/portal/order/staging`, dev-secret JWT,
  scoped by token customer). `orderTest/patient?patientId=` gives full demographics.
  `allTests` is 404 on staging — use prod Azure MySQL package_price_mapping instead
  (uniqueemrcode column) for code discovery.
- emr-v2 staging DB without VPN: `kubectl exec` into the AKS staging pod, write a
  node script into `/app/temp/`, `require('@prisma/client')` resolves there (NOT
  from /tmp — module resolution walks up from the script dir).

- validate-git-push hook false-positives on `rm -f` inside unrelated commands.
- VAREQUISTION463 — the "known good test code" from VP-16934 memory — turned out to
  BE Gut Zoomer (GZ_EMR_CODE), so the happy-path test and the NY-swap test use the
  same product family. VAREQUISTION99 = PSA duo (item 376+377) = the male-only rule.

## The two defects (why mocks missed them)
Both defects live in the seam BETWEEN components that unit tests mocked away:
- Finalizer caught placeOrder errors to mirror legacy sendOrder semantics; intake
  service trusted `sampleId:null` as placed. Each component "correct" locally.
- Mapper read OrderFrontend.customerId/patientId; spec baseOrder() set them; the
  real assemble pipeline never does (grep: zero assignments repo-wide). customer_id=0
  went to the wire. Only a live call could catch either.
Fix PR #249 (branch bugfix/leo/VP-17286). Same silent-failure class as VP-17342 —
second instance in this codebase in a week; pattern: "legacy-tolerant catch reused
on a path whose caller interprets null as success".
